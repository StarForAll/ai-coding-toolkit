# 连锁门店巡检系统试运行交付

## Goal

在 2 周内交付一个供连锁餐饮客户试运行的门店巡检系统 Web 版本，支持巡检填报、总部看板、整改报表导出，并以 `trial_authorization` 轨道控制尾款前后的授权与最终移交。

## Requirements

- 提供门店经理巡检表单提交和历史记录查看能力
- 提供总部问题汇总看板，支持按门店、日期、问题类型筛选
- 支持整改报表导出
- 试运行版本必须显示授权状态，并在到期后进入只读模式
- 只读模式必须保留历史数据查看与导出能力
- 尾款到账后 3 个工作日内完成永久授权切换、源码移交、控制权移交
- PRD 冻结后，纯澄清若不改变范围、接口契约、验收标准、成本、工期，可留在当前阶段处理
- PRD 冻结后，新增 / 修改 / 删除需求统一进入 `§2.5 需求变更管理`，获批后再回到受影响的最早阶段更新基线

## Acceptance Criteria

- [ ] 门店经理可提交巡检记录并查看历史记录
- [ ] 总部可查看筛选后的问题汇总并导出整改报表
- [ ] 试运行授权状态对客户可见，含剩余有效期和升级说明
- [ ] 试运行到期后系统进入只读模式且仍可导出历史数据
- [ ] 永久授权切换无需重装且不丢失数据
- [ ] 交付文档明确区分试运行交付与最终控制权移交

## 冻结与变更约定（对客）

本文档确认的是本轮交付范围与验收标准。

- 如果后续只是对原有需求的澄清，且不改变范围、接口契约、验收标准、成本、工期，则留在当前阶段直接处理
- 如果后续出现新增、修改或删除需求，则先进入 `§2.5 需求变更管理`
- 变更会先评估影响范围、时间和费用，待客户确认后再并入执行

## Definition of Done

- 关键用户流程在设计文档和任务拆解中都有对应说明
- 试运行授权条款与最终移交门禁在 `assessment.md`、`task_plan.md`、`delivery/` 中一致
- 设计文档完整性检查通过
- `task_plan.md` 结构验证通过

## Technical Approach

- 交付轨道采用 `trial_authorization`
- 设计阶段导入 `delivery-control`、`authorization-management`、`transfer-checklist`，并在最终移交场景下补 `secrets-and-config`
- 任务拆解中显式区分：
  - 试运行版交付任务
  - 永久授权切换任务
  - 源码移交任务
  - 控制权移交任务
  - 密钥配置移交任务
- 冻结后若出现新增 / 修改 / 删除需求，不直接在设计、计划、交付阶段吸收，而是先走正式变更流程

## Out of Scope

- 不做移动端 App
- 不做复杂 BI 或多租户能力
- 不在尾款前移交正式生产控制权
