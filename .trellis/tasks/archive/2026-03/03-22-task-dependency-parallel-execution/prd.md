# brainstorm: task dependency ordering and parallel execution

## Goal

补充 `docs/workflows/新项目开发工作流/` 中关于“任务拆分后如何根据依赖关系决定是否可开始执行”的规则说明，让工作流不仅能列出前置任务、阻塞任务、并行任务，还能明确执行时机、等待条件、并行边界和用户确认节点。

## What I already know

* 用户要补充的场景是：需求已经明确、进入具体任务划分后，需要区分任务之间是否存在先后顺序，以及哪些任务可以并行执行。
* 用户给出的典型例子是：`A -> B -> C` 为串行链路，`A2` 与它们无依赖，因此在执行 `A` 时可以执行 `A2`，但不能提前执行 `B` 或 `C`。
* 当前文档已经有“依赖关系”字段模板，包括“前置任务 / 阻塞任务 / 并行任务”：
  * `docs/workflows/新项目开发工作流/工作流总纲.md`
  * `docs/workflows/新项目开发工作流/commands/plan.md`
* 当前文档已经提到“识别可并行执行的任务”和“根据依赖关系排期”，但缺少明确的执行规则：
  * 没有定义“任务何时允许进入执行”
  * 没有定义“执行中的任务会阻塞哪些后续任务”
  * 没有定义“并行任务的判定条件和冲突边界”
* 当前校验脚本 `docs/workflows/新项目开发工作流/commands/shell/plan-validate.py` 原本只检查是否出现“依赖/前置/并行”等关键词，后续已按本任务补齐结构校验能力。

## Assumptions (temporary)

* 本次需求的核心是补充“规则说明与流程约束”，不要求立即设计完整的任务调度算法或自动化编排器。
* 文档补充至少会影响 `工作流总纲.md`，并且需要同步调整 `commands/plan.md`，以免总纲与命令说明不一致。
* “并行可执行”默认不是只看是否无前置任务，还需要补充至少一层“资源/文件/环境/人工确认点不冲突”的判断规则。

## Open Questions

* 后续是否还要继续细化“冲突说明”的固定分类枚举。

## Requirements

* 在任务拆解阶段，必须显式区分“串行依赖任务”和“可并行任务”。
* 对每个任务，除了列出依赖关系，还要说明“何时允许开始执行”。
* 对执行中的任务，要明确其会阻塞哪些后续任务启动。
* 对可并行任务，要明确最小判断标准，避免表面无依赖但实际产生冲突。
* 需要有一个贴近真实项目开发的示例，覆盖 `A -> B -> C` 与 `A2` 可独立执行的场景。
* 在进入实现前，仍需保留“用户确认方案后再实施”的门禁。
* 本轮补充深度采用“标准版”：
  * 包含基础开始/等待/并行规则
  * 包含并行冲突判定标准
  * 包含推荐排期顺序
  * 不扩展到完整任务状态机说明

## Acceptance Criteria

* [ ] 工作流文档中明确说明：若任务 `B` 依赖 `A`，则 `A` 未完成时不得开始执行 `B`。
* [ ] 工作流文档中明确说明：若任务 `C` 依赖 `B`，则 `B` 未完成时不得开始执行 `C`。
* [ ] 工作流文档中明确说明：与依赖链无关的独立任务（如 `A2`）在满足自身开始条件时可以并行执行。
* [ ] 工作流文档中给出并行判定的最小标准，例如依赖满足、资源不冲突、文件/模块边界可隔离、人工确认点不冲突。
* [ ] 至少一个现有文档位置补入示例或规则表，使读者可以据此判断“现在该做哪个任务，哪些任务必须等待”。
* [ ] `工作流总纲.md` 与 `commands/plan.md` 对相同规则使用一致术语。
* [ ] 文档中给出推荐排期方法：先安排无前置且高优先级任务，再识别可并行分支，最后回到依赖链后续任务。
* [ ] Step 4 的文档承诺与 `plan-validate.py` 的实际校验能力保持一致。
* [ ] 计划阶段增强后，实施阶段与 Phase Router 有配套闭环说明。

## Definition of Done

* 需求边界和补充范围经用户确认。
* 正式文档修改后，相关引用位置前后一致。
* 如变更了模板或验证要求，相关命令文档与脚本同步更新。
* 说明中包含可执行判断规则，而不只是概念性名词。

## Out of Scope

* 现在就实现自动任务调度器、依赖图求解器或新的任务管理程序。
* 修改当前项目下 `.agents/`、`.claude/` 或其他运行时目录。
* 为 `/trellis:parallel` 设计完整独立命令体系，除非后续明确要求扩展。

## Constraints

* 允许修改的正式内容仅限 `docs/workflows/新项目开发工作流/` 目录。
* `.trellis/tasks/03-22-task-dependency-parallel-execution/` 作为当前任务记录允许保留和更新。

## Decision (ADR-lite)

**Context**: 需要补充“任务依赖顺序与并行执行”的工作流规则，同时保证嵌入前的源码工作流文档可直接复用。  
**Decision**:
* 修改范围限定为 `docs/workflows/新项目开发工作流/` 目录，以及当前任务记录目录。
* 规则深度采用“标准版”，即补基础规则 + 并行冲突判定 + 推荐排期顺序，不扩展到完整任务状态机。
* 在后续审阅中确认“文档验证要求与脚本实现脱节”后，将 `commands/shell/plan-validate.py` 一并纳入修改范围。
* 在后续审阅中确认“计划阶段增强后，下游实施与路由未配套改造”以及“parallel 引用口径不清”后，同步补充了 `start-patch-phase-router.md`、`test-first.md`、`命令映射.md` 等源码文档。

**Consequences**:
* 工作流源码文档可直接用于后续嵌入前分发。
* 不再触碰当前项目的运行时目录，避免把“待嵌入源码”与“当前项目运行态”混在一起。

## Technical Approach

* 在 `工作流总纲.md` 中补充任务依赖、执行安排、任务执行矩阵、状态枚举、最小状态流转、阶段五实施闭环。
* 在 `commands/plan.md` 中同步要求生成 `task_plan.md` 时输出上述信息，并保证 Step 4 校验说明与脚本能力一致。
* 在 `commands/shell/plan-validate.py` 中补齐结构完整性与关键字段一致性校验，并新增 `test_plan_validate.py` 作为脚本回归样例。
* 在 `commands/start-patch-phase-router.md` 中补充执行矩阵驱动的路由逻辑。
* 在 `commands/test-first.md` 与 `命令映射.md` 中统一并行开发入口口径。
