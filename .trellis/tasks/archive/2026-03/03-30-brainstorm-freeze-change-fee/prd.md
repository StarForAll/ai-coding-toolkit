# brainstorm: 新项目工作流需求冻结收费规则

## Goal

分析 `docs/workflows/新项目开发工作流/` 中，如何把“需求冻结后，客户新增或修改需求需额外收费，否则不做修改”这一规则并入现有流程，并形成既保护开发方范围与成本、又尽量不损伤客户体验的表达方式。当前阶段先从开发者实际项目经验视角做方案分析，不立即修改工作流文档。

## What I already know

* `docs/workflows/新项目开发工作流/工作流总纲.md` 已明确主链路是“需求冻结 → 意图确认 → 测试设计 → AI 实现 → 自审 → 人工复核 → 证据回归”。
* 同一文件 `§2.4 需求冻结` 已定义冻结对象，强调业务意图、验收证据、接口契约冻结，范围/接口/成本/工期变化时必须进入 `2.5 变更流程`。
* 同一文件 `§2.5 需求变更管理` 已覆盖变更类型、影响评估、审批流、变更定价原则与沟通机制。
* `§2.5.5 变更定价原则` 已说明新增需求按新增计费、修改需求按返工或新增处理，但没有把“冻结后默认不免费改动”明确成前置客户规则与执行门禁。
* `§1.3 项目评估与报价` 已要求给出“变更单价规则”，说明现有工作流允许在更早阶段埋入变更收费基础。

* `docs/workflows/新项目开发工作流/commands/plan.md` 与 `commands/delivery.md` 已存在“门禁式”写法，说明该工作流本身适合把规则写成阶段门禁而不是零散备注。
* `docs/workflows/新项目开发工作流/命令映射.md` 已把“冻结”作为从 brainstorm 进入 design 的关键节点，但未增加“冻结后变更默认进入收费评估”的分支说明。
* `docs/workflows/新项目开发工作流/commands/start-patch-phase-router.md` 已定义 `/trellis:start` 的阶段检测与自然语言触发词，但尚未定义“冻结后任一阶段命中需求修改口语时，优先转入统一变更流程”的全局路由规则。
* `docs/workflows/新项目开发工作流/commands/design.md` 当前仅在“需求可能需要调整”时建议回到 `/trellis:brainstorm`，更像重新讨论需求，而不是进入冻结后的正式变更处理。

## Assumptions (temporary)

* 用户现在更关心的是“先把规则设计对、说顺、落点找准”，不是马上让我直接改文档。
* 最优做法不是强硬增加一句“改需求加钱”，而是把它拆成三个层面：前置预期管理、冻结时确认、冻结后标准变更处理。
* 客户体验的关键不在于是否收费，而在于是否让客户提前知道规则、知道为什么、知道怎么走后续流程。
* 从开发者角度，这条规则的核心目的是控制范围蔓延、返工成本、排期失真和责任边界，而不是借机制造对立。

## Research Notes

### What similar professional practice usually does

* 在外包、定制开发、小团队交付中，常见做法不是承诺“无限次免费改”，而是先冻结 MVP 范围，再通过 change request / change order 处理后续变更。
* 更成熟的做法会把“澄清不收费、范围变更需评估”分开，否则客户会把合理澄清和新增需求混为一谈。
* 实战中最容易出问题的阶段不是开发尾声，而是“客户看到原型或半成品后开始临时改想法”。

### Constraints from this repo/workflow

* 该工作流强调门禁、证据、冻结、验收标准，因此适合把规则写成阶段门禁和文档契约，而不是写成一句零散提醒。
* 当前工作流已经有定价原则与变更流程，不需要新增全新章节体系，更适合补强“默认规则”“客户确认动作”“不执行条件”。

### Feasible approaches here

**Approach A: 三层并入法** (Recommended)

* How it works:
  * 在报价/可行性阶段先披露“冻结后变更需重新评估成本与排期”
  * 在需求冻结阶段让客户显式确认“冻结版本”
  * 在变更管理阶段补上“未确认报价与排期前不进入开发”的硬门禁
  * 在阶段路由中补上“冻结后任一阶段命中需求变更口语 → 优先转入变更流程”的统一跳转
* Pros:
  * 与当前工作流结构最一致
  * 客户体验最好，因为预期是逐步建立的
  * 开发执行最清晰，可直接转成流程节点
* Cons:
  * 文档要多处联动，不是一句补丁能解决

**Approach B: 只在 2.5 变更管理中强化收费规则**

* How it works:
  * 不动前置流程，只在冻结后的变更章节中强调“新增/修改默认收费，否则不做”
* Pros:
  * 改动最少
  * 文档维护成本低
* Cons:
  * 客户会在冻结后才第一次清晰感知这条规则，体验较差
  * 对开发者的预防价值有限

**Approach C: 在客户版 PRD 模板中增加《冻结与变更约定》**

* How it works:
  * 把规则主要做成对外文案模板，流程文档只做简要指向
* Pros:
  * 直接作用于客户沟通
  * 易落地到实际项目沟通中
* Cons:
  * 如果流程层没同步加门禁，内部执行容易走样
