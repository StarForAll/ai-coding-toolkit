# brainstorm: 分析新项目开发工作流中的需求冻结后变更收费场景

## Goal

从开发者实际项目经验视角，分析 `docs/workflows/新项目开发工作流/` 中“需求冻结后发生新增 / 修改 / 删除需求时的收费处理”相关内容是否已经足够、是否存在过度复杂化、是否有重复铺陈或执行负担过高的问题。当前阶段先做内容分析和质疑式审视，不立即修改文档。

## What I already know

* `docs/workflows/新项目开发工作流/工作流总纲.md` 已在 `§2.2` 增加“冻结与变更约定（对客）”，提前向客户说明冻结后变更需先评估范围、排期和费用。
* 同一文件 `§2.4` 已定义冻结对象，并在 `§2.4.1` 明确冻结后任一阶段都先走“澄清 / 正式变更”分流。
* 同一文件 `§2.5` 已覆盖变更管理、审批权限、快速估价公式、定价原则、轻量变更单、完整变更单、对客沟通原则与话术。
* `docs/workflows/新项目开发工作流/命令映射.md` 已把冻结后的新增 / 修改 / 删除统一路由到 `§2.5 需求变更管理`。
* `docs/workflows/新项目开发工作流/完整流程演练.md` 已复述同一条规则，并补充轻量落地示例。
* 多个阶段命令文档（如 `design.md`、`plan.md`、`delivery.md`、`self-review.md`、`check.md`）也都重复写了“冻结后变更先走 2.5”的门禁说明。
* 仓库里已经存在一次类似主题的历史 brainstorm：`.trellis/tasks/archive/2026-03/03-30-brainstorm-freeze-change-fee/prd.md`，其中已明确识别出“规则存在，但可能需要更前置、更门禁化，而不是仅在 2.5 强化”。

## Assumptions (temporary)

* 当前主要问题不是“没有收费规则”，而是“规则是否写得过重、过散、过重复”。
* 从个人开发者视角，最重要的是防止范围偷偷混入，而不是把价格机制写成一套过于精密的制度。
* 用户这次更关心“质疑式内容分析”，而不是马上形成一版重写方案。

## Open Questions

* 这套内容在当前仓库中的主要问题，是“复杂度过高”，还是“主纲与分支重复过多”？
* 是否需要把“快速估价公式 / AI 系数 / 完整变更单”降级为可选附录，而不是主流程核心内容？

## Requirements (evolving)

* 分析现有“冻结后需求修改收费处理”相关内容是否已经覆盖开发者真实工作需要。
* 质疑现有内容中可能过度复杂、过度制度化、过度重复的部分。
* 从开发者视角判断哪些内容是必要最小集合，哪些内容是锦上添花甚至增加维护负担。
* 输出应偏简洁，不做过深的流程再设计。

## Acceptance Criteria (evolving)

* [ ] 能指出当前相关内容的主要落点和分布位置。
* [ ] 能明确判断“是否完善”与“是否过复杂”。
* [ ] 能给出至少 2-3 个值得质疑的复杂点或冗余点。
* [ ] 能提出一个更偏轻量化的收敛方向。

## Definition of Done (team quality bar)

* 基于现有文档内容做出有依据的分析，而不是凭印象判断。
* 结论能回链到具体文档位置。
* 不新增实现层改动。

## Out of Scope (explicit)

* 不直接修改 `docs/workflows/新项目开发工作流/` 中的文档。
* 不设计完整的新收费体系或新命令体系。
* 不扩展到合同法、商务谈判或法务模板层面。

## Technical Notes

* 主要分析目标：
  * `docs/workflows/新项目开发工作流/工作流总纲.md`
  * `docs/workflows/新项目开发工作流/完整流程演练.md`
  * `docs/workflows/新项目开发工作流/命令映射.md`
  * `docs/workflows/新项目开发工作流/commands/*.md`
* 历史参考：
  * `.trellis/tasks/archive/2026-03/03-30-brainstorm-freeze-change-fee/prd.md`
* 当前初步观察：
  * 规则主线已经存在，甚至已被扩散到多个阶段与样例文件。
  * 潜在问题更像“文档膨胀”和“执行复杂度”而非“规则缺失”。
* 本轮新增细化判断：
  * `§2.5.1 变更类型定义` 对读者有帮助，但可以压缩，不必展开成完整说明体系。
  * `§2.5.2 重大变更影响评估` 可以保留，但 AI 资产因子更适合降级为可选补充，不宜放在主流程里形成阅读负担。
  * `§2.5.3 变更处理流程` 是真正的核心，应保留。
  * `§2.5.3.1 变更执行 checklist` 与 `§2.5.3.2 变更验收标准` 更像执行附表，可降级为可选模板或收进演练/附录。
  * `§2.5.4 变更审批权限（个人开发版）` 制度感偏重，可大幅简化或并入流程路由说明。
  * `§2.5.5 快速估价公式` 可以降级为经验参考；其中阶段系数、复杂度系数、AI 系数容易形成伪精确。
  * `§2.5.6 轻量/完整变更单模板` 建议保留轻量版作为默认，完整版降级为重大变更可选模板。
  * `§2.5.7 对客沟通原则与话术` 有价值，但不必在主纲展开过多，适合压缩成 1 条原则 + 1 个短模板，或迁移到演练/附录。
