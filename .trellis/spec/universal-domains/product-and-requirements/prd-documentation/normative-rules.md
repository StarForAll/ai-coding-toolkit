# Normative Rules

* A PRD documentation set must declare whether it is customer-facing, developer-facing, or a paired set of both documents.
* Shared requirement facts must remain consistent across PRD variants, including the problem statement, target users, business goal, in-scope items, out-of-scope items, and success criteria.
* Every PRD must separate confirmed requirements from assumptions, open questions, and pending decisions.
* Every PRD must state explicit scope boundaries, including in-scope and out-of-scope items.
* If a customer-facing PRD and a developer-facing PRD both exist, the documentation set must identify how they map to the same requirement baseline.
* Requirement changes after review must be recorded as clarifications, scope changes, or decision updates instead of being silently rewritten into the PRD.
* Audience-specific PRDs must not contradict each other on expected behavior, delivery scope, or success criteria.
* A PRD must not treat implementation detail as a substitute for problem definition or acceptance intent.
