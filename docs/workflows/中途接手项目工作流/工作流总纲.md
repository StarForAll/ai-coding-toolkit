# 中途接手项目工作流 V1.0.0

基于《AI Workflow 项目开发工作流 V1.1.0》的接手场景扩展版，目标是在**文档不全、实现偏差、测试缺失、环境不可复现**的现实条件下，快速建立可验证证据基线，稳定交付节奏，降低接手风险。

---

## 适用范围

- 已经进行中的项目中途交接或二次开发
- 需要在已有项目基础上实现新增需求或修复问题
- 允许或不允许修改框架与代码结构的两类约束场景

---

## 2026 接手场景核心原则

1. 证据优先：先建立事实基线，再做结论与改动
2. 先测试后编码（Test-First）：新增或修复必须先补最小测试
3. 人类最终责任：关键节点人工确认，AI 仅为工具
4. 最小侵入：无证据不做大改造，优先稳态交付
5. 上下文健康：每个阶段产出决策摘要，防止污染

---

## 执行顺序铁律

> 现状证据与测试基线 → 需求对齐 → 接手计划 → 开发循环 → 回归验证 → 验收交付

> 任何跳过证据与验证环节的执行都是不被允许的

---

## 📋 阶段一：接手可行性评估与边界确认

### 1.1 安全基线快速筛查（Secret Scanning + 后门排查）

**目的**：在接手初期尽早发现高危风险，避免后续交付背锅

- Secret Scanning：扫描硬编码密钥、Token、证书、私钥、敏感配置
- 后门排查：异常远程调用、隐藏管理入口、可疑依赖与脚本

**输出物**：
- 《Secret Scanning 报告》
- 《后门排查记录》

### 1.2 合法性与授权
- 合同与授权是否允许继续开发、访问数据与环境
- 不符合合规则停止接手

### 1.3 “结构可改” vs “结构不可改”决策

必须在接手初期形成明确决策并签字确认，**分支路径在此冻结，开发阶段不再变更**：

| 决策项 | 结构可改 | 结构不可改 |
|---|---|---|
| 允许更换框架/架构 | 可讨论 | 禁止 |
| 允许调整目录结构/模块边界 | 可讨论 | 禁止 |
| 允许重构核心模块 | 可讨论 | 禁止 |
| 允许新增功能 | 允许 | 允许 |

> 若未明确，默认视为“结构不可改”，仅允许新增需求实现。

### 1.4 接手范围确认
- 现有功能范围与新增需求范围
- 接手目标：修复、稳定、交付、上线

### 1.5 接手风险预判
- 文档缺失或口径冲突
- 实现偏差与需求不一致
- 测试与验证缺失
- 环境不可复现或依赖不稳定
- 安全或数据迁移风险

### 1.6 交接检查清单（模板）

| 维度 | 检查项 | 结论 | 证据/链接 |
|---|---|---|---|
| 代码 | 仓库地址、分支策略、构建脚本、依赖锁定 | 通过/风险 |  |
| 数据 | 数据库版本、迁移脚本、关键表结构、备份策略 | 通过/风险 |  |
| 权限 | 账号权限、密钥管理、访问控制、审计记录 | 通过/风险 |  |
| 文档 | README/IDD/运维说明/变更记录 | 通过/风险 |  |

### 1.7 输出物
- 《接手评估报告》
- 《接手边界确认记录》
- 《初步风险清单》
- 《交接检查清单》

### 2.1 现状快照
- 代码仓库、分支、依赖版本、构建脚本
- 环境与配置：本地/测试/生产差异

| 环境差异 |  | 高/中/低 |  |
| 数据与接口 |  | 高/中/低 |  |
| CI/发布流程 |  | 高/中/低 |  |

### 2.2 文档盘点与缺口清单
对照 BRD/TAD/DDD/IDD/ODD 等文档体系，输出缺口清单。

| 文档类型 | 是否存在 | 最新版本 | 缺口说明 |
|---|---|---|---|
| 需求文档 |  |  |  |
| 架构文档 |  |  |  |
| 接口文档 |  |  |  |
| 运维文档 |  |  |  |

### 2.3 功能核对与偏差清单
- 需求点 → 实现点 → 证据链接
- 标记偏差、遗漏、行为不一致

### 2.3.x AI 诊断质疑报告（强制）

**规则**：AI 在分析旧项目时**必须以现有 README / IDD / 代码注释为依据**，禁止无证据猜测逻辑。

**输出内容**：
- 代码与文档不一致点
- 逻辑自相矛盾点
- 需要人工确认的“高风险推断点”

**交付物**：《AI 诊断质疑报告》

### 2.4 测试与质量盘点
- 是否存在可运行测试集
- Lint/TypeCheck/CI 是否可用
- 现有门禁覆盖的功能范围

### 2.5 证据基线建立
- 运行现有测试，记录通过/失败
- 对关键流程建立最小回归用例
- 基线作为后续变更的“安全网”
- 日志与指标采集基线（日志结构、关键指标、告警阈值）

### 2.6 输出物
- 《现状快照》
- 《文档缺口清单》
- 《功能偏差清单》
- 《基线回归用例集》
- 《日志与指标基线记录》

## 🎯 阶段三：需求对齐与接手版文档

### 3.1 需求澄清循环
- 现有需求 → 澄清 → 确认
- 形成“歧义问题列表”并逐条关闭

### 3.2 接手版文档输出
- 需求方版需求说明书（范围与验收）
- 开发版需求说明书（技术约束与实现边界）
- **新增：结构约束清单**（结构不可改时必须输出）

### 3.3 需求冻结
冻结对象：
- 业务意图
- 验收证据
- 接口与数据契约
- 结构变更策略（可改/不可改）

### 3.4 变更管理
- 接手后新增需求必须走变更流程
- 涉及结构调整必须走结构变更评审

### 3.5 输出物
- 《接手版PRD》
- 《接手版技术约束说明书》
- 《变更记录》

## 📝 阶段四：任务拆解与接手计划

### 4.1 Plan.md 机制
- 以接手版文档为输入，生成 task_plan.md
- 每个任务明确：范围、风险、验收、回归范围、时间/人力估算

**task_plan.md 字段要求**：
- 范围与边界
- 风险级别与风险点
- 验收标准与回归范围
- 时间/人力估算（含评审与回归成本）

### 4.2 DoR / DoD 门槛
**DoR（可量化）**：
- 需求范围与边界已确认并有版本号
- 验收标准 ≥ 1 条/功能点
- 接口/数据契约已确认（给出文档链接或版本号）
- 关键回归用例 ≥ 1 条且可运行
- 日志与指标基线已记录（至少 3 项关键指标）
- 风险级别已标注（L0/L1/L2）
- 时间/人力估算已完成

**DoD（可量化）**：
- 功能验收通过率 100%
- 绑定回归用例通过率 100%
- 本地门禁通过（Lint/TypeCheck/UnitTest 全绿）
- 关键日志与指标不劣化（对比基线）
- 文档同步完成（需求/接口/变更记录）
- 结构约束未被破坏（结构不可改时必须满足）

### 4.3 风险分级与时间盒
- L0 低风险：自动门禁为主
- L1 中风险：关键节点人工确认
- L2 高风险：完整人工评审与安全复核

### 4.4 输出物
- `task_plan.md`
- 里程碑计划与风险清单

## 🔄 阶段五：开发实施循环（双分支）

### 5.1 通用执行流程
1. DoR 检查通过
2. 先测后改（补最小测试）
3. 实现与自审（AI 可参与，必须100%人工复核）
4. 本地门禁（Lint/TypeCheck/UnitTest）
5. 风险评审与人工复核
6. CI 回归与合并
7. 输出决策摘要与偏差清单

### 5.1.x AI 辅助约束（如使用）

- AI 仅用于辅助分析与代码草拟，**所有输出必须100%人工复核**后才可提交
- 人工复核必须覆盖：行为一致性、接口兼容性、回归用例、结构约束与性能不劣化（如适用）

### 5.2 分支A：结构可改流程

**分支来源**：由 1.3 阶段决策确认，开发阶段不再切换

核心要求：
- 结构变更需先出“重构计划”与“回滚方案”
- 引入兼容层，优先渐进式迁移
- 任何结构调整必须绑定回归集

结构变更评审流程：
1. 现状证据与痛点说明
2. 目标架构与迁移路径
3. 影响范围评估与回滚方案
4. 人工评审批准

### 5.3 分支B：结构不可改，仅新增需求实现

**分支来源**：由 1.3 阶段决策确认，开发阶段不再切换

执行约束：
- 输出《结构冻结清单》
- 仅允许在既有扩展点内新增功能
- 优先使用适配层、配置扩展、Feature Flag
- 禁止修改核心模块边界与目录结构

例外升级机制：
- 如新增需求无法实现，必须提出结构变更申请
- 评估范围与风险后由需求方确认是否破例

### 5.4 文档同步与追溯
- 每个任务完成后同步更新相关文档
- 关键决策写入“接手决策日志”

### 6.1 自动化测试

### 6.2 可用性与场景测试
- 核心业务流程验证

- 核心场景 100% 通过
- P0/P1 缺陷为 0
- 回归通过率达标
- 结构约束与变更记录完整

## ✅ 常见问题与处理策略（高频优先）

| 高频问题 | 处理策略 |
|---|---|
| 文档不全/口径冲突 | 先盘点缺口 → 形成接手版文档 → 冻结验收证据 |
| 实现偏差/功能错位 | 建立需求-实现矩阵 → 定义偏差清单 → 先补最小测试 |
| 测试缺失 | 先补最小回归 → 再改代码 |
| 环境不可复现 | 输出环境快照 → 锁定版本 → 建立可运行基线 |
| 结构不可改约束 | 冻结结构清单 → 仅走扩展点与适配层 |
