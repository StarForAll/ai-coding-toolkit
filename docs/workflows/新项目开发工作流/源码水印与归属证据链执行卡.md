# 源码水印与归属证据链执行卡

> 用途：把“作者归属保护”从抽象建议落成当前 workflow 中可设计、可拆任务、可校验、可交付的一组标准动作。

---

## 适用范围

适用于以下场景：

- 个人开发项目
- 需要交付源码或可执行产物的项目
- 希望降低“被冒名顶替 / 被删署名后二次发布 / 被改头换面洗稿”的风险

不适用于以下误用：

- 试图依靠单一水印达成“绝对不可伪造”
- 未披露的后门、隐藏 kill switch、远程锁定
- 会改变业务逻辑、权限判断、数据正确性的“水印”

说明：

- 当前执行卡不依赖额外的 `trellis-library` watermark spec 资产
- 它通过 workflow 阶段文档、任务产物和 `ownership-proof-validate.py` 形成可执行门禁

---

## 核心原则

1. 水印只是归属证据链的一层，不能替代法律确权、签名、时间戳和交付记录。
2. 默认采用**轻量但多层**的源码水印，而不是默认启用高侵入 AST / 动态运行时水印。
3. 零宽字符水印只能放在**非执行语义载体**里：注释、文档字符串、Markdown；不得放进标识符、路径、配置键、SQL、执行脚本关键字。
4. “不起眼代码标识”只能放在**稳定、低风险、非业务关键**位置；不得污染权限逻辑、接口契约、数据库值或性能热点。
5. 所有水印策略都必须可被提取、可被说明、可被验证。

## 默认分层模型

### Level 语义

| `source_watermark_level` | 含义 | 最低要求 |
|---|---|---|
| `none` | 不启用源码水印与归属证明门禁 | 不要求 |
| `basic` | 最低档：至少要求可见源码水印 | `visible` 必须启用；其余通道按需选择 |
| `hybrid` | 默认推荐档：可见水印 + 若干隐蔽辅助层 | `visible` 必须启用；其他通道按需组合 |
| `forensic` | 取证强化档：尽量保留多层水印与证明 | `visible` 必须启用；建议启用全部已确认通道 |

说明：

- 当前 workflow 中，是否真正启用归属证明门禁，仍以 `ownership_proof_required` 为准。
- 非 `none` 档位若启用了归属证明门禁，`source_watermark_channels` 必须至少包含 `visible`。
- 其他通道是否校验，以 `source_watermark_channels` 实际声明为准，而不是只看 level 名称。
- `basic` / `hybrid` / `forensic` 在当前 workflow 里不是三套不同的 validator 代码路径；它们主要表达策略强度与默认期望，真正决定 design / plan / delivery 检查范围的仍是 `source_watermark_channels`。

### W0：可见源码水印

用于直接宣示归属。

最低要求：

- 核心源码文件头包含版权 / 作者 / 年份
- 包含稳定的 `WMID`

示例：

```text
/*
 * Copyright (c) 2026 <Author>
 * Project: <project-slug>
 * WMID: wm_<stable-id>   # 例如 short-hash，也可以是项目定义的稳定标识
 * Ownership: see NOTICE / ownership-proof.md
 */
```

### W1：零宽字符水印

用于增加删除成本，但不承担唯一证明责任。

允许位置：

- 注释块
- 文档字符串
- 项目内 Markdown 说明片段

禁止位置：

- 标识符、变量名、函数名、类名
- import/export 路径
- JSON/YAML/TOML key
- SQL / shell / 正则表达式主体
- 任何会参与真实执行语义解析的位置

实施要求：

- 不只放一个点，至少做 2 到 3 处分片冗余
- 在 `source-watermark-plan.md` 中记录使用的字符集合、嵌入规则和提取方式

### W2：不起眼代码标识

用于在源码中留下不显眼但可解释的归属片段。

推荐形式：

- 模块私有常量片段
- 说明性注释碎片
- 合法 metadata 片段
- 无害初始化表达式片段

禁止形式：

- 业务关键分支
- 权限判断
- API 契约字段
- 数据库存储值
- 编译器 / formatter 易重写的脆弱位置

### W3：零水印记录

不继续改源码语义，只记录：

- 选择了哪些文件
- 哪些片段组成完整 `WMID`
- 对应 checksum / hash
- 提取步骤与验证步骤

## 阶段动作

### 1. feasibility

在 `assessment.md` 中至少冻结：

- `source_watermark_level`
- `source_watermark_channels`
- `zero_width_watermark_enabled`
- `subtle_code_marker_enabled`
- `ownership_proof_required`

### 2. design

在 `$TASK_DIR/design/source-watermark-plan.md` 中正式设计：

- `WMID`
- 采用哪些通道：`visible / zero-width / subtle-markers / zero-watermark`
- 零宽字符可用边界与禁区
- 不起眼代码标识的嵌入位置与禁区
- `excluded_paths`
- 提取步骤
- 验证步骤

### 3. plan

至少拆出以下 Trellis task：

- `可见源码水印任务`
- `零宽字符水印任务`（若启用）
- `隐蔽代码标识任务`（若启用）
- `水印验证任务`
- `归属证明包任务`

### 4. delivery

在 `$TASK_DIR/delivery/` 至少产出：

- `ownership-proof.md`
- `source-watermark-verification.md`
- `deliverables.md` 中列出归属证明交付物
- `transfer-checklist.md` 中列出归属证明检查项

建议至少记录：

- `WMID`

建议至少记录：

- 可见源码水印验证结果

- 隐蔽代码标识验证结果（若启用）
- 零水印 / 指纹验证结果

## 现实边界

- 零宽字符和不起眼代码标识都可能被格式化、重构、重写或重新实现冲掉。
- 因此，推荐始终把“源码水印”与以下证据一起使用：
  - 版权声明 / `NOTICE`
  - Git commit / tag 签名
  - checksum
  - 时间戳 / 存证
  - 交付时的归属证明文档

这张执行卡的目标不是制造“删不掉的魔法水印”，而是把攻击者的抹除成本拉高，并留下足够强的解释链。
