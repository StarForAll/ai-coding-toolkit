基于AI辅助开发实践的实战工作流，目标是：更快交付、可验证正确、可控成本、可控风险。

配套 walkthrough 见：[双轨交付控制完整流程演练](./完整流程演练.md)

## 2026 核心方法论升级

> 本文档是 **AI 辅助开发实战工作流**，目标：更快交付、可验证正确、可控成本、可控风险。

### 核心理念升级（2026）

基于 2025 年 AI Coding 领域的实践经验，我们升级以下核心方法论：

#### 1. 规划先行 + 证据门禁

- **所有 AI 参与环节必须先规范后实现**
- 不允许"边想边做"的无序 AI 消耗
- 每个 AI 输出必须有对应的验证证据

#### 2. 先测试后编码

- 践行 **Test-First AI** 原则
- 测试用例是验收标准的客观载体
- AI 生成代码必须通过预先定义的测试集

#### 3. 自纠错反射

- 引入"广义自审"机制
- AI 实现后必须对照 spec 自检
- 输出"偏差清单"供人工快速确认

#### 4. 人类最终责任

- 明确 AI 是"工具"而非"决策者"
- 关键节点必须"人类在环"
- 所有输出的人类最终责任不可转移

### 执行顺序铁律

> 需求冻结 → 意图确认 → 测试设计 → AI 实现 → 自审 → 人工复核 → 证据回归

> **任何跳过验证环节的 AI 执行都是不被允许的**

---

适用模式：个人开发 / 小团队协作（单人项目中"项目经理/评审者"可由开发者本人承担，客户负责需求与验收确认）。

## 📋 阶段一：项目可行性评估

### 1.1 合法性审查
- **法律合规性检查**：确认项目在当地法律法规框架下是否合规
- **决策点**：如果不符合法律法规，**终止项目**，不进行任何后续工作

### 1.2 粗略需求沟通
- 与客户进行初步沟通，明确大致需求方向和范围
- 识别核心功能点和技术难点

### 1.3 项目评估与报价
- 工作量评估：功能开发工时 + 评审工时 + 测试工时 + 文档工时
- 时间成本分析：按里程碑拆分，给出最早/最晚交付窗口
- 资源成本核算：人力、技术资源投入成本
- AI相关成本核算：
  - Token成本：纳入项目整体成本规划，合理估算
  - 重试预算：根据功能复杂度预留合理冗余
  - 缓存收益：合理设计前缀以优化成本
  - 评估成本：纳入整体成本规划
- 风险预判：
  - 技术风险、进度风险、依赖风险
  - AI特有风险：幻觉、提示词注入、上下文污染、模型不可用或限流
- 报价输出：给出基础版范围、变更单价规则、交付SLO草案

#### 1.3.1 进入需求发现前的门禁

- 对于**新项目 / 新客户需求 / 外包接单场景**，在进入阶段二的详细需求分析或 `/trellis:brainstorm` 前，必须先完成一次可行性评估。
- 可行性评估应至少产出 `assessment.md`，记录：
  - 总体决策：接 / 谈判后接 / 暂停 / 拒绝
  - 是否可做、是否值得做
  - 关键风险、必须谈判条件、最小补充信息集
  - 是否允许进入 `brainstorm`
- **风险分析执行**：
  - 使用 `/trellis:feasibility` 命令的 `--step risk-analysis` 步骤
  - 加载 `demand-risk-assessment` skill 执行结构化风险评估
  - 评估流程：结构化抽取 → 踩坑信号扫描 → 冲突检测 → 红线检查 → 结构化评分 → Pre-mortem → 风险登记表 → 决策/谈判条件
  - 将分析结果写入 `assessment.md`，确保包含完整的风险评估结论
- 若结论为 `暂停`，应先补齐缺口后重新执行可行性评估，不应直接进入阶段二。
- 若结论为 `拒绝`，应终止该项目链路，仅保留评估记录。
- 只有 `assessment.md` 明确允许进入 `brainstorm`，或存在经确认仍有效的历史评估记录时，才应进入阶段二。

### 1.4 项目确认与初始化
- **客户确认报价**：与客户就报价方案达成一致
- **签署合同/协议**：明确双方权责
- **项目级初始化**：
  - 报价成功且项目确定启动后，进行项目状态初始化
  - 建立项目档案和后续追踪机制
  - 选择合适的AI开发工具链
  - 建立提示词模板库和上下文管理规范
  - 若已嵌入对应的自定义工作流，**立即补充 `trellis-library` 中的需求发现基础资产**
    - 默认补充：`pack.requirements-discovery-foundation`
    - 最低要求：至少补齐需求发现核心规范 `problem-definition`、`scope-boundary`、`requirement-clarification`、`acceptance-criteria`，再补齐面向客户与面向开发人员的 PRD 规范、模板、检查清单
    - 目的：在正式进入需求发现前，先建立统一的 PRD 约束基线，避免后续只有流程没有规范

#### 1.4.1 外部项目交付控制原则

对于外包、定制开发、新客户项目，默认采用以下"双轨交付控制"：

- **首选轨：托管部署**
  - 尾款前只提供由开发者控制的演示/试运行环境，客户验证功能与流程
  - 尾款前不移交源码仓库、生产环境密钥、管理员账号、最终部署权限
  - 尾款到账后，再执行源码、部署权限、密钥、账号、运维材料的正式移交
- **备选轨：试运行授权**
  - 仅当客户明确要求先拿可运行版本，且双方接受授权方案时使用
  - 必须在报价、合同、交付说明、验收口径中明确披露"试运行授权"的期限、续期方式、到期行为、永久授权触发条件
  - 到期行为必须可预期，优先限制为"演示模式"或"只读模式"，不得破坏已有数据
  - 尾款到账后，交付永久授权、源码及最终控制权

禁止项：

- 不得植入未披露的后门、隐藏开关、远程锁定或暗门控制逻辑
- 不得使用会破坏数据、阻断导出、导致客户无法恢复业务的失效机制
- 不得使用"不可解码交付文件"伪装为正常源码交付
- 若已明确交付的是试运行版，不得在流程或文档中表述为"正式无限制交付"

**Spec 导入要求（外部项目专用）**：

进入设计阶段（`/trellis:design`）后，技术架构确认前，必须从 `trellis-library` 导入以下交付控制相关 spec 到当前项目 `.trellis/spec/`：

- 外部项目基础必选：
  - `spec.universal-domains.project-governance.delivery-control`（交付控制总控规范）
  - `checklist.universal-domains.project-governance.transfer-checklist`（交付事件 checklist）
- 仅当交付轨道为 `trial_authorization` 时额外导入：
  - `spec.universal-domains.project-governance.authorization-management`（试运行授权规范）
- 仅当本项目会发生密钥、环境变量、第三方平台配置等正式移交时，额外导入：
  - `spec.universal-domains.security.secrets-and-config`

这些 spec 定义了双轨交付控制的详细规则、验证方法和合规检查点。内部项目（非外部交付）无需导入。

**双轨交付控制速查表**：

| 决策点 | `hosted_deployment` | `trial_authorization` |
|---|---|---|
| 适用前提 | 客户接受尾款前仅使用托管环境 | 客户明确要求先拿可运行版本，且双方接受授权条款 |
| 机器字段 | `delivery_control_track = hosted_deployment` | `delivery_control_track = trial_authorization` |
| 必选资产 | `delivery-control` + `transfer-checklist` | `delivery-control` + `transfer-checklist` + `authorization-management` |
| 条件资产 | 正式移交时若涉及密钥/配置，再加 `secrets-and-config` | 正式移交时若涉及密钥/配置，再加 `secrets-and-config` |
| 尾款前默认保留 | 源码、生产密钥、管理员账号、最终部署控制权 | 永久授权、完整源码、最终控制权；并明确试运行限制 |
| 最终移交触发 | 一般为 `final_payment_received` | 一般为 `final_payment_received` |

> 此阶段体现了"先调研再实现"的方法论，确保项目在启动前已充分评估。

## 🎯 阶段二：需求明确与确认

> **前置条件**：新项目默认已完成阶段一可行性评估，并且 `assessment.md` 明确允许进入需求发现；否则应先回到 `/trellis:feasibility` 补做评估，而不是直接开始 `brainstorm`。

### 2.1 详细需求分析
- **需求沟通流程**：与客户讨论 → 手动填写需求框架 → AI优化完善 → 客户确认
- 深入调研，与客户充分沟通需求细节
- 分析需求的完整性，进行需求完整性评分
- 识别缺失信息、隐含假设、边界条件
- **AI可实现性评估**：评估需求的可AI化程度，区分可直接AI生成、需要人工设计、需要RAG增强、需要Agent架构等类型

### 2.2 需求说明书编制

#### 客户版需求说明书（对外）

形成《客户版需求说明书》(PRD)，仅包含功能描述，不含AI技术细节。

默认对齐以下 `trellis-library` 资产：

- `spec.universal-domains.product-and-requirements.problem-definition`
- `spec.universal-domains.product-and-requirements.scope-boundary`
- `spec.universal-domains.product-and-requirements.requirement-clarification`
- `spec.universal-domains.product-and-requirements.acceptance-criteria`
- `spec.universal-domains.product-and-requirements.prd-documentation-customer-facing`
- `template.universal-domains.product-and-requirements.customer-facing-prd-template`
- `checklist.universal-domains.product-and-requirements.customer-facing-prd-checklist`

说明：以上 `spec.*` ID 指向一个 concern directory，实际展开为 `overview.md`、`scope-boundary.md`、`normative-rules.md`、`verification.md` 四个子文件。

##### 项目规则定义

- **架构边界**：
  - 系统边界：明确系统与外部系统的交互范围
  - 模块边界：各功能模块之间的职责划分
  - 技术边界：禁止使用的技术/框架

- **质量门槛**：
  - 性能要求：响应时间、加载速度
  - 可用性要求：在线时长、故障恢复
  - 安全要求：数据安全等级、权限要求

- **兼容性要求**：
  - 浏览器兼容：支持的浏览器版本
  - 设备兼容：支持的手机/平板尺寸
  - 系统兼容：支持的操作系统版本

##### 用户故事

格式：作为[角色]，我希望[功能]，以便[价值]

##### 功能边界

- **核心功能**：必须实现的功能
- **扩展功能**：可选实现的功能
- **明确不做**：主动排除的功能范围

##### 验收标准

- 功能验收：每个功能的验收条件
- 性能验收：性能指标达标标准
- 体验验收：用户操作体验要求

##### 模糊点澄清记录

- 歧义问题1：问题描述 → 澄清结果
- 歧义问题2：问题描述 → 澄清结果
- 遗留问题：暂时无法确定的问题及后续跟进计划

##### 冻结与变更约定（对客）

客户版 PRD 建议新增一个简短且固定的“冻结与变更约定”小节，用于提前管理预期，而不是等客户在后续阶段提变更时才第一次说明。

至少要写清三件事：

- 本轮冻结的范围是什么
- 冻结后“纯澄清”和“正式变更”的区别
- 正式变更会先评估影响范围、工期和费用，确认后再并入当前项目

推荐对客写法：

> 本文档确认的是本轮交付范围与验收标准。为保证当前版本按时交付，需求冻结后，如有新增、修改或删除需求，我们会先评估对范围、排期和费用的影响，并在你确认后再纳入执行；如果只是对原有需求的澄清，且不改变范围和验收标准，我们会直接在当前阶段处理。

#### 2.2.1 技术开发人员文档切分规范

##### 文档版本规范

- 每个文档独立版本号（如 TAD v1.0.0）

- 文档头部包含：文档编号、版本号、创建日期、最后更新日期、作者

##### 项目规则定义

- **架构边界**：
  - 系统边界：API接口定义、集成方式
  - 模块边界：模块职责、依赖关系
  - 技术边界：技术栈限制、版本要求

- **命名规则**：
  - 变量命名：代码中的命名规范
  - 数据库命名：表名、字段名规范
  - 接口命名：API路径、参数规范
  - 文件命名：项目文件组织规范

- **质量门槛**：
  - 代码质量：项目自动化检查矩阵（静态分析/类型检查/构建/质量平台等）与注释要求
  - 测试覆盖：单元测试覆盖率要求
  - 性能指标：具体的技术性能指标

- **兼容性要求**：
  - 浏览器兼容：各浏览器版本支持情况
  - 后端兼容：各语言/框架版本要求
  - 数据库兼容：数据库版本要求

##### 模块结构说明

- 模块划分：各模块名称及职责
- 层次结构：MVC/Vue/React等架构分层
- 目录结构：项目文件组织方式

##### 技术方案约束

- 技术选型：具体技术栈及版本
- 第三方库：引入的第三方库清单
- 设计模式：使用的设计模式规范
- 代码规范：具体的编码规范要求

##### 数据与接口约束

- 数据模型：数据库表结构定义
- 接口规范：API接口设计规范
- 通信协议：HTTP/WebSocket/GraphQL等
- 数据格式：JSON/XML等数据格式要求

##### 用户故事（含技术实现要点）

每个用户故事附带：实现要点、技术难点、风险点

##### 功能边界（含技术边界说明）

- 核心功能：技术实现方案
- 扩展功能：技术实现方案
- 不做功能：技术原因说明

##### 验收标准（含测试要点）

- 功能验收：测试方法、测试用例
- 性能验收：性能测试方法、达标指标
- 安全验收：安全测试方法
- 集成验收：集成测试方案

##### AI实现要求

- AI模型选型要求
- 上下文窗口限制
- 提示词设计约束
- AI生成代码的审查级别要求
- AI性能指标要求

#### 客户版确认
```

     ↑                                              ↓
     └───────────── 循环直到完全确认 ←──────────────┘
```
- 仅确认功能描述和验收标准

#### 开发人员版确认
```
开发版需求说明书 → 技术审核 → 反馈修改 → 再次确认
     ↑                                           ↓
     └───────────── 循环直到完全确认 ←────────────┘
```
- 确认功能 + AI实现方案 + 技术约束

### 2.4 需求冻结

> **核心原则：冻结的是"业务意图与验收证据"，而非"实现细节"**

#### 冻结对象（明确）

| 冻结内容 | 冻结程度 | 变更方式 |
|---------|---------|---------|
| **业务意图** | 绝对冻结 | 需走完整变更流程 |
| **验收证据** | 绝对冻结 | 需新增证据而非修改 |
| **接口契约** | 绝对冻结 | 需走技术变更评审 |
| ~~Prompt细节~~ | 不冻结 | 可持续优化 |
| ~~模型参数~~ | 不冻结 | 可调优 |

- 冻结对象：
  - 客户版PRD（业务范围与验收标准）
  - 开发版技术约束（接口契约、数据契约、非功能约束）
  - 评估基线（核心测试集与通过阈值）
- 非冻结对象：
  - Prompt细节、模型参数、工具调用策略允许迭代优化
- 优化前提：
  - 每次Prompt或模型调整必须记录变更原因
  - 必须通过评估基线回归后才能进入下一阶段
  - **Prompt迭代必须在评估基线回归后才能生效，不允许直接上线未评估的Prompt变更**
  - **模型版本升级视同重大变更，需重跑全量评估集**
  - **工具调用策略变更若影响输出格式或接口契约，必须进入2.5变更流程**
- 后续变更：
  - 涉及范围、接口、成本、工期变化时，必须进入2.5变更流程

#### 2.4.1 冻结后的跨阶段分流规则

从需求冻结完成开始，后续任一阶段（设计、计划、测试先行、实施、自审、交付）如再次出现需求讨论，先按“澄清 / 变更”分流，不得在当前阶段直接混入处理。

- **纯澄清**：允许留在当前阶段直接处理
  - 前提：不改变范围、接口契约、验收标准、成本、工期
  - 典型情况：解释原有需求含义、补充术语说明、确认已有描述指向什么行为
- **正式变更**：统一进入 `2.5 需求变更管理`
  - 触发类型：新增需求、修改需求、删除需求
  - 处理要求：先评估影响范围、成本、工期、测试与回归范围；未确认前，不并入当前阶段继续推进

#### 2.4.2 澄清与变更的判定口径

不要按客户是否说出“新增”“修改”字样来判断，而按影响来判断。

只要命中以下任一项，就视为正式变更：

- 功能范围发生变化
- 页面或业务流程行为发生变化
- 接口契约或数据契约发生变化
- 验收标准或测试证据发生变化
- 成本发生变化
- 工期或里程碑发生变化

只有在**不改变**上述六项时，才可视为“纯澄清”，留在当前阶段处理。

#### 2.4.3 典型口语触发示例

以下口语表达，若发生在需求冻结后，默认按正式变更处理：

- “这个功能再加一个”
- “这个页面流程我要改一下”
- “这个接口字段得调整”
- “这个验收标准我想改”
- “这个功能先不做了”

以下口语表达，若不改变范围和验收，可按纯澄清处理：

- “这里我确认一下是什么意思”
- “这个描述对应的是不是原来那个流程”
- “我不是加功能，只是想确认你理解的是否和我一致”

#### 2.4.4 冻结后变更的执行出口

- 当前阶段的责任是**识别并中断**，而不是直接吸收正式变更继续推进
- 若判断为正式变更，先执行 `2.5 需求变更管理`
- 若变更获批，再回到**受影响的最早阶段**继续，例如：
  - 影响业务范围或验收标准：回到需求/设计层
  - 影响任务拆解、排期、测试范围：回到计划或测试先行
  - 仅在批准后影响实现与验证：回到实施或后续验证阶段

#### 2.4.x AI 意图映射确认（新增）

在需求冻结前，增加"AI 意图映射确认"循环：

```
AI 意图映射流程：

① 需求冻结触发
    └── 开发者将客户需求输入 AI

② AI 生成结构化意图列表
    └── 格式：
    - 用户故事 → 核心意图 → 关键约束 → 验收证据

    示例：
    | 用户故事 | 核心意图 | 关键约束 | 验收证据 |
    |---------|---------|---------|---------|
    | 作为用户，我希望能... | 实现xxx功能 | 响应时间<200ms | 性能测试通过 |

③ 开发者复核
    └── 核对 AI 理解与客户描述是否一致
    └── 标记潜在误解或假设

④ 客户签字确认
    └── 客户确认意图理解正确
    └── 确认后纳入冻结范围

⑤ 后续争议仲裁
    └── 如实现过程中出现理解偏差
    └── 以 Intent.md 中签字确认的意图为准
```

> 这一阶段体现了"证据优先"和充分调研的原则，为后续工作奠定基础。

### 2.5 需求变更管理

当客户在需求冻结后提出新增需求、修改需求或删除需求时，需要通过规范化的变更管理流程进行处理。纯澄清若不改变范围、接口契约、验收标准、成本、工期，可留在当前阶段直接处理，不必升级为正式变更。

#### 2.5.1 变更类型定义

| 变更类型 | 定义 | 示例 |
|---------|------|------|
| **新增需求** | 在原有需求基础上增加全新的功能模块 | 增加用户评论功能、增加支付方式 |
| **修改需求** | 对已确认的需求内容进行修改 | 修改登录流程、调整页面布局 |
| **删除需求** | 移除已确认的需求功能 | 取消某个不需要的功能 |
| **需求澄清** | 对已有需求的模糊部分进行澄清 | 明确某个功能的详细行为 |

#### 2.5.2 变更影响评估（重大变更可选）

> 本节仅适用于**重大变更**（通常 >2 人天、改接口、改验收标准或明显影响排期）。轻微和一般变更默认走 `§2.5.3` 的轻量闭环。

重大变更建议至少回答这四类问题：

| 评估维度 | 需要回答的问题 |
|---------|-------------|
| **功能影响** | 影响哪些页面、流程、模块或接口？ |
| **技术影响** | 需要改哪些代码或结构？是否有明显技术风险？ |
| **测试影响** | 需要新增/修改哪些测试？回归范围多大？ |
| **项目影响** | 预计增加多少工时？是否影响当前里程碑、费用或交付节奏？ |

如项目明显依赖 AI 资产，可额外备注 Prompt、评估基线或上下文是否需要重建，但默认不要求单独维护一套 AI 系数表。

评估结论统一收敛为：**接受 / 有条件接受 / 拒绝**。

#### 2.5.3 变更处理流程

变更处理只保留一个核心闭环：**估价 → 确认 → 执行 → 验证**。

**核心流程（所有正式变更必走）**：

```text
① 估价
   └── 给出时间、费用、是否影响当前交付的快速判断

② 确认
   └── 客户选择：
       接受并执行 / 暂缓 / 拒绝

③ 执行
   └── 按确认结果更新 PRD、计划、实现与测试

④ 验证
   └── 完成变更验证与必要回归，再由客户确认
```

**按级别路由**：

| 变更级别 | 估价范围 | 流程路由 |
|---------|---------|---------|
| **轻微** | ≤0.5天 | 走完①②③④，口头确认即可，事后补一行记录 |
| **一般** | 0.5-2天 | 走完①②③④，使用**轻量变更单**（§2.5.5） |
| **重大** | >2天 或明显影响接口/验收/排期 | 走完①②③④，补充 **§2.5.2 影响评估**，并使用**完整变更单**（§2.5.5）或等效书面记录 |
| **紧急** | 不限 | 先止血，24小时内补齐①②③④记录 |

> 所有正式变更建议保留文字记录（微信/邮件/变更单均可），避免口头约定产生歧义。

**变更流程结束后的返回原则**：

- **未获批 / 暂缓 / 拒绝**：保持当前阶段冻结基线，不把变更混入当前交付
- **已获批**：回到受影响的最早阶段，更新 PRD / 设计 / 计划 / 测试 / 实施基线后，再恢复主链路

#### 2.5.4 变更估价参考（可选）

快速估价的目标是**尽快报出可执行判断**，不是制造一套看起来精确但难以落地的计算公式。

建议按这个顺序估：

1. 先估算新增或返工的**基准工时**
2. 再判断当前阶段的**返工成本**（越晚提出，通常越贵）
3. 再判断是否存在**跨模块 / 跨层连锁影响**
4. 最后补上**测试与回归成本**

对外通常只需要明确三件事：

- 预计新增工期
- 预计新增费用
- 是否影响当前交付版本

> 如果短时间内无法给出精确数值，可以先给**区间报价**，不要为了“看起来专业”强行套复杂系数。

#### 2.5.5 变更单模板

个人开发或小团队场景，默认只保留两档模板：**轻量版（默认）** 和 **完整版（重大变更可选）**。

#### 2.5.6 PRD变更更新规范（可选执行细则）

变更获批并执行后，按以下规范更新 PRD 文档：

**版本号规则**
- 格式：`v主版本.次版本.修订号`
- 变更类型对应：
  - 新增/修改需求 → 次版本 +1（如 `v1.2.0 → v1.3.0`）
  - 需求澄清/错误修正 → 修订号 +1（如 `v1.2.0 → v1.2.1`）

**变更标记规范**
- 在 PRD 末尾新增“变更记录”章节
- 每次变更添加一行记录：

| 版本 | 日期 | 变更单号 | 变更内容摘要 | 变更人 |
|------|------|---------|-------------|--------|
| v1.3.0 | 2024-03-20 | CHANGE-003 | 增加订单导出功能 | 张三 |

- 在正文中用高亮标记变更部分：
  > [变更v1.3.0] 新增订单导出功能，支持 Excel 格式下载

**历史版本保留**
- 保留最近 3 个版本的 PRD 文件
- 命名格式：`PRD-项目名称-v版本号-日期.md`
- 归档位置：`/docs/requirements/history/`

#### 2.5.7 对客沟通要点（简版）

冻结后谈变更，重点不是“强调收费”，而是守住三个目标：

- 本轮交付边界不被悄悄改写
- 客户知道“不是不能改，而是先评估再决定”
- 流程透明，可选择，不靠临时口头承诺推进

建议只保留三条表达原则：

- **先接住诉求，再判定澄清还是正式变更**
- **先说明不直接混入当前交付，再谈时间和费用**
- **给客户明确选项：本轮不动 / 并入本轮并调整排期 / 留到下一轮**

推荐短话术：

> 这个想法可以做。为了不影响当前已经确认的交付，我先把它按正式变更评估一下，给你时间、费用和是否影响本轮排期的判断；你确认后，我再决定并入本轮还是排到下一轮。

### 3.7 技术架构确认后的项目 Spec 对齐

> **触发条件**：技术架构已经被用户明确确认。

在进入任务拆解前，必须先完成当前项目 `.trellis/spec/` 的项目化对齐。这个动作不是首次嵌入工作流时的基础资产门禁，而是技术架构确认后的第二道门禁。

必须按以下顺序执行：

1. **从 `trellis-library` 导入合适 spec**
   - 根据已确认的技术架构、技术栈、部署方式、数据与接口约束
   - 从 `trellis-library` 中选择适配当前项目的 spec
   - 添加到当前项目 `.trellis/spec/`

2. **对当前项目 `.trellis/spec/` 做项目化完善**
   - 结合当前项目的作用、背景、业务边界、技术架构
   - 分析现有 `.trellis/spec/` 内容是否准确
   - 删除错误、不适用或冲突的内容
   - 补充缺失的项目约束、边界、验证要求

3. **明确项目自动化检查矩阵**
   - 基于已确认的语言、框架、包管理器、部署方式、CI 能力与质量平台，写出当前项目真实执行的自动化检查清单
   - 每个检查项至少明确：检查类别、工具/命令、执行时机、通过标准、失败后的处理动作
   - 不再允许使用“默认 Lint”“基础检查已覆盖”这类模糊表述；必须写出项目实际会跑的检查
   - 若项目实际采用 SonarQube、ESLint、Ruff、golangci-lint、Checkstyle、SpotBugs、Build 校验、依赖漏洞扫描等，必须显式写入
   - 当前项目 `.trellis/spec/`、设计文档或 `task_plan.md` 中，至少要有一处可直接查到这份检查矩阵

4. **同步适配当前项目的 `/trellis:finish-work`**
   - 这是 `finish-work` 的**主适配阶段**
   - 必须基于任务 3 中已经写清的自动化检查矩阵，修正对应项目里的 `finish-work`
   - 至少补齐：真实 lint / typecheck / test / build / scan / delivery gate、失败后的处理动作、哪些场景必须更新 spec / checklist / docs
   - 不允许把通用模板原样留给项目继续使用，也不允许只写“按项目情况执行检查”

5. **同步适配当前项目的 `/trellis:record-session` 基线**
   - 在进入任务拆解前，先明确当前项目记录 session 的基线规则
   - 至少补齐：最终记录入口（是否要求 helper）、何时允许进入 `record-session`、是否必须先 archive、哪些 `.trellis/` 元数据允许自动提交、哪些情况必须先阻断
   - 这里先完成**基线适配**，避免项目直到开发收尾时还沿用通用 `record-session`

6. **标记 `§4 plan` 后是否需要对 `/trellis:record-session` 做轻量校正**
   - 若任务拆解完成后，“当前任务完成”的定义、归档节点、交付节点、会话记录粒度比 `§3.7` 更明确，可以再补一次轻量修正
   - 这一步只允许小范围校正，不应把 `record-session` 的首次项目化适配拖到 `§4` 或更后面
   - 一般不需要在 `§4` 后再次大改 `finish-work`，除非计划阶段新增了新的强制检查门禁

门禁要求：

- 任务 1 未完成，不进入任务 2
- 任务 1 和任务 2 未完成，不进入任务 3
- 任务 1、任务 2、任务 3 未完成，不进入任务 4
- 任务 1、任务 2、任务 3、任务 4 未完成，不进入任务 5
- 任务 1、任务 2、任务 3、任务 4、任务 5 未完成，不进入任务 6
- 任务 1、任务 2、任务 3、任务 4、任务 5 未完成，不进入 `/trellis:plan`
- `/trellis:plan` 只负责生成 `task_plan.md`，不替代 spec 导入、spec 修订、自动化检查矩阵定义，以及 `finish-work` / `record-session` 的项目化基线适配动作

目的：

- 先把通用规范映射到当前项目
- 再把当前项目自己的规则边界补完整
- 再把项目真实要执行的自动化门禁写具体
- 再把项目真实收尾门禁和会话记录规则绑定到对应命令
- 避免 AI 在任务拆解阶段只拿到通用设计文档，却缺少项目级 spec 约束与检查基线

任务拆解完成后，不是所有任务都可以立即启动。除了写清楚“前置任务 / 阻塞任务 / 并行任务”，还必须明确每个任务的开始条件、等待原因和并行边界。

**开始规则**：

- 有前置任务的任务，只有在全部前置任务完成后才允许开始执行
- 当前任务所需输入、接口、Mock、环境、资源未就绪时，不允许开始执行
- 需要人工确认的关键节点未完成时，后续依赖任务保持等待状态

**当前状态枚举**：

- `可开始`：前置任务、输入和资源已满足，可以立即启动
- `等待中`：暂时不能启动，必须写明等待对象或等待原因
- `进行中`：已经开始执行，但尚未完成
- `已完成`：产出已完成，并满足当前阶段约定的完成条件

为避免口径漂移，`任务执行矩阵` 中的“当前状态”字段默认只使用以上四个值；如确实需要扩展，必须在 `task_plan.md` 中先定义状态含义再使用。

**等待规则**：

- 当某任务仍在执行中时，所有直接依赖它的后续任务都必须等待
- 若存在串行链路 `A → B → C`，则 `A` 未完成时不能启动 `B`，`B` 未完成时不能启动 `C`
- 等待中的任务必须标明等待原因，避免团队误以为“还没开始”只是排期问题

**并行规则**：

- 无前置依赖的任务，可以进入候选并行集合
- 即使无依赖，也要通过冲突检查后才能判定为可并行执行
- 推荐将“关键前置主链”与“独立支线任务”分开描述，避免混淆

**并行属性说明**：

- `候选可并行`：表示该任务当前属于可并行候选集，但不代表可与所有任务并行，仍需结合“冲突说明”判断具体能与谁并行
- `依赖不可并行`：表示该任务当前受前置依赖约束，不应作为可并行候选任务启动

**并行冲突判定标准**：

- 同时修改同一模块、同一批核心文件或同一共享接口契约时，不建议并行
- 共享同一测试环境、数据库状态、部署窗口或外部资源时，不建议并行
- 都依赖同一个人工确认点、评审结论或上游决策时，不建议并行
- 虽然表面上无依赖，但一个任务的输出会改变另一个任务的实现前提时，应改判为串行或“先确认再并行”

**推荐排期顺序**：

1. 先识别并安排关键前置任务，优先打通主链路
2. 再识别无依赖、低冲突的独立任务，作为可并行支线穿插执行
3. 当前置任务完成后，再启动其后续依赖任务
4. 若并行任务出现资源或边界冲突，优先回退到串行安排，不强行追求并行

**示例**：

- 已拆出任务 `A`、`B`、`C`、`A2`
- 依赖关系为：`A → B → C`
- `A2` 与 `A/B/C` 无直接依赖，也不共享模块、文件、环境或人工确认点
- 则执行顺序应为：
- 执行 `A` 时，`B` 与 `C` 必须等待
- 同一时间可以并行执行 `A2`
- `A` 完成后，才允许开始 `B`
- `B` 完成后，才允许开始 `C`

**最小状态流转**：

- `可开始 → 进行中`：任务被正式启动
- `进行中 → 已完成`：任务已交付当前阶段产物并满足完成条件
- `等待中 → 可开始`：等待原因消除，例如前置任务完成或资源就绪
- 若任务执行过程中重新出现外部阻塞，可从 `进行中` 回到 `等待中`，但必须补充新的等待原因

建议在 `task_plan.md` 中同时保留“计划级执行安排”和“任务执行矩阵”：

- 前者用于说明当前阶段的整体推进顺序
- 后者用于逐任务记录开始条件、等待原因、并行属性和冲突依据
- 若实际执行过程中依赖关系变化，应优先更新“任务执行矩阵”，再同步更新计划级执行安排

**最小可执行示例**：

```markdown
## 依赖关系

- 前置任务：
  - TASK-B 依赖 TASK-A
  - TASK-C 依赖 TASK-B
- 阻塞任务：
  - TASK-A 阻塞 TASK-B
  - TASK-B 阻塞 TASK-C
- 并行任务：
  - TASK-A2 可与 TASK-A 并行

## 执行安排

- 当前可开始任务：TASK-A, TASK-A2
- 等待中任务：
  - TASK-B（等待 TASK-A 完成）
  - TASK-C（等待 TASK-B 完成）
- 推荐并行组：TASK-A + TASK-A2
- 串行主链：TASK-A → TASK-B → TASK-C

## 任务执行矩阵

| 任务ID | 前置任务 | 当前状态 | 开始条件 | 等待原因 | 并行属性 | 冲突说明 |
|-------|---------|---------|---------|---------|---------|---------|
| TASK-A | 无 | 可开始 | 需求、接口、环境、资源已就绪 | 无 | 候选可并行 | 当前可与 TASK-A2 并行，不可与其后续依赖任务并行 |
| TASK-B | TASK-A | 等待中 | TASK-A 完成后可开始 | 等待 TASK-A 完成 | 依赖不可并行 | 依赖 TASK-A 输出 |
| TASK-C | TASK-B | 等待中 | TASK-B 完成后可开始 | 等待 TASK-B 完成 | 依赖不可并行 | 依赖 TASK-B 输出 |
| TASK-A2 | 无 | 可开始 | 需求、接口、环境、资源已就绪 | 无 | 候选可并行 | 不修改主链模块 |
```

### 4.3 AI 测试驱动开发（Test-First AI）

#### 4.3.1 核心理念

**原流程痛点**：人工写测试用例耗时，且难以保证覆盖度

**纠错方案**：AI 自动生成测试套件，人工审核确认

#### 4.3.2 Test-First AI 执行流程

```
AI 测试生成流程（Test-First）：

① 需求输入
    ├── 任务需求描述
    ├── 关联文档路径
    └── 已有代码上下文

② AI 自动生成测试代码
    ├── 单元测试代码
    ├── 集成测试代码（如适用）
    ├── 边界测试用例
    └── 异常场景测试用例

③ 人工 Review 测试代码
    ├── 审核测试覆盖度
    ├── 审核测试逻辑正确性
    ├── 确认验收标准清晰
    └── 批准测试代码

④ 测试代码进入门禁
    ├── 测试代码作为客观验证标准
    ├── 后续业务代码必须通过此测试
    └── 自动化门禁，无需人工逐行看代码
```

## 🔄 阶段五：开发实施循环

### 5.1 单个需求开发流程（提速版：时间盒 + 证据门禁）

① 读取执行矩阵并选定当前任务
- 先读取 `task_plan.md` 中的“任务执行矩阵”
- 在单线程实施模式下，优先从 `当前状态=可开始` 的任务中选择本轮要执行的唯一任务
- 若采用 Git worktree 或已安装的 `/trellis:parallel` 进行并行实施，则可按“推荐并行组”同时启动多个 `可开始` 任务，但每个并行分支仍只负责各自选定的一个任务
- 选定后立即将其状态更新为 `进行中`
- 若当前没有任何 `可开始` 任务，但仍有 `等待中` 任务，则说明当前被上游阻塞，不应直接进入实现
- 每次矩阵变更后，必须同步更新“执行安排”中的当前可开始任务、等待中任务、推荐并行组、串行主链

② DoR检查
- 需求边界、输入输出、验收标准、回归范围明确后才开始实现

③ 先测后改
- 先补/写最小测试（单测或集成测试），再进入实现

④ AI实现（可选多轮）
- 结构化Prompt生成代码草案
- 对关键输出启用Schema约束，减少格式与字段错误

⑤ 本地自动门禁
- 必须执行在阶段 3.7 已明确的“项目自动化检查矩阵”
- 不允许只跑默认 `Lint` 后就进入提交或 PR
- 至少覆盖当前项目真实需要的静态分析、类型/编译检查、测试、安全扫描；若项目启用 SonarQube 等质量平台，也必须纳入门禁

⑥ 风险分级评审
- L0低风险：自动门禁通过即可进入PR
- L1中风险：增加人工快速审查
- L2高风险：必须完整人工审查 + 安全复核

⑦ CI验证与合并
- PR状态检查全部通过后合并
- 合并后自动部署测试环境并执行回归集

⑧ 任务状态更新与级联解锁
- 当当前任务通过本轮完成条件后，将其状态从 `进行中` 更新为 `已完成`
- 更新完成后，重新检查其下游依赖任务；若其 `前置任务` 列中的依赖已全部满足，则将下游任务从 `等待中` 更新为 `可开始`
- 级联解锁只基于 `前置任务` 列判断，不自动判断“开始条件”中的接口、环境、资源等附加约束；这些附加约束在下一轮 `DoR检查` 时再次验证
- 若任务在执行中被外部条件阻塞，则从 `进行中` 回退为 `等待中`，并补写等待原因
- 每次状态更新后，必须回写“执行安排”，确保计划级视图与任务执行矩阵保持一致
- 若当前任务命中“任务级多 CLI 补充审查门禁”，则必须先完成 `/trellis:check` 的判定与后续处理（或明确 `skip`），才允许将该任务标记为 `已完成`
- 当任务执行矩阵中的任务全部变为 `已完成` 时，退出阶段五，进入收尾链路（`/trellis:self-review` → `/trellis:check` → `/trellis:finish-work` → `/trellis:delivery`）

⑨ 上下文健康检查与决策摘要（每轮时间盒结束时）
- 检查当前上下文是否包含矛盾指令、过期信息、幻觉累积
- 若检测到上下文污染迹象（AI开始重复之前已修复的错误、输出与需求方向偏离）
  → 必须开启新会话，重新注入稳定前缀，不允许带污染上下文继续
- 定期导出关键决策为结构化摘要，注入新会话而非依赖完整历史
  - 每次里程碑完成后生成决策日志
  - 摘要格式：结构化 JSON/Markdown
  - 包含：决策内容、依据、预期影响

### 5.1.x 反思/批判阶段（新增）

#### 5.1.x.1 广义自审机制

AI 实现后、提交前，必须执行一次广义自审：

```
自审清单：
□ 对照 Spec 逐项核对实现完整性
□ 对照测试用例验证功能正确性
□ 边界场景覆盖度检查（空值、极值、异常值）
□ 安全风险自检（注入、越权、泄露）
□ 性能影响评估（计算复杂度、资源占用）
```

输出"偏差清单"：
- 与需求/设计不一致的点
- 测试未覆盖的风险区域
- 建议人工重点审查的模块

#### 5.1.x.2 任务级多 CLI 补充审查门禁

当前 CLI 完成**当前任务原本的 review** 后，不是默认立即标记任务结束，而是要先判断是否需要进入“任务级多 CLI 补充审查层”。

这一层的定位是：**面向高风险、高不确定、高影响面任务的补充审查门禁**。它不是对 `self-review` 的重复执行，也不是常规任务的默认重流程。

触发顺序：

```
当前任务实现完成
  → 当前 CLI 完成原本 review（自审 / 本地门禁 / 当前任务验证）
  → `/trellis:check` 判断是否触发任务级多 CLI 审查
  → 若触发：其他 CLI 产出报告 → 当前 CLI 汇总修复 → 当前 CLI 重新验证
  → 任务才允许标记为已完成
```

角色分工：

- **当前 CLI**：协调者 + 唯一修复者
- **其他 CLI**：reviewer，只审查，不直接修改代码
- **用户**：手动在其他 CLI 中执行标准命令；在冲突、越界或超阈值时做最终决策

触发模型：

1. **硬条件（命中即触发）**
   - 认证、授权、权限边界、敏感信息处理
   - 数据迁移、schema 变更、删除与回填
   - 公共 API、跨层 contract、外部系统集成
   - 支付、消息队列、缓存一致性、并发状态
   - 共享核心模块且 blast radius 明显
   - 用户显式要求执行任务级多 CLI 审查

2. **软条件门槛（分层判断）**
   - **复杂度层**：改动文件数、改动行数、涉及模块/层数、异常路径数量
   - **影响面层**：公共模块、跨层边界、外部集成、blast radius
   - **可信度层**：测试覆盖不足、当前 CLI 不确定性高、AI 生成比例高、历史缺陷密度高

门槛判定：

- 任一层达到高门槛 → `required`
- 两层达到中门槛 → `recommended`
- 均未达到门槛 → `skip`

执行约束：

- 默认由用户手动调用 **1 个其他 CLI**；必要时可扩展到 **4 个 reviewer**
- 单任务最多 **3 轮**补充审查
- 若当前轮没有新的有效修复建议，可提前关闭
- 达到 3 轮上限且仍有未解决问题，或 reviewer 建议互斥、越出任务边界时，必须停止自动推进并要求人工介入
- 进入该层前，目标 reviewer CLI 必须已具备 `multi-cli-review` 能力；当前 CLI 必须已具备 `multi-cli-review-action` 能力。若缺失，先补齐对应 skill，再进入补充审查
- reviewer 只消费当前 CLI 已创建好的目录结构；**reviewer 不负责创建目录**

交接协议：

- 当前 CLI 负责先创建 `tmp/multi-cli-review/<task-id>/review-round-<N>/`
- 当前 CLI 输出给其他 CLI 的**完整标准命令**
- 其他 CLI 使用 `multi-cli-review` 写入各自的 `{reviewer-id}.md`
- 当前 CLI 使用 `multi-cli-review-action` 统一读取、去重、冲突标记、执行修复
- 修复后，当前 CLI 必须重新跑该任务原本的 review / 验证，之后才能关闭该任务

推荐输出物：

- `$TASK_DIR/check/review-gate-round-<N>.md`：记录本轮为何触发 `required / recommended / skip`
- `$TASK_DIR/check/reviewer-commands-round-<N>.md`：给其他 CLI 的标准命令模板
- `tmp/multi-cli-review/<task-id>/review-round-<N>/{reviewer-id}.md`：各 reviewer 报告
- `tmp/multi-cli-review/<task-id>/summary-round-<N>.md`：聚合结果
- `tmp/multi-cli-review/<task-id>/action.md`：当前 CLI 决策与执行记录
- `tmp/multi-cli-review/<task-id>/.processed.json`：由 `multi-cli-review-action` 维护的内部消费状态

#### 5.1.x.3 双模态执行原则

根据任务风险级别采用不同执行策略：

| 风险级别 | 任务类型 | 执行策略 |
|---------|---------|---------|
| **低风险** | 样板代码生成、测试用例生成、文档同步 | AI 激进执行 → 快速 Review → 自动门禁 |
| **中风险** | 功能实现、逻辑编写 | AI 主导 → 关键节点人工确认 |
| **高风险** | 核心算法、安全相关、数据迁移 | 人工主导 → AI 辅助 → 完整人工评审 |

#### 5.1.x.4 上下文污染检测升级

检测到污染迹象时的强化处理：
```
① 立即停止当前会话
② 基于最新决策摘要（而非完整历史）开启新会话
③ 标记本次会话异常供后续复盘
```

**时间盒规则**：
- 简单任务（L0，单文件修改）：根据复杂度合理设定
- 标准任务（L1，跨模块）：根据复杂度合理设定
- 复杂任务（L2，架构级）：拆分为多个子时间盒，确保单次可完成

**退出条件（按优先级）**：
① 证据通过（DoD满足）→ 正常退出
② 时间盒耗尽但进度>70% → 延长一个半时间盒，仍未完成进入5.2
③ 同类错误连续2轮未收敛 → 立即进入5.2，不等时间盒结束
④ 以"证据通过"作为退出条件，不使用主观置信度阈值

### 5.2 人工介入机制
- 触发条件1：同类缺陷连续2轮未解决
- 触发条件2：任务耗时超估算50%仍未达到DoD
- 触发条件3：出现安全高危、数据一致性风险、线上阻断风险
- 触发条件4：模型输出不稳定（同输入多次结果差异过大）
- 触发条件5：任务级多 CLI 审查达到 3 轮仍未收敛，或 reviewer 建议互斥 / 越界

介入动作：
- 先做根因分类（需求问题/设计问题/实现问题/评估问题）
- 调整策略（重写验收、拆任务、换模型、改Prompt、改工具链）
- 产出"介入结论记录"，作为后续同类任务模板

### 5.3 阶段性检查点
- 交付效率指标（建议周维度复盘）：
  - 部署频率
  - 变更前置时间（从提交到上线）
  - 变更失败率
  - 平均恢复时间（MTTR）
  - 返工率（rework）
- AI效率指标：
  - 首轮通过率（一次实现即通过DoD的比例）
  - 评估通过率（评估集通过比例）
  - 单任务Token成本（按被验收任务计）
  - 缓存命中率（`cached_tokens`占比）
  - 降级/回退触发率
- 要求：
  - 任一核心指标连续两周恶化，必须触发流程复盘

### 5.4 人工代码校验（风险分级必选）

- 低风险任务：抽样校验
- 中风险任务：必须人工审查关键逻辑与异常路径
- 高风险任务：必须完整人工审查、威胁建模、回归验证
- 单人项目执行方式：至少间隔30分钟后进行"冷启动复审"，避免同一上下文盲点

> 这一阶段体现了"实现后强制自检"和"优化-评审"的闭环思想。

### 5.5 代码与文档同步机制

#### 5.5.1 同步原则

- **任务粒度同步**：每个任务完成后，必须同步更新相关文档
- **上下文变更同步**：每次上下文有重大变更（如需求变更、架构调整），需同步更新文档
- **即时优先**：文档更新应尽可能在代码变更后立即完成，避免事后遗忘

#### 5.5.2 同步内容清单

| 变更类型 | 必须更新的文档 | 更新时机 |
|---------|--------------|---------|
| 新增功能 | 需求文档、功能规格说明、接口文档 | 功能验收通过后 |
| 代码重构 | 技术架构文档、模块说明 | 重构完成通过测试后 |
| 数据库变更 | 数据库设计文档、ER图、数据字典 | 迁移/变更生效后 |
| 接口变更 | 接口文档、集成说明 | 接口联调完成后 |
| Prompt调整 | 提示词模板库、评估集 | 评估基线回归通过后 |

#### 5.5.3 同步执行规范

```
代码与文档同步流程：

① 触发条件检测
    ├── 代码提交后检查是否有文档依赖变更
    ├── 上下文更新后检查文档时效性
    └── 任务验收通过后触发同步检查

② 文档受影响范围分析
    ├── 识别本次变更影响的文档范围
    ├── 区分必须更新/建议更新/无需更新
    └── 标记文档更新优先级

③ 执行文档同步
    ├── 按受影响范围逐项更新
    ├── 更新内容必须与代码/上下文一致
    └── 添加版本记录和变更说明

④ 同步验证
    ├── 检查文档与代码的一致性
    ├── 验证文档完整性
    └── 确认无遗漏项
```

#### 5.5.4 文档版本管理

- 所有文档采用与代码关联的版本号
- 重大变更需在文档中记录变更日志
- 保留历史版本以便追溯

### 5.6 单次对话任务闭环规范

#### 5.6.1 闭环定义

单次对话上下文内完成一个独立任务，包含完整的工作流程：

```
单次对话任务闭环流程：

① 任务理解与确认
    ├── 明确任务目标与验收标准
    ├── 确认DoR满足条件
    └── 与用户确认理解一致

② 代码实现
    ├── 按任务规格进行代码编写
    ├── 遵循项目编码规范
    └── 确保代码可运行

③ 代码质量检测（按项目自动化检查矩阵执行）
    ├── 静态分析/规范检查（如 ESLint、Ruff、Checkstyle）
    ├── 类型检查/编译校验（如 Type Check、mypy、Build）
    ├── 质量平台/安全扫描（如 SonarQube、依赖漏洞、敏感信息、注入风险）
    └── 单元/集成测试（核心逻辑覆盖）

④ 代码优化
    ├── 性能优化（减少不必要的计算/请求）
    ├── 可读性优化（命名、注释、结构）
    └── 重复代码提取

⑤ Bug修复（如有）
    ├── 根据测试反馈定位问题
    ├── 分析根因
    └── 修复并验证

⑥ 回归验证
    ├── 确保不破坏现有功能
    ├── 运行绑定评估集
    └── 确认满足DoD标准
```

#### 5.6.2 任务拆分原则

- 单次对话任务应在30-90分钟可完成
- 复杂任务需拆分为多个可闭环的子任务
- 跨对话任务必须有明确的状态传递机制

#### 5.6.3 上下文健康管理

- 每次任务开始前检查上下文清洁度
- 任务完成后导出关键结论到外部存储
- 新任务开始时优先从外部存储恢复上下文，而非依赖对话历史

### 6.1 自动化测试
- 执行自动化测试脚本
- 覆盖所有功能模块

- **需求实现验证**：所有需求点已实现
- **用户场景测试**：模拟真实用户操作流程
- **AI功能专项测试**：
  - AI响应质量评估
  - 提示词鲁棒性测试
  - 边界输入处理测试

- **用户体验测试**：响应流畅度、无明显卡顿
- **并发性能测试**：支持预期并发量（如需要）
- **性能指标**：响应时间、吞吐量、资源占用
- **AI性能指标**：
  - 首Token响应时间 (TTFT)
  - Token间延迟 (ITL)
  - AI API调用延迟

- 按客户版需求说明书与冻结验收标准执行验收

- 验收必须基于"结果质量"，不基于"AI参与比例"

验收门槛建议：
- 需求场景覆盖率达标（核心场景100%通过）
- 自动化回归通过率达标
- P0/P1缺陷为0
- 性能与稳定性满足约定SLO
- 安全扫描无高危未处置项
- AI相关能力（如有）通过基线评估集回归

#### 6.5.3 Bug修复执行流程

```
Bug修复执行流程：

① 任务领取与确认
   ├── 从Bug池中领取修复任务
   ├── 确认理解问题描述
   └── 确认测试指标清晰

② 原因分析
   ├── 定位问题根因
   ├── 分析影响范围
   └── 制定修复方案

③ 代码修复
   ├── 按照修复方案进行代码修改
   ├── 确保不影响现有功能
   └── 编写或更新相关测试

④ 自测验证
   ├── 按照测试指标逐项验证
   ├── 记录验证结果
   ├── 确认所有指标达标
   └── 如不达标则继续修复

⑤ 提交审核
   ├── 提交代码修改
   ├── 说明修复内容和验证结果
   └── 等待审核

⑥ 审核通过
   ├── 审核通过后标记任务完成
   ├── 更新Bug状态为已修复
   └── 关闭Bug记录
```

#### 6.5.4 修复优先级定义

| 优先级 | 定义 | 响应时间 | 修复周期 |
|--------|------|---------|---------|
| **P0 - 阻塞** | 功能完全不可用、数据丢失等致命问题 | 立即响应 | 24小时内 |
| **P1 - 严重** | 功能严重受损、重要功能不可用 | 当天响应 | 2-3天 |
| **P2 - 一般** | 功能有瑕疵但可用、性能轻微问题 | 1-2天内 | 1周内 |
| **P3 - 轻微** | UI细节、文案错误、体验问题 | 1周内 | 2周内 |

#### 6.5.5 Bug修复验收

每个Bug修复完成后需要进行验收：

```
Bug修复验收检查表：
```

### 6.6 Bug反馈学习机制

每个Bug修复完成后，需要进行反馈学习，避免同类问题再次出现。

#### 6.6.1 问题分类与根因分析

对修复的Bug进行分类和根因分析，形成知识沉淀：

| 字段 | 说明 |
|------|------|
| Bug编号 | BUG-001 |
| 问题分类 | 需求理解/设计缺陷/编码错误/测试遗漏/技术债务/其他 |
| 根因类型 | 遗漏/误解/疏忽/经验不足/流程问题/其他 |
| 根因描述 | 详细描述问题产生的根本原因 |
| 5Why分析 | 逐层追问为什么，直到找到根本原因 |

**示例 - 5Why分析：**
- 问题：用户登录后显示空页面
  - Why 1: 为什么显示空页面？→ 页面数据未加载
  - Why 2: 为什么数据未加载？→ API返回空数据
  - Why 3: 为什么API返回空数据？→ 数据库查询条件错误
  - Why 4: 为什么查询条件错误？→ 代码中使用了错误的字段名
  - Why 5: 为什么使用错误字段名？→ 需求文档和数据库设计文档字段名不一致，未核对
- 根本原因：需求文档和数据库设计文档字段名不一致，开发时未进行核对

#### 7.2.1 外部项目双轨交付门禁

若项目属于外包、定制开发或新客户项目，交付前必须先确认在阶段一约定的交付轨道：

- **托管部署轨**
  - 尾款未到账时，只交付演示地址、试运行环境访问、验收材料、用户手册、运维说明
  - 尾款未到账时，不交付源码仓库权限、生产环境密钥、管理员账号、最终部署控制权
- **试运行授权轨**
  - 尾款未到账时，只交付已明确披露限制条件的试运行版本
  - 交付材料必须写清：授权有效期、续期方式、到期行为、永久授权触发条件
  - 到期行为不得删改数据，不得造成不可恢复损失

#### 7.2.2 尾款到账后的最终移交

仅在开发者明确确认尾款到账后，才执行最终移交：

- 源码仓库或源码包
- 永久授权文件或移除试运行限制的版本
- 构建脚本、部署脚本、CI/CD 配置
- 生产环境变量、密钥、证书、第三方平台配置
- 服务器、域名、数据库、对象存储等管理员权限
- 最终运维文档、回滚说明、交接记录

建议把以上移交项做成 checklist，由交付双方逐项签收。

### 7.4 会话记录与元数据提交

- 代码、文档或配置变更已由人工测试并提交后，再执行会话记录
- 这里只有**当前执行任务已完成后的最终收尾记录**，才允许进入 `record-session`
- 已完成任务在记录前应先显式归档；未完成任务不得因记录会话而误归档；**非当前任务**不得借本轮收尾顺手自动提交
- 进入 `record-session` 前，`.trellis/tasks` 与 `.trellis/.current-task` 必须已经 clean；否则先处理 archive 或其它任务元数据问题
- 进入 `record-session` helper 前，staged 区未混入非目标变更；否则应中断
- `record-session` 应通过工作流 helper 执行，不应直接把 `add_session.py` 作为最终收尾入口
- 不允许仅依据"task archived"或"session added successfully"字样就判定完成，必须以 git 状态为准

收尾校验：

```bash
git status --short .trellis/tasks .trellis/.current-task
git status --short .trellis/workspace .trellis/tasks
```

- 预期输出为空；若仍有 `.trellis/tasks`、`.trellis/workspace` 或 `.trellis/.current-task` 变更，说明归档/记录未真正闭环
- 若自动提交失败，应视为 `record-session` 未完成，先处理 git 写入失败原因，再结束本轮工作
- 若目标不是当前任务，或 staged 区存在非目标变更，应视为流程前置条件不满足，先清理边界问题，再继续收尾

> 这一阶段体现了"错误沉淀与复用"的方法论价值。

## 🧪 附录一：AI评估驱动开发 (Evaluation-Driven Development)

在项目开始前，应建立AI评估体系作为基础。

### 附录1.1 评估指标定义
- **准确率/通过率**：AI生成代码通过测试的比例
- **处理时间**：系统级 + 组件级响应时间
- **系统性能**：吞吐量、并发能力
- **幻觉率**：AI产生错误输出的频率
- **上下文利用效率**：上下文窗口的有效利用率

### 附录1.2 测试用例设计

- **快乐路径用例**：核心功能场景
- **边界用例**：AI易失败的场景
- **对抗用例**：提示词注入、恶意输入
- 使用LLM生成扩展测试用例

**评估集规模建议（参考 OpenAI Evals 和 Anthropic 内部实践）**：
- 最小可信评估集：≥50条（低于此数量，统计波动过大，结果不可信）
- 生产级评估集：200~500条（核心路径覆盖率≥80%）
- 对抗用例比例：≥15%（否则鲁棒性测试形同虚设）
- 评估集增长触发：每次线上故障、每次重大Prompt变更，至少沉淀5~10条新用例

**评估结果解读门槛**：
- 单次评估波动在±3%以内属正常，不应触发回滚
- 连续2次下降且累计超过5% → 触发回归阻断

### 附录1.3 评估方法选择
- **确定性检查**：结构、路径、工具调用正确性
- **语义检查**：使用LLM-as-Judge进行语义等价判断
- **混合策略**：轻量检查 + LLM评估

### 附录1.4 评估自动化
- 建立持续评估pipeline
- 配置评估频率（每次commit/每日/每周）
- 设定回归检测阈值

### 附录1.5 持续评估门禁
- PR级快速评估：每次PR运行最小评估集（核心路径 + 最近高频缺陷）
- 夜间全量评估：每日运行完整评估集，跟踪趋势变化
- 回归阻断：关键指标低于阈值时阻断发布
- 评估集演进：每次线上故障与高优Bug必须沉淀为新评估用例

> 评估体系是AI Workflow的核心基础，没有评估就没有改进

## 🔒 附录二：AI代码与AI应用安全审查（OWASP LLM Top 10 + Agentic Applications 2026 映射）

### 审查重点：OWASP LLM Top 10

- Prompt Injection（提示词注入）
- Sensitive Information Disclosure（敏感信息泄露）
- Supply Chain（供应链风险）
- Data/Model Poisoning（数据与模型投毒）
- Improper Output Handling（输出处理不当）
- Excessive Agency（过度代理权限）
- System Prompt Leakage（系统提示词泄露）
- Vector/Embedding Weaknesses（向量与嵌入弱点）
- Misinformation（错误信息）
- Unbounded Consumption（无界资源消耗）

### 审查重点：OWASP Top 10 for Agentic Applications 2025/2026（新增）

| 风险项 | 描述 | 防护建议 |
|-------|------|---------|
| **Tool Misuse** | 工具被错误调用或超出预期范围使用 | 工具入参必须白名单校验，权限最小化 |
| **Identity & Memory Management** | Agent 身份混淆、内存/上下文信息泄露 | 严格隔离不同任务上下文，会话状态加密 |
| **Autonomous Decision Errors** | Agent 过度自主决策导致错误结果 | 关键决策必须人工确认，输出必须可追溯 |
| **Goal Hierarchy Failure** | Agent 目标层级混乱，优先级误判 | 明确目标优先级，建立冲突仲裁机制 |
| **Unbounded Execution** | Agent 执行无外部边界限制 | 设置执行预算上限，超限强制中断 |

### agentic 工作流防护（新增）

1. **自主操作限制**：
   - 所有涉及外部系统写操作、敏感数据访问、费用产生等高影响行为，必须插入人工审批节点
   - Agent 不得在无确认情况下执行不可逆操作

2. **人类最终责任**：
   - 明确宣示：AI 生成内容的所有最终责任归人类决策者
   - 关键输出必须经过人类审核后方可生效
   - 建立"人类在环"（Human-in-the-Loop）机制作为强制门禁

3. **执行边界约束**：
   - 单次任务最大执行轮次设定上限
   - 累计资源消耗（Token、时间、费用）设定告警阈值
   - 连续失败自动触发人工介入流程

4. **审计追踪**：
   - 所有AI的工具调用（尤其是写操作）必须留有可追溯日志
   - 日志必须包含：输入摘要、调用时间、模型版本、输出结果、执行用户

**最低安全要求（可落地版）**：

1. **Prompt Injection防护**：
   - 用户输入必须与系统指令在结构上隔离（使用角色分隔或XML标签，不拼接到system prompt）
   - 禁止将外部内容（如用户上传文件、爬取内容）直接注入高权限上下文

2. **输出处理**：
   - AI输出不得未经验证直接执行（eval、exec、shell注入等）
   - 结构化输出必须经过Schema验证，验证失败必须拒绝而非降级处理

3. **敏感信息防护**：
   - 系统Prompt不得包含可被提取的密钥、内部路径或用户PII
   - 生产环境API Key必须通过环境变量注入，不得硬编码在Prompt模板中

4. **访问控制**：
   - 工具调用的权限范围必须最小化（只读工具不赋写权限）
   - Agent自主操作外部系统前，必须有人工审批节点或可撤销确认机制
