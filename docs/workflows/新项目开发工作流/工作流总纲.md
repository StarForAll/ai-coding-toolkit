## 使用前提与安装时序

这套 workflow 的定位一直是：**在目标项目已经具备 Trellis 基线后，再做嵌入和增强**。

还要明确一个运行时边界：当前 workflow 的收尾链路虽然会对 `record-session` 做 workflow 补丁增强，但最终的 `archive` 仍直接调用目标项目 Trellis 基线里的 `python3 ./.trellis/scripts/task.py archive`。因此，目标项目应优先使用当前最新 Trellis 基线；至少要包含 archive metadata auto-commit 的 pathspec 修复，否则收尾阶段仍可能因为旧基线行为失败。

### 当前仓库维护本 workflow 时的 Trellis 升级兼容基线（强制规则）

以后凡是**在当前仓库内维护这套以 Trellis 为基底的 workflow 内容**，需要判断 Trellis 升级后命令、补丁、脚本和文档该如何兼容升级时，参考的“初始 Trellis 框架代码”必须来自：

1. 在 `/tmp` 下新建临时项目
2. 对该临时项目执行 `trellis init`
3. 以这份纯净初始化产物作为 Trellis 原始基线

不要直接把当前仓库里的 `.trellis/`、`.claude/`、`.opencode/`、`.agents/skills/`、`.codex/` 当作“初始 Trellis 基线”。这些目录在本仓库中已经混入了项目规则、workflow 补丁、共享 skills 与平台适配，只能作为**当前已落地实现参考**。

这条规则的边界需要明确：

- 它适用于当前仓库维护 workflow 源内容时的兼容升级判断
- 它**不直接等于**使用该 workflow 的目标项目在执行 `upgrade-compat.py` 时的判定标准

目标项目已经有自己的历史状态、已安装补丁、备份、安装记录和部署漂移，不能拿“完全初始的 Trellis”直接要求目标项目按同一标准判断升级兼容。

因此，当前仓库维护 workflow 内容时，至少应区分两份参考对象：

- `/tmp` 临时项目中的纯净 Trellis 基线：判断当前 Trellis 版本的原始结构
- 当前 workflow 源资产：判断 workflow 需要依赖和注入哪些补丁点

若还要评估这次升级会如何影响已经安装过 workflow 的项目，再**额外**查看真实目标项目当前安装状态。

目标项目侧的兼容升级应采用“`A/B/C` 三态分析优先，`upgrade-compat.py` 仅处理低风险漂移”的方式；`upgrade-compat.py` 提供的是“版本记录 + 已部署资产完整性检查 + 补丁重注入”的本地自动化处理层，不替代当前仓库维护 workflow 内容时的纯净基线对照，也不替代目标项目的三态分析。

因此，目标项目若要真正使用这套 workflow，必须同时满足：

- 目标项目本身是一个 Git 项目
- 目标项目的 `origin` 已至少配置两个 push URL（例如 GitHub + Gitee 多 push）
- 目标项目已经执行过 `trellis init`
- 能检测到 `trellis init` 产物，例如 `.trellis/.version`

参考配置命令：

```bash
git remote add origin <你的第一个远程仓库URL>
git remote set-url --add --push origin <第一个仓库URL>
git remote set-url --add --push origin <第二个仓库URL>
```

若 `origin` 已存在，跳过第一条。

如果任一前提不成立，就不应把后续命令、skills、hooks、阶段门禁视为“已经嵌入成功”。

当前 workflow 的嵌入执行必须遵循《[工作流嵌入执行规范](./工作流嵌入执行规范.md)》。

主链只有一条：

1. 目标项目先完成 `trellis init`
2. 先运行 `commands/detect-embed-state.py`，确认目标项目处于 `INITIAL_BASELINE_READY`
3. 再运行 `commands/install-workflow.py --dry-run`
4. 再运行正式安装
5. 安装器按 Claude Code / OpenCode / Codex 各自官方原生格式，把 workflow 命令、skills、共享 helper 与对应 baseline patch 嵌入到目标项目；`.claude/settings*.json`、`opencode.json.instructions`、`.codex/config.toml`、`.codex/hooks.json`、`.codex/hooks/*.py` 等手动维护资产不由安装器分发
6. 安装器自动导入 `pack.requirements-discovery-foundation`；若目标项目存在 `trellis init` 生成的 `00-bootstrap-guidelines`，则一并清理；若 `.current-task` 仍指向该 bootstrap task，也会同步清理悬空引用；目标项目本就不存在该任务时跳过，`workflow-installed.json` 中的 `bootstrap_task_removed` 字段只记录"本次安装是否实际执行过删除"，`bootstrap_cleanup_status` 提供更细的清理状态（如 `removed` / `absent` / `dry-run-removed`），二者都不等同于"该任务历史上一定存在过"
7. 正式安装开始后，安装器会先写入 `.trellis/workflow-embed-attempt.json`；只有装后自检通过后，该文件才会被清理
8. 安装完成后，先按《装后隐藏目录与托管边界核对清单》完成装后核对；至少确认 `.trellis/workflow-installed.json`、`.trellis/workflow-embed-attempt.json`、`.trellis/workflow.md`、`.trellis/library-lock.yaml`、`AGENTS.md` managed block、`.agents/skills/` 的共享 skills 状态，以及 `.codex/skills/` 中是否只保留 Codex 独有 skills
9. 核对通过后，再在目标项目里按对应 CLI 的原生入口使用这套 workflow

推荐安装方式：

```bash
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/install-workflow.py \
--project-root <target-project>
```

如需过滤 CLI：

```bash
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/install-workflow.py \
--project-root <target-project> \
--cli claude,opencode,codex
```

这里的关键点是：

- 必须先有 `trellis init`，再执行安装脚本
- 只有纯净初始态目标项目才允许执行首次嵌入；非初始态一律阻断
- 安装脚本只负责 workflow-managed assets，不负责替项目补建手动维护的 hooks / settings / config
- 安装完成后必须先做装后核对，再视为 workflow 可直接使用

安装完成后，实际交流与执行时仍然可以采用**渐进性披露**：

- 先进入当前阶段
- 再按阶段按需展开对应命令、文档、MCP、skills 和平台配置
- 而不是在单轮上下文里一次性灌入所有材料

也就是说：

- `trellis init -> install-workflow.py -> 装后核对` 是**安装层**
- 安装完成后按阶段逐步展开，是**使用层的渐进性披露**

## 2026 核心方法论升级

> 本文档是 **AI 辅助开发实战工作流**，目标：更快交付、可验证正确、可控成本、可控风险。

### 核心理念升级（2026）

基于 2025 年 AI Coding 领域的实践经验，我们升级以下核心方法论：

#### 1. 规划先行 + 证据门禁

- **所有 AI 参与环节必须先规范后实现**
- 不允许"边想边做"的无序 AI 消耗
- 每个 AI 输出必须有对应的验证证据

#### 2. 先测试后编码

- 践行 **Test-First AI** 原则
- 测试用例是验收标准的客观载体
- AI 生成代码必须通过预先定义的测试集

#### 3. 实现后质量检查

- 引入基于 Trellis 原生 `check` 合并增强后的质量检查机制
- AI 实现后必须对照 spec、验证证据与风险清单完成检查
- 输出结构化检查结果供 `review-gate` / `finish-work` 快速消费

#### 4. 人类最终责任

- 明确 AI 是"工具"而非"决策者"
- 关键节点必须"人类在环"
- 所有输出的人类最终责任不可转移

### 执行顺序铁律

> 需求冻结 → 意图确认 → 测试设计 → AI 实现 → 质量检查 → 补充审查 / 人工复核 → 证据回归

> **任何跳过验证环节的 AI 执行都是不被允许的**

## MCP / Skills 配置原则（安装完成后的渐进性披露）

这套 workflow 面向 **Claude Code / OpenCode / Codex** 的多 CLI 原生适配模型，而不是单一 CLI 的临时说明。因此，MCP 与 skills 的规则必须同时满足两件事：

1. 让多 CLI 在同一项目里共享一致的能力路由原则
2. 避免把大段平台细节直接塞进默认注入上下文，造成不必要的上下文占用

这里的 **渐进性披露** 专指：**workflow 已经安装到目标项目之后**，在实际使用时如何按阶段按需展开上下文，而不是替代安装动作本身。

默认采用 **渐进性披露**：

| 披露层级 | 主要载体 | 默认是否应进入主上下文 | 负责内容 |
|---------|---------|----------------------|---------|
| 主规则层 | `工作流总纲.md` | 是 | 能力路由原则、配置分层原则、降级与证据边界 |
| 映射层 | `命令映射.md` | 按阶段需要引用 | 阶段 × 场景 × 工具 / skill × 配置层矩阵 |
| 平台展开层 | `commands/claude/README.md` `commands/opencode/README.md` `commands/codex/README.md` | 否，按需查阅 | 各 CLI 的原生配置入口、承载方式、最小验证与注意事项 |

约束：

- 主规则层只保留**最小必要原则**，不内嵌大段平台配置样例
- 长配置示例、平台特有约束、验证命令放到对应平台 README
- 若某条规则只对单一 CLI 生效，不应反向污染所有 CLI 的默认上下文

### 配置分层原则

MCP / skills 的配置与使用，应拆成以下层次：

| 层次 | 典型载体 | 负责内容 |
|------|---------|---------|
| 项目长期稳定规则 | `AGENTS.md` | 证据优先、风险边界、工具优先级、何时必须联网、何时标记 `[Evidence Gap]` |
| workflow 规则与阶段路由 | `工作流总纲.md`、`命令映射.md`、各阶段命令文档 | 在什么阶段、遇到什么问题时应该优先调用哪类能力 |
| CLI 原生配置层 | Claude：`.claude/settings.json` / `.claude/settings.local.json` / `.claude/hooks/`；OpenCode：`opencode.json.instructions` 与平台原生配置；Codex：`.codex/config.toml` / `.codex/hooks.json` / `.codex/hooks/` / `.agents/skills/`（共享 skills 主承载面） / `.codex/skills/`（仅 Codex 独有或项目自定义 skills） | 把规则真正挂接到对应 CLI：命令、hooks、权限、注入、skills、agents |
| 复用能力资产 | skills、agents、通用脚本 | 浏览器操作、需求评估、计划拆解、多 CLI 审查等可复用能力本身 |

补充边界：

- `.agents/skills/` 不是“所有 CLI 的共享技能目录”
- 当前已确认会受 `.agents/skills/` 影响的是：OpenCode、Codex
- Claude Code 官方技能目录是 `.claude/skills/`；当前没有官方证据表明 Claude Code 会读取 `.agents/skills/`
- 对当前 workflow 来说，OpenCode 的正式主入口仍是 `.opencode/commands/trellis/`；`.agents/skills/` 只代表 OpenCode 也可能发现同一份 skill，不代表它替代了命令入口

### 能力路由基线

以下规则属于**主规则层**，应被各 CLI 共同遵守：

- **代码上下文定位**：优先 `ace.search_context`
- **第三方库 / SDK / 框架官方文档**：优先 `Context7`（`resolve-library-id -> query-docs`）
- **最新信息 / 版本 / 新闻 / 今日或本周类事实**：必须走 live 检索，默认优先级为 `grok-search > exa > 内置 Web tools`
- **浏览器交互 / 页面操作 / 站点探索**：优先 `agent-browser` skill
- **需求模糊 / 增强**：优先 `ace.enhance_prompt`，回退 `sequential-thinking`
- **复杂多步推理（>3步）**：优先 `sequential-thinking`
- **GitHub 项目文档 / 架构理解**：优先 `deepwiki`，回退 `exa_web_search_exa`
- **深度研究 / 竞品分析 / 技术调研**：优先 `exa_web_search_advanced_exa(type=deep-reasoning)`，回退 `grok-search`
- **知识可视化 / 流程图 / 依赖图**：优先 `markmap`
- **复杂多步问题**：优先结构化思考或分阶段 research，而不是在实现阶段边想边改
- **关键能力不可用**：必须显式写出 `[Evidence Gap]`、已尝试链路、当前降级路径，不能用猜测替代证据

### 配置边界

- workflow 文档负责定义“**应该用什么能力**”与“**能力优先级**”
- 各 CLI 原生配置负责定义“**这些能力如何被挂接、授权、注入和发现**”
- skills / agents 负责承载“**复用能力本身**”，不应把整份 skill 说明复制进阶段命令正文
- 若某能力需要凭据、MCP server 注册或本机权限，配置应留在 CLI 的原生配置面，而不是写进 `AGENTS.md`

## 📋 阶段一：项目可行性评估

### 1.1 合法性审查
- **法律合规性检查**：确认项目在当地法律法规框架下是否合规
- **决策点**：如果不符合法律法规，**终止项目**，不进行任何后续工作

### 1.2 粗略需求沟通
- 与客户进行初步沟通，明确大致需求方向和范围
- 识别核心功能点和技术难点

### 1.3 项目评估与报价
- 工作量评估：功能开发工时 + 评审工时 + 测试工时 + 文档工时
- 时间成本分析：按里程碑拆分，给出最早/最晚交付窗口
- 资源成本核算：人力、技术资源投入成本
- AI相关成本核算：
  - Token成本：纳入项目整体成本规划，合理估算
  - 重试预算：根据功能复杂度预留合理冗余
  - 缓存收益：合理设计前缀以优化成本
  - 评估成本：纳入整体成本规划
- 风险预判：
  - 技术风险、进度风险、依赖风险
  - AI特有风险：幻觉、提示词注入、上下文污染、模型不可用或限流
- 报价输出：给出基础版范围、变更单价规则、交付SLO草案
- 本阶段的时间/报价输出默认是**商务预判**与区间策略，不替代需求澄清后的正式项目级粗估；正式粗估必须在阶段二 `brainstorm` 收口前落盘

#### 1.3.1 进入需求发现前的门禁

- 对于**新项目 / 新客户需求 / 外包接单场景**，在第一次进入当前 workflow 时，必须先完成一次可行性评估。
- 若目标项目是**新建仓库**，在**第一次进入当前 workflow** 时就应先确认本地主分支和初始分支为 `main`；若仓库尚无本地提交历史且当前不在 `main`，先修正分支，再继续 workflow。
- 若目标项目已经存在本地提交历史或已有存量开发内容，则不强制切换成 `main`，但应把当前分支策略作为初始化边界记录清楚。
- 可行性评估应至少产出 `assessment.md`，记录：
  - 总体决策：接 / 谈判后接 / 暂停 / 拒绝
  - 是否可做、是否值得做
  - 关键风险、必须谈判条件、最小补充信息集
  - 是否允许进入 `brainstorm`
  - `project_engagement_type`：当前项目是 `external_outsourcing` 还是 `non_outsourcing`
- **风险分析执行**：
  - 使用 `/trellis:feasibility` 命令的 `--step risk-analysis` 步骤
  - 使用 Skill 工具执行 `demand-risk-assessment`，完成结构化风险评估
  - 评估流程：结构化抽取 → 踩坑信号扫描 → 冲突检测 → 红线检查 → 结构化评分 → Pre-mortem → 风险登记表 → 决策/谈判条件
  - 将分析结果写入 `assessment.md`，确保包含完整的风险评估结论
- 若结论为 `暂停`，应先补齐缺口后重新执行可行性评估，不应直接进入阶段二。
- 若结论为 `拒绝`，应终止该项目链路，仅保留评估记录。
- 只有 `assessment.md` 明确允许进入 `brainstorm`，或存在经确认仍有效且前提未变化的历史评估记录时，才应进入阶段二；这属于复用已完成阶段结果，不属于跳过 feasibility。

### 1.4 项目确认与初始化
- **客户确认继续推进与预算框架**：基于 feasibility 的商务预判，确认是否继续进入需求澄清与正式估算
- **签署合同/协议**：明确双方权责
- **项目级初始化**：
  - 若客户确认继续推进，则先进行项目状态初始化
  - 正式项目级粗估在阶段二 `brainstorm` 收口后补齐，并回写到 `task_dir/prd.md` 与目标项目 `customer-facing-prd.md`
  - 建立项目档案和后续追踪机制
  - 选择合适的AI开发工具链
  - 建立提示词模板库和上下文管理规范
  - 在目标项目根目录初始化 `todo.txt`
    - 默认初始化内容仅一条：`文档内容需要和实际当前的代码同步`
    - 该文件用于记录项目常规定时任务与协作提醒，不额外改变当前工作流阶段、门禁或命令链路
    - 目标项目根 `README.md` 需说明 `todo.txt` 的存在与用途，避免协作者忽略该约定
  - 若已嵌入对应的自定义工作流，安装脚本应**立即通过脚本导入 `trellis-library` 中的需求发现基础资产**
    - 默认导入：`pack.requirements-discovery-foundation`
    - 导入方式：使用 `trellis-library/cli.py assemble`，不靠人工复制资产或自然语言提示“补装”
    - 导入完成后，若目标项目存在 `trellis init` 创建的 `00-bootstrap-guidelines`，则一并清理；若 `.current-task` 仍指向该 bootstrap task，也同步清理悬空引用；目标项目本就不存在该任务时跳过
    - 最低要求：至少补齐需求发现核心规范 `problem-definition`、`scope-boundary`、`requirement-clarification`、`acceptance-criteria`，再补齐面向客户与面向开发人员的 PRD 规范、模板、检查清单
    - 目的：在正式进入需求发现前，先建立统一的 PRD 约束基线，避免后续只有流程没有规范

#### 1.4.1 外部项目交付控制原则

对于外包、定制开发、新客户项目，默认采用以下"双轨交付控制"：

- **开工前置门禁**
  - 必须先冻结 `kickoff_payment_ratio`
  - 启动款比例最低不少于 `30%`，默认建议 `30%` 或 `40%`
  - 在 `kickoff_payment_received = yes` 前，不得进入 implementation / test-first
- **首选轨：托管部署**
  - 尾款前只提供由开发者控制的演示/试运行环境，客户验证功能与流程
  - 尾款前不移交源码仓库、生产环境密钥、管理员账号、最终部署权限
  - 尾款到账后，再执行源码、部署权限、密钥、账号、运维材料的正式移交
- **备选轨：试运行授权**
  - 仅当客户明确要求先拿可运行版本，且双方接受授权方案时使用
  - 必须在报价、合同、交付说明、验收口径中明确披露"试运行授权"的期限、续期方式、到期行为、永久授权触发条件
  - 到期行为必须可预期，优先限制为"演示模式"或"只读模式"，不得破坏已有数据
  - 尾款到账后，交付永久授权、源码及最终控制权

禁止项：

- 不得植入未披露的后门、隐藏开关、远程锁定或暗门控制逻辑
- 不得使用会破坏数据、阻断导出、导致客户无法恢复业务的失效机制
- 不得使用"不可解码交付文件"伪装为正常源码交付
- 若已明确交付的是试运行版，不得在流程或文档中表述为"正式无限制交付"

若项目要求作者归属保护，还应在 `assessment.md` 冻结：

- `source_watermark_level`
- `source_watermark_channels`
- `zero_width_watermark_enabled`
- `subtle_code_marker_enabled`
- `ownership_proof_required`

这组字段用于决定后续 design / plan / delivery 是否必须补齐源码水印与归属证明链。
它们适用于**所有需要作者归属保护的项目**，不只适用于外部交付项目；这里只是因为 `assessment.md` 的冻结动作在当前阶段一并说明。

**Spec 导入要求（外部项目专用）**：

进入设计阶段（`/trellis:design`）后，技术架构确认前，必须从 `trellis-library` 导入以下交付控制相关 spec 到当前项目 `.trellis/spec/`：

- 外部项目基础必选：
  - `spec.universal-domains.project-governance.delivery-control`（交付控制总控规范）
  - `checklist.universal-domains.project-governance.transfer-checklist`（交付事件 checklist）
- 仅当交付轨道为 `trial_authorization` 时额外导入：
  - `spec.universal-domains.project-governance.authorization-management`（试运行授权规范）
- 仅当本项目会发生密钥、环境变量、第三方平台配置等正式移交时，额外导入：
  - `spec.universal-domains.security.secrets-and-config`

这些 spec 定义了双轨交付控制的详细规则、验证方法和合规检查点。内部项目（非外部交付）无需导入。

这一步只是在外部项目场景下，先补齐**交付控制基线资产**，避免设计阶段没有最基本的交付约束；它**不替代**技术架构确认后的项目级 spec 对齐。等技术架构被用户明确确认后，仍必须继续执行 [§3.7 技术架构确认后的项目 Spec 对齐](#37-技术架构确认后的项目-spec-对齐)。

**双轨交付控制速查表**：

| 决策点 | `hosted_deployment` | `trial_authorization` |
|---|---|---|
| 适用前提 | 客户接受尾款前仅使用托管环境 | 客户明确要求先拿可运行版本，且双方接受授权条款 |
| 开工前提 | `kickoff_payment_ratio >= 30%` 且 `kickoff_payment_received = yes` 后才允许 implementation / test-first | 同左 |
| 机器字段 | `delivery_control_track = hosted_deployment` | `delivery_control_track = trial_authorization` |
| 必选资产 | `delivery-control` + `transfer-checklist` | `delivery-control` + `transfer-checklist` + `authorization-management` |
| 条件资产 | 正式移交时若涉及密钥/配置，再加 `secrets-and-config` | 正式移交时若涉及密钥/配置，再加 `secrets-and-config` |
| 尾款前默认保留 | 源码、生产密钥、管理员账号、最终部署控制权 | 永久授权、完整源码、最终控制权；并明确试运行限制 |
| 最终移交触发 | 一般为 `final_payment_received` | 一般为 `final_payment_received` |

> 此阶段体现了"先调研再实现"的方法论，确保项目在启动前已充分评估。

## 🎯 阶段二：需求明确与确认

> **前置条件**：新项目默认已完成阶段一可行性评估，并且 `assessment.md` 明确允许进入需求发现；否则应先回到 `/trellis:feasibility` 补做评估，而不是直接开始 `brainstorm`。

### 2.1 详细需求分析
- **Task-first**：先确保当前讨论落到 `task_dir/prd.md`，不让需求只停留在对话里
- **Action-before-asking**：先读仓库、文档、现有任务与历史约束，能查不问
- **Research-first**：涉及方案取舍、实现路径或行业约定时，先调研再收敛
- **Diverge → Converge**：先扩展边界、异常路径和演进场景，再收敛到本轮 MVP
- **AI可实现性评估**：评估需求的可 AI 化程度，区分可直接 AI 生成、需要人工设计、需要 RAG 增强、需要 Agent 架构等类型

### 2.1.1 `/trellis:brainstorm` 合并规则

`/trellis:brainstorm` 采用“**Trellis 原生需求发现主链 + 当前 workflow 门禁扩展**”的合并模型。

在生成任何具体任务、子任务或执行安排前，必须先沿用原生 `brainstorm` 的主链完成需求发现，再进入当前 workflow 的统一前置判定，不能直接跳到任务拆解或实现。

前置判定顺序固定为：

1. **先检查 feasibility 是否已完成且 assessment 仍有效**
   - 若缺少有效 `assessment.md`，或当前前提已变化，先回 `/trellis:feasibility`，不允许直接开始 `brainstorm`
   - 若 assessment 仍有效，则复用既有评估结果继续需求发现，不需要重复执行 feasibility

2. **先保证任务与工作底稿存在**
   - 若当前没有任务目录，先创建任务，再种下 `task_dir/prd.md`
   - 若当前已有任务目录，先复用并更新 `prd.md`

3. **先自动收集上下文，不先问低价值问题**
   - 先读取 `assessment.md`、现有 `prd.md` / `task_plan.md` 摘要、已存在的 task / 子任务列表、目标项目 `docs/requirements/*.md`
   - 先做仓库检查、模式对照和必要调研，再决定是否需要问用户
   - 问题只允许是阻塞型或偏好型，而且一次只问一个

4. **需求描述准确性校验**
   - 判断需求目标、范围、验收标准、关键边界是否足够清晰
   - 若仍存在明显歧义、缺失信息或隐含假设未处理，继续停留在 `brainstorm`，不生成具体任务

5. **复杂度统一分类（`L0/L1/L2`）**
   - `L0 简单任务`：需求已准确，单个上下文可闭环，通常只涉及 1-2 个文件或一个明确变更点
   - `L1 标准任务`：需求已准确，但涉及多文件、多步骤或明确依赖，需要进入 `plan` 拆任务
   - `L2 复杂任务`：跨模块、跨层、架构级、高风险或高不确定，需要先发散补充，再拆成多个子任务

6. **是否需要先发散补信息**
   - 若当前边界、相邻流程一致性、历史兼容约束、异常路径或跨层契约会影响拆分方式，先补信息，再进入具体任务生成
   - `L2` 默认必须先发散补充；`L1` 按需进行；`L0` 通常可直接收敛

7. **是否需要拆 `sub task`**
   - 若当前事项无法在单个上下文中完整闭环，则必须拆成多个可独立验证、可独立收尾的子任务
   - 若当前事项已满足单任务闭环条件，则允许作为单任务继续推进

8. **项目级正式需求文档门禁**
   - 在进入 `design` 之前，必须先在目标项目生成并同步：
     - `docs/requirements/customer-facing-prd.md`
   - 在离开 `brainstorm` 之前，还必须先产出**不可跳过的项目级粗估**：
     - `task_dir/prd.md` 中的 `## 项目级粗估`
     - `docs/requirements/customer-facing-prd.md` 中的 `## 项目级粗估摘要`
   - 若暂时无法给出单点值，也必须给出区间、置信度与适用前提；不得以“后续再估”为由跳过
   - `docs/requirements/developer-facing-prd.md` 不在架构确认前强制生成；它应在 design 阶段的技术架构确认后再正式落盘
   - `task_dir/prd.md` 仅作为当前任务的工作底稿，不可替代项目级正式需求文档
   - 当前 workflow 通过 `workflow-state.py validate` 将这条粗估门禁卡在 `design` 及后续阶段入口上；缺失时不得继续推进
   - 后续若命中 [需求变更管理执行卡](./需求变更管理执行卡.md) 且变更获批，必须同步更新受影响的项目级正式文档
   - 若下一步进入 `design`，则在 `design -> 3.7 技术架构确认后的项目 Spec 对齐` 阶段必须明确项目自动化检查矩阵：采用 Sonar 的项目必须写真实命令，未采用时必须写替代门禁和原因

9. **阶段状态机与强门禁**
   - 当前 workflow 使用 `.trellis/.current-task -> 当前叶子任务 -> $TASK_DIR/workflow-state.json` 作为唯一阶段判定链
   - 每个阶段完成后都必须先进入 `awaiting_user_confirmation`，用户明确确认后才能切到下一阶段
   - `/trellis:start` 只能重入当前已确认阶段，不能自动跨阶段推进

### 2.1.2 单上下文单任务原则

- 单个上下文只负责一个任务，不允许在一个上下文里并行推进多个任务并混用验收口径。
- `L0` 任务允许在 `brainstorm` 完成后直接收敛为单任务执行，不强制先生成 `task_plan.md`。
- `L1/L2` 任务在进入实施前，必须先形成明确的拆解结果；其中 `L2` 先补信息，再拆多个子任务。
- 单个任务上下文必须覆盖完整闭环：**代码实现 → 代码检测 → 质量检测 → 代码修复**。
- 当前 workflow 将 implementation 阶段内部链统一定义为：`research -> implement -> check-agent`。
- 这里的 `check-agent` 属于 implementation 内部自检角色，不等于正式 `/trellis:check` 阶段。
- implementation 内部链完成后，只能把 `/trellis:check` 作为候选下一阶段，等待用户明确确认，不得自动切换。

### 2.1.3 历史数据防漂移约束

- 新增规则默认只约束**后续新生成或重新确认的任务**，不回填或重写历史 `prd.md`、历史 `task_plan.md`、历史任务状态枚举和既有审查记录。
- 若旧任务、已冻结需求或已批准计划需要调整范围、拆分方式、验收标准或执行路径，统一走 [需求变更管理执行卡](./需求变更管理执行卡.md)，不允许静默修改历史基线。
- 在生成新任务或子任务前，先检查当前 `prd.md`、`task_plan.md` 摘要和已存在的 task / 子任务列表；若已存在等价任务，则直接复用，避免重复建任务导致历史数据漂移。

### 2.2 需求说明书编制

#### 2.2.0 项目级正式需求文档落位（强制）

在当前 workflow 中，项目级正式需求文档统一放在目标项目：

```text
docs/requirements/
├── customer-facing-prd.md
└── developer-facing-prd.md
```

说明：

- 这是**新项目开发工作流**对使用方项目的要求，不是当前 workflow 仓库自身必须常驻生成这两份文件
- 当这套 workflow 被嵌入并运行在某个目标项目时，该规则才对目标项目生效

约束：

- `customer-facing-prd.md` 在进入 design 前必须存在
- `customer-facing-prd.md` 在进入 design 前还必须补齐 `## 项目级粗估摘要`
- `developer-facing-prd.md` 在技术架构确认前不强制生成；它应在 design 阶段技术架构确认后再正式落盘
- 两份文档最终覆盖项目中已确认的全部需求功能点
- 客户向与开发向文档的目标、范围、验收、边界含义必须一致
- `task_dir/prd.md` 只承担阶段内澄清与路由判断，不替代上述正式文档
- `task_dir/prd.md` 在离开 brainstorm 前必须补齐 `## 项目级粗估`，否则不得进入 design
- 后续正式需求变更获批后，必须持续同步回写受影响的正式文档

#### 客户版需求说明书（对外）

形成《客户版需求说明书》(PRD)，仅包含功能描述，不含AI技术细节。

推荐目标路径：`docs/requirements/customer-facing-prd.md`

默认对齐以下 `trellis-library` 资产：

- `spec.universal-domains.product-and-requirements.problem-definition`
- `spec.universal-domains.product-and-requirements.scope-boundary`
- `spec.universal-domains.product-and-requirements.requirement-clarification`
- `spec.universal-domains.product-and-requirements.acceptance-criteria`
- `spec.universal-domains.product-and-requirements.prd-documentation-customer-facing`
- `template.universal-domains.product-and-requirements.customer-facing-prd-template`
- `checklist.universal-domains.product-and-requirements.customer-facing-prd-checklist`

说明：以上 `spec.*` ID 指向一个 concern directory，实际展开为 `overview.md`、`scope-boundary.md`、`normative-rules.md`、`verification.md` 四个子文件。

##### 项目规则定义

- **架构边界**：
  - 系统边界：明确系统与外部系统的交互范围
  - 模块边界：各功能模块之间的职责划分
  - 技术边界：禁止使用的技术/框架

- **质量门槛**：
  - 性能要求：响应时间、加载速度
  - 可用性要求：在线时长、故障恢复
  - 安全要求：数据安全等级、权限要求

- **兼容性要求**：
  - 浏览器兼容：支持的浏览器版本
  - 设备兼容：支持的手机/平板尺寸
  - 系统兼容：支持的操作系统版本

##### 用户故事

格式：作为[角色]，我希望[功能]，以便[价值]

##### 功能边界

- **核心功能**：必须实现的功能
- **扩展功能**：可选实现的功能
- **明确不做**：主动排除的功能范围

##### 验收标准

- 功能验收：每个功能的验收条件
- 性能验收：性能指标达标标准
- 体验验收：用户操作体验要求

##### 模糊点澄清记录

- 歧义问题1：问题描述 → 澄清结果
- 歧义问题2：问题描述 → 澄清结果
- 遗留问题：暂时无法确定的问题及后续跟进计划

##### 冻结与变更约定（对客）

客户版 PRD 建议新增一个简短且固定的“冻结与变更约定”小节，用于提前管理预期，而不是等客户在后续阶段提变更时才第一次说明。

至少要写清三件事：

- 本轮冻结的范围是什么
- 冻结后“纯澄清”和“正式变更”的区别
- 正式变更会先评估影响范围、工期和费用，确认后再并入当前项目

推荐对客写法：

> 本文档确认的是本轮交付范围与验收标准。为保证当前版本按时交付，需求冻结后，如有新增、修改或删除需求，我们会先评估对范围、排期和费用的影响，并在你确认后再纳入执行；如果只是对原有需求的澄清，且不改变范围和验收标准，我们会直接在当前阶段处理。

### 2.4 需求冻结

> **核心原则：冻结的是"业务意图与验收证据"，而非"实现细节"**

#### 冻结对象（明确）

| 冻结内容 | 冻结程度 | 变更方式 |
|---------|---------|---------|
| **业务意图** | 绝对冻结 | 需走完整变更流程 |
| **验收证据** | 绝对冻结 | 需新增证据而非修改 |
| **接口契约** | 绝对冻结 | 需走技术变更评审 |
| ~~Prompt细节~~ | 不冻结 | 可持续优化 |
| ~~模型参数~~ | 不冻结 | 可调优 |

- 冻结对象：
  - 客户版PRD：`docs/requirements/customer-facing-prd.md`（业务范围与验收标准）
  - 开发版需求文档：`docs/requirements/developer-facing-prd.md`（需求实现说明、模块拆解与任务边界、场景/规则/验收映射；接口/数据库正文以跳转链接承接）
  - 评估基线（核心测试集与通过阈值）
- 非冻结对象：
  - Prompt细节、模型参数、工具调用策略允许迭代优化
- 优化前提：
  - 每次Prompt或模型调整必须记录变更原因
  - 必须通过评估基线回归后才能进入下一阶段
  - **Prompt迭代必须在评估基线回归后才能生效，不允许直接上线未评估的Prompt变更**
  - **模型版本升级视同重大变更，需重跑全量评估集**
  - **工具调用策略变更若影响输出格式或接口契约，必须进入2.5变更流程**
- 后续变更：
  - 涉及范围、接口、成本、工期变化时，必须进入2.5变更流程

#### 2.4.1 冻结后的跨阶段分流规则

从需求冻结完成开始，后续任一阶段（设计、计划、测试先行、实施、质量检查、交付）如再次出现需求讨论，先按“澄清 / 变更”分流，不得在当前阶段直接混入处理。

- **纯澄清**：允许留在当前阶段直接处理
  - 前提：不改变范围、接口契约、验收标准、成本、工期
  - 典型情况：解释原有需求含义、补充术语说明、确认已有描述指向什么行为
- **正式变更**：统一进入 `2.5 需求变更管理`
  - 触发类型：新增需求、修改需求、删除需求
  - 处理要求：先评估影响范围、成本、工期、测试与回归范围；未确认前，不并入当前阶段继续推进

#### 2.4.2 澄清与变更的判定口径

不要按客户是否说出“新增”“修改”字样来判断，而按影响来判断。

只要命中以下任一项，就视为正式变更：

- 功能范围发生变化
- 页面或业务流程行为发生变化
- 接口契约或数据契约发生变化
- 验收标准或测试证据发生变化
- 成本发生变化
- 工期或里程碑发生变化

只有在**不改变**上述六项时，才可视为“纯澄清”，留在当前阶段处理。

#### 2.4.3 典型口语触发示例

以下口语表达，若发生在需求冻结后，默认按正式变更处理：

- “这个功能再加一个”
- “这个页面流程我要改一下”
- “这个接口字段得调整”
- “这个验收标准我想改”
- “这个功能先不做了”

以下口语表达，若不改变范围和验收，可按纯澄清处理：

- “这里我确认一下是什么意思”
- “这个描述对应的是不是原来那个流程”
- “我不是加功能，只是想确认你理解的是否和我一致”

#### 2.4.4 冻结后变更的执行出口

- 当前阶段的责任是**识别并中断**，而不是直接吸收正式变更继续推进
- 若判断为正式变更，先执行 `2.5 需求变更管理`
- 若变更获批，再回到**受影响的最早阶段**继续，例如：
  - 影响业务范围或验收标准：回到需求/设计层
  - 影响任务拆解、排期、测试范围：回到计划或测试先行
  - 仅在批准后影响实现与验证：回到实施或后续验证阶段

#### 2.4.5 AI 意图映射确认

在需求冻结前，增加"AI 意图映射确认"循环：

```
AI 意图映射流程：

① 需求冻结触发
    └── 开发者将客户需求输入 AI

② AI 生成结构化意图列表
    └── 格式：
    - 用户故事 → 核心意图 → 关键约束 → 验收证据

    示例：
    | 用户故事 | 核心意图 | 关键约束 | 验收证据 |
    |---------|---------|---------|---------|
    | 作为用户，我希望能... | 实现xxx功能 | 响应时间<200ms | 性能测试通过 |

③ 开发者复核
    └── 核对 AI 理解与客户描述是否一致
    └── 标记潜在误解或假设

④ 客户签字确认
    └── 客户确认意图理解正确
    └── 确认后纳入冻结范围

⑤ 后续争议仲裁
    └── 如实现过程中出现理解偏差
    └── 以 Intent.md 中签字确认的意图为准
```

> 这一阶段体现了"证据优先"和充分调研的原则，为后续工作奠定基础。

### 2.5 需求变更管理

当客户在需求冻结后提出新增需求、修改需求或删除需求时，需要通过规范化的变更管理流程进行处理。纯澄清若不改变范围、接口契约、验收标准、成本、工期，可留在当前阶段直接处理，不必升级为正式变更。

#### 2.5.1 变更类型定义

| 变更类型 | 定义 | 示例 |
|---------|------|------|
| **新增需求** | 在原有需求基础上增加全新的功能模块 | 增加用户评论功能、增加支付方式 |
| **修改需求** | 对已确认的需求内容进行修改 | 修改登录流程、调整页面布局 |
| **删除需求** | 移除已确认的需求功能 | 取消某个不需要的功能 |
| **需求澄清** | 对已有需求的模糊部分进行澄清 | 明确某个功能的详细行为 |

#### 2.5.2 变更影响评估（重大变更可选）

> 本节仅适用于**重大变更**（通常 >2 人天、改接口、改验收标准或明显影响排期）。轻微和一般变更默认走 `§2.5.3` 的轻量闭环。

重大变更建议至少回答这四类问题：

| 评估维度 | 需要回答的问题 |
|---------|-------------|
| **功能影响** | 影响哪些页面、流程、模块或接口？ |
| **技术影响** | 需要改哪些代码或结构？是否有明显技术风险？ |
| **测试影响** | 需要新增/修改哪些测试？回归范围多大？ |
| **项目影响** | 预计增加多少工时？是否影响当前里程碑、费用或交付节奏？ |

如项目明显依赖 AI 资产，可额外备注 Prompt、评估基线或上下文是否需要重建，但默认不要求单独维护一套 AI 系数表。

评估结论统一收敛为：**接受 / 有条件接受 / 拒绝**。

#### 2.5.3 变更处理流程

变更处理只保留一个核心闭环：**估价 → 确认 → 执行 → 验证**。

**核心流程（所有正式变更必走）**：

```text
① 估价
   └── 给出时间、费用、是否影响当前交付的快速判断

② 确认
   └── 客户选择：
       接受并执行 / 暂缓 / 拒绝

③ 执行
   └── 按确认结果更新项目级双需求文档、任务底稿、计划、实现与测试

④ 验证
   └── 完成变更验证与必要回归，再由客户确认
```

**按级别路由**：

| 变更级别 | 估价范围 | 流程路由 |
|---------|---------|---------|
| **轻微** | ≤0.5天 | 走完①②③④，口头确认即可，事后补一行记录 |
| **一般** | 0.5-2天 | 走完①②③④，使用**轻量变更单**（§2.5.5） |
| **重大** | >2天 或明显影响接口/验收/排期 | 走完①②③④，补充 **§2.5.2 影响评估**，并使用**完整变更单**（§2.5.5）或等效书面记录 |
| **紧急** | 不限 | 先止血，24小时内补齐①②③④记录 |

> 所有正式变更建议保留文字记录（微信/邮件/变更单均可），避免口头约定产生歧义。

**变更流程结束后的返回原则**：

- **未获批 / 暂缓 / 拒绝**：保持当前阶段冻结基线，不把变更混入当前交付
- **已获批**：回到受影响的最早阶段，更新 PRD / 设计 / 计划 / 测试 / 实施基线后，再恢复主链路

#### 2.5.4 变更估价参考（可选）

快速估价的目标是**尽快报出可执行判断**，不是制造一套看起来精确但难以落地的计算公式。

建议按这个顺序估：

1. 先估算新增或返工的**基准工时**
2. 再判断当前阶段的**返工成本**（越晚提出，通常越贵）
3. 再判断是否存在**跨模块 / 跨层连锁影响**
4. 最后补上**测试与回归成本**

对外通常只需要明确三件事：

- 预计新增工期
- 预计新增费用
- 是否影响当前交付版本

> 如果短时间内无法给出精确数值，可以先给**区间报价**，不要为了“看起来专业”强行套复杂系数。

#### 2.5.5 变更单模板

个人开发或小团队场景，默认只保留两档模板：**轻量版（默认）** 和 **完整版（重大变更可选）**。

| 场景 | 使用模板 |

| 重大正式变更 | 完整变更单 + §2.5.2 影响评估 |

#### 2.5.6 项目级需求文档更新规范（推荐执行细则）

变更获批并执行后，按以下规范更新项目级双需求文档：

- `docs/requirements/customer-facing-prd.md`
- `docs/requirements/developer-facing-prd.md`

同步要求：

- 两份文档必须描述同一组已确认需求，禁止只更新其中一份导致语义漂移
- 客户向文档更新业务可见变化、范围边界、验收影响
- 开发向文档同步更新实现约束、接口/数据契约、测试与风险信息
- 若当前执行任务仍依赖 `task_dir/prd.md` 作为上下文依据，也应同步修正，但它不是正式项目基线

**版本号规则**
- 格式：`v主版本.次版本.修订号`
- 变更类型对应：
  - 新增/修改需求 → 次版本 +1（如 `v1.2.0 → v1.3.0`）
  - 需求澄清/错误修正 → 修订号 +1（如 `v1.2.0 → v1.2.1`）

**变更标记规范**
- 在 PRD 末尾新增“变更记录”章节
- 每次变更添加一行记录：

| 版本 | 日期 | 变更单号 | 变更内容摘要 | 变更人 |
|------|------|---------|-------------|--------|
| v1.3.0 | 2024-03-20 | CHANGE-003 | 增加订单导出功能 | 张三 |

- 在正文中用高亮标记变更部分：
  > [变更v1.3.0] 新增订单导出功能，支持 Excel 格式下载

**历史版本保留**
- 保留最近 3 个版本的正式需求文档文件
- 命名格式示例：
  - `customer-facing-prd-v1.3.0-2024-03-20.md`
  - `developer-facing-prd-v1.3.0-2024-03-20.md`
- 归档位置：`/docs/requirements/history/`

#### 2.5.7 对客沟通要点（简版）

冻结后谈变更，重点不是“强调收费”，而是守住三个目标：

- 本轮交付边界不被悄悄改写
- 客户知道“不是不能改，而是先评估再决定”
- 流程透明，可选择，不靠临时口头承诺推进

建议只保留三条表达原则：

- **先接住诉求，再判定澄清还是正式变更**
- **先说明不直接混入当前交付，再谈时间和费用**
- **给客户明确选项：本轮不动 / 并入本轮并调整排期 / 留到下一轮**

推荐短话术：

> 这个想法可以做。为了不影响当前已经确认的交付，我先把它按正式变更评估一下，给你时间、费用和是否影响本轮排期的判断；你确认后，我再决定并入本轮还是排到下一轮。

### 3.1 UI/UX设计

#### 3.1.1 AI 粗稿生成（如有条件）

**目的**：利用AI快速生成设计初稿，加速设计流程

**阶段提醒（强制）**：

- 只要项目已经进入“需要外部 UI 设计/页面原型设计”的阶段，就必须明确提醒用户离开当前 CLI，去外部设计站点完成 UI 设计动作。
- 推荐固定链路：先去 `https://www.uiprompt.site/zh/styles` 获取 UI 风格提示词，再由当前 CLI 生成 `design/STITCH-PROMPT.md`，最后去 `https://stitch.withgoogle.com/` 按页面 / 流程生成 UI 原型。
- 不允许只在当前对话里停留在抽象描述，却不提醒用户去实际完成外部 UI 原型。
- **执行边界（强制）**：
  - `UI 原型生成` 这一步禁止使用 Codex 作为主执行器
  - 允许作为主执行器的 CLI 只有 Claude Code / OpenCode
  - Codex 只能做文档整理、Prompt 文本润色、原型结果回收，不能承担视觉原型主执行入口

**执行流程**：
```
AI 粗稿生成流程：

① 输入准备
    ├── 需求文档（BRD/PRD）
    ├── 业务流程说明
    └── 目标用户画像

② AI 生成粗稿
    ├── 页面布局建议
    ├── 组件清单
    ├── 交互流程草图
    └── 配色方案参考

③ 人工初审
    ├── 审核设计方向
    ├── 标记需要调整的点
    └── 确认进入精细化设计
```

**AI 粗稿输出示例**：
```markdown
# AI 设计粗稿

## 页面结构
- 首页：Header + Hero + Features + CTA
- 列表页：Sidebar + Content Grid
- 详情页：Title + Media + Description + Related

## 组件建议
- 按钮：Primary/Secondary/Ghost
- 卡片：Image + Title + Description + Action
- 表单：Input + Select + DatePicker

## 交互流程
- 用户登录流程
- 商品浏览 → 加入购物车 → 结账
```

#### 3.1.2 外部工具精细化设计与 Stitch Prompt 固定模板

**推荐工具**：Stitch、Figma、Sketch 等

**执行流程**：
```
外部工具精细化设计：

① 外部 UI 提示词准备
    ├── 从 PRD/BRD 提取页面目标、核心流程、目标用户
    ├── 访问 https://www.uiprompt.site/zh/styles
    └── 获取或整理贴近目标风格的 UI 提示词

② 生成固定骨架 Prompt
    ├── 在当前 CLI 中整理 design/STITCH-PROMPT.md
    ├── 先写项目级总上下文 Prompt
    └── 再按页面 / 流程写执行 Prompt 小节

③ AI 粗稿导入 / 原型生成
    ├── 从 design/STITCH-PROMPT.md 中提取当前页面 / 流程 Prompt
    ├── 访问 https://stitch.withgoogle.com/
    └── 按页面 / 流程逐个生成首版 UI 原型

④ 精细化设计
    ├── 像素级UI设计
    ├── 交互原型制作
    ├── 动效设计
    └── 设计规范定义

⑤ 设计交付
    ├── 设计稿切片/标注
    ├── 组件规范文档
    └── 开发切图资源
```

**进入本阶段时的建议提醒话术**：

> 当前已经到达需要外部 UI 设计的阶段。请先去 `https://www.uiprompt.site/zh/styles` 获取合适的 UI 风格提示词，再在当前 CLI 中按固定模板整理 `design/STITCH-PROMPT.md`，然后按页面 / 流程把 Prompt 带到 `https://stitch.withgoogle.com/` 生成 UI 原型；完成后再回到工作流继续编写页面说明、规格说明和技术方案。注意：`UI 原型生成` 这一步不能使用 Codex 作为主执行 CLI。

**Stitch 工具集成（如使用）**：
- 将 `design/STITCH-PROMPT.md` 中当前页面 / 流程 Prompt 与风格词一起导入 Stitch
- 使用 Stitch 的 AI 功能进一步优化
- 导出结构与视觉参考资料，用于回收设计结论，不直接作为正式实现输入

**`design/STITCH-PROMPT.md` 固定规则**：

- 默认一个文件同时承载：
  - 项目级总上下文 Prompt
  - 页面 / 流程级执行 Prompt
- 默认按**单页面 / 单流程**给 Stitch 执行，不采用“整站一次性大 Prompt”
- 页面明显复杂时，才允许补充页面级 Prompt 文件
- 必须带“去 AI 味”全局禁止项，而且项目只能增补，不能删减基线项

建议默认基线禁止项至少包括：

- 不要通用 SaaS 模板感
- 不要廉价渐变和无意义炫光装饰
- 不要过度圆角、过度玻璃拟态、过度悬浮阴影
- 不要无信息密度的卡片堆砌
- 不要与业务无关的装饰性图形或占位文案
- 不要“英雄区 + 三栏卖点 + 泛化插画”的通用 AI 生成组合

#### 3.1.5 设计阶段文档矩阵（强制）

设计阶段不再要求在 `design/` 目录内单独创建 `BRD.md`。业务需求主文档由：

- `docs/requirements/customer-facing-prd.md`：承担 BRD 主文档职责；进入 design 前必须存在
- `docs/requirements/customer-facing-prd.md`：承担 BRD 主文档职责；进入 design 前必须存在，并补齐 `## 项目级粗估摘要`

设计阶段的硬必选文档：

- `design/index.md`（设计目录索引文件；供 scaffold / validate 与协作者导航使用）
- `design/TAD.md`
- `design/ODD-dev.md`
- `design/ODD-user.md`
- 项目根 `README.md`（最低可用版：项目简介、技术栈、目录结构、文档索引、启动方式占位、部署/使用文档链接）

设计阶段的条件必选文档：

### 3.3 可执行原型（HTML / Figma / Storybook 三选一）

将关键流程做成可验证原型，目标是快速确认交互与状态，而不是固定工具形式。

#### 3.3.0 原型资产隔离规则

原型只用于验证交互与视觉结论，不直接进入正式实现。

**禁止直接带入正式实现**：

- 照片、截图、录屏、预览图
- Figma / Stitch / HTML / Storybook 原型文件、分享链接
- 原型工具导出的组件代码、页面源码、临时演示代码
- 为原型验证临时拼接的网页源码、类名、演示数据结构

**允许保留的结论**：

- 页面层级、区块划分、组件清单
- 主流程 / 支流程 / 异常流程
- 状态集合、交互反馈、可访问性约束
- 视觉方向、布局约束、必要动效结论

**执行要求**：

- 允许保留的结论必须先转写为结构化设计输入，再供正式实现使用
- 若后续存在 `UI -> 首版代码界面` task，应在该 task 完成时沉淀 `design/frontend-ui-spec.md`
- 即使原型产物附带网页源码，也不能直接把该源码当作正式实现输入

#### 3.3.1 最小原型范围
- 必须覆盖1条主流程 + 1条异常流程 + 1条空数据流程

#### 3.3.2 状态完整性
- 正常态、加载态、空态、错误态、权限不足态

#### 3.3.3 交互可验证性
- 标注入口、出口、前置条件、成功/失败反馈

#### 3.3.4 文案与错误码
- 关键提示文案统一维护，前后端错误码映射清晰

### 3.4 页面与交互说明

为每个页面提供详细的设计说明文档，确保开发人员准确理解设计意图。

#### 3.4.1 页面清单
- 列出所有页面及其编号

#### 3.4.2 页面定位
- **入口来源**：从哪个页面跳转而来
- **出口去向**：可以跳转到哪些页面
- **使用场景**：什么情况下会访问此页面

#### 3.4.3 页面结构
- **顶部区域**：标题、返回按钮、更多操作
- **内容区域**：主要展示内容
- **底部区域**：操作按钮、导航栏

#### 3.4.4 关键流程
- **主流程**：核心用户路径
- **支流程**：分支场景处理
- **异常流程**：错误处理路径

- **加载态**：数据获取中的展示
- **空数据态**：无数据时的展示

- **编辑态**：可编辑状态

- **滑动事件**：滑动行为及反馈

#### 3.4.7 提示文案汇总
| 场景 | 文案内容 | 触发条件 |
|------|---------|---------|

## 📝 阶段四：任务分解与准备（Trellis task-first）

### 4.1 AI 驱动的任务自动拆解

#### 4.1.1 任务图生成机制

**原流程痛点**：把复杂执行长期堆在 `task_plan.md` 里，会让摘要文档承担运行态控制，逐步偏离 Trellis 的 task-first 主模型。

**修正方案**：`plan` 阶段优先拆成真实 Trellis task，`task_plan.md` 只保留任务图与门禁摘要。

**硬禁令**：`plan` 阶段必须强制避免执行具体任务。它只允许拆 task、写摘要、定依赖与门禁，不允许生成项目基础代码、不允许编写实现代码、不允许直接进入 implementation。

**状态机门禁**：`plan` 阶段 `checkpoints.execution_authorized` 必须保持 `false`；只有用户明确确认进入 `implementation` / `test-first` 后，才允许显式切换为 `true`。

**吸收 Trellis plan-agent 的优点，但不接入 dispatch 自动链**：

- 先做“是否已经可以拆”的 readiness gate，而不是一上来就硬拆 task
- 至少把当前推荐执行的 leaf task 补成 task-ready 状态，避免进入 Trellis 基线后被判定为 `NOT READY`
- 仍然保持当前 workflow 的人工确认门禁，不恢复 `plan -> dispatch -> create-pr` 自动流水线

**执行流程**：

```

    ├── 关联文档路径（BRD/TAD/DDD/IDD等）
```

#### 4.1.2 拆分原则

- 单个 task 独立可测试
- 单个上下文可完整实现（Token 预算内）
- 先拆真实 task，再写摘要
- 当前推荐执行的 leaf task 至少要具备最小 `prd.md`
- 同一项目域内默认串行
- 多项目域可分 lane，但 lane 内仍默认串行
- 串行不等于自动续跑
- 若某个 task 过大、跨越太多上下文，必须继续拆成多个串行 task
- **AI 可执行性**：标记哪些 task 可以由 AI 主导完成
- **人工介入点**：标记必须人工确认的关键节点

##### 执行顺序规则

- 有前置依赖的 task，只有前置 task 收口后才允许作为下一个候选
- 同一项目域内默认串行，不再把“推荐并行组”作为主链规划中心
- 若目标项目确有多个相对独立的项目域，可按项目域分别维护串行 lane
- 即使前一 task 已完成，也不会自动启动下一 task；必须重新进入 `start`

##### task_plan 的边界

- `task_plan.md` 是摘要层，不再记录运行态状态矩阵
- 真实执行状态依赖 Trellis tasks、`.current-task`、`before-dev.md`、`check.md`
- 不再把“当前可开始任务 / 等待中任务 / 推荐并行组 / 任务执行矩阵”作为计划文件的主结构

| 任务名称 | [任务名称] |
| 所属模块 | [模块名称] |
| 任务类型 | 新增/修改/删除/重构 |
| 风险级别 | L0/L1/L2（低/中/高） |
| AI执行模式 | 自动生成/AI草拟+人工修订/纯人工 |
| 评估集ID | EVAL-xxx |
| 上下文管理 | 确保不超出模型处理能力 |
| 回退策略 | 模型失败后的备选路径 |
| 预估工时 | X小时或X人天 |

##### 具体需求

- 需求描述：清晰描述需要实现的功能
- 需求来源：来自哪个用户故事或需求
- 需求优先级：必须/重要/可选

##### 功能边界

- 包含范围：明确任务要做什么
- 排除范围：明确任务不做什么
- 边界条件：特殊的边界情况处理

##### 技术约束

- 技术要求：使用的技术栈
- 接口要求：涉及的接口
- 数据要求：涉及的数据表
- 性能要求：性能指标要求

##### 验收标准

- **功能验收**：
  - 验收点1：描述 → 测试方法 → 预期结果
  - 验收点2：...
- **边界验收**：
  - 边界点1：边界情况 → 测试方法 → 预期结果
- **异常验收**：
  - 异常场景1：异常情况 → 处理方式 → 提示文案
- **性能验收**（如果适用）：
  - 响应时间要求
  - 并发能力要求

##### 依赖关系

- 前置任务：需要先完成的 task
- 所属项目域：当前 task 属于哪个串行 lane
- 是否为终局任务：例如 PROJECT-AUDIT / 交付控制类 task

##### 执行条件

- 开始条件：前置 task 完成、输入资源就绪、无冲突确认点
- 进入实现前必须再次执行一次 before-dev
- before-dev 产出的 `$TASK_DIR/before-dev.md` 是本次进入实现前的有效门禁快照

##### Definition of Ready (DoR) — 开始前必须满足

- [ ] 需求描述无歧义，验收标准可测试
- [ ] 接口/数据契约已确认（或已有Mock）
- [ ] 依赖任务已完成或可并行进行
- [ ] 评估集ID已关联，回归用例不少于1条
- [ ] 上下文预算已估算，超预算处理方案明确

##### Definition of Done (DoD) — 完成标准

- [ ] 功能验收全部通过
- [ ] 本地门禁全通过（按项目自动化检查矩阵执行，如 ESLint/Ruff/TypeCheck/Build/质量平台/单测）
- [ ] 绑定的评估集回归通过
- [ ] 代码已经过风险分级对应的评审
- [ ] 无新增安全高危项

补充要求：
- 每个任务必须写明 `Definition of Ready (DoR)` 与 `Definition of Done (DoD)`
- 每个任务必须绑定至少1条回归用例
- 高风险任务必须指定人工评审人（单人项目可指定"次日复审"机制）

#### 4.1.4 任务边界明确清单（AI预检 + 人工确认）

每个任务在开始前必须确认以下内容：

**□ 需求理解确认**

**□ 测试门禁确认**

**边界条件验收**：

### 4.2 任务排期（基于 task 图摘要）

- 基于 AI 生成的 task 图摘要进行排期
- 根据 task 优先级、项目域和依赖关系安排主链
- 多项目域时可以并行推进不同 lane，但每个 lane 内默认串行
- **AI 加速效应考虑**：AI 生成代码的审查时间预估（按代码行数/复杂度）

排期时应遵循以下最小规则：

- 先安排无可替代的前置 task，避免后续 task 整体空等
- 再安排真正独立的其他项目域 lane
- 并行只是优化手段，不是强制目标；边界不清晰时默认退回串行

### 4.3 手动测试驱动入口（Test-First）

`test-first` 不再是默认主链阶段。

默认主链中：

- task 级测试门禁不在 plan 阶段预造
- 进入某个 task 实现前，由 `/trellis:start` 自动执行 `before-dev`
- 当前 task 的有效门禁快照落到：

```text
$TASK_DIR/before-dev.md
```

显式进入 `/trellis:test-first` 的场景：

- 用户明确要求 TDD
- 需要先把某个 task 的验收标准转成测试
- `check` 明确指出应先补测试证据

## 🔄 阶段五：开发实施循环

### 5.1 单个需求开发流程（task-first + 自动 before-dev）

① 选定当前 task
- 优先基于真实 Trellis task 图选择本轮唯一 task
- 默认一次只推进一个具体 task
- 串行不等于自动续跑；即使前一个 task 已完成，也必须显式重新进入主链选定下一个 task

② 自动执行 before-dev
- `/trellis:start` 在真正实现前自动执行一次 before-dev
- 不要求用户显式输入 `/trellis:before-dev`
- before-dev 负责读取当前 task 相关 spec / guides / 项目约束

③ 生成当前 task 的门禁快照
- 自动生成或刷新 `$TASK_DIR/before-dev.md`
- 记录：
  - 当前 task 适用的 spec / guides
  - 项目级全局测试基线中当前 task 必须继承的项
  - 当前 task 新补充的测试门禁
  - 当前不做的验证项及原因

④ DoR 检查
- 需求边界、输入输出、验收标准、回归范围明确后才开始实现
- 若 `before-dev.md` 未完成，不进入实现

⑤ AI 实现（可选多轮）
- 结构化 Prompt 生成代码草案
- 对关键输出启用 Schema 约束，减少格式与字段错误

⑥ 本地自动门禁
- 必须执行在阶段 3.7 已明确的项目自动化检查矩阵
- 不允许只跑默认 `Lint` 后就进入提交或交付
- 至少覆盖当前项目真实需要的静态分析、类型/编译检查、测试、安全扫描；必须有明确质量平台门禁

⑦ 风险分级评审
- L0 低风险：自动门禁通过即可进入 PR
- L1 中风险：增加人工快速审查
- L2 高风险：必须完整人工审查 + 安全复核

⑧ 当前 task 收口
- 当前 task 通过本轮完成条件后，才允许标记为已完成
- 若命中任务级多 CLI 补充审查门禁，必须先完成 `/trellis:review-gate`
- 当全部代码相关 task 完成且 `PROJECT-AUDIT` 尚未完成时，必须先进入 `/trellis:project-audit`
- `PROJECT-AUDIT` 完成后默认进入 `/trellis:check`

⑨ 上下文健康检查与决策摘要（每轮时间盒结束时）
- 检查当前上下文是否包含矛盾指令、过期信息、幻觉累积
- 若检测到上下文污染迹象，必须开启新会话，重新注入稳定前缀
- 定期导出关键决策为结构化摘要，注入新会话而非依赖完整历史
  - 摘要格式：结构化 JSON/Markdown
  - 包含：决策内容、依据、预期影响

### 5.1.x 反思/批判阶段（新增）

#### 5.1.x.1 实现后质量检查机制

AI 实现后、提交前，必须执行一次实现后质量检查：

```
质量检查清单：
□ 对照 Spec 逐项核对实现完整性
□ 对照测试用例验证功能正确性
□ 边界场景覆盖度检查（空值、极值、异常值）
□ 安全风险检查（注入、越权、泄露）
□ 性能影响评估（计算复杂度、资源占用）
```

输出"检查结果"：
- 改动范围与适用 spec / guideline
- 验证命令的 pass / fail / not run 结果
- 与需求/设计不一致的点、未覆盖风险、建议人工关注模块

#### 5.1.x.2 任务级多 CLI 补充审查门禁

当前 CLI 完成**当前任务的质量检查与本地验证**后，不是默认立即标记任务结束，而是要先判断是否需要进入“任务级多 CLI 补充审查层”。

这一层的定位是：**面向高风险、高不确定、高影响面任务的补充审查门禁**。它不是对 `check` 的重复执行，也不是常规任务的默认重流程。

触发顺序：

```
当前任务实现完成
  → 当前 CLI 完成质量检查（`check` / 本地门禁 / 当前任务验证）
  → `/trellis:review-gate` 判断是否触发任务级多 CLI 审查
  → 若触发：其他 CLI 产出报告 → 当前 CLI 汇总修复 → 当前 CLI 重新验证
  → 任务才允许标记为已完成
```

角色分工：

- **当前 CLI**：协调者 + 唯一修复者
- **其他 CLI**：reviewer，只审查，不直接修改代码
- **用户**：手动在其他 CLI 中执行标准命令；在冲突、越界或超阈值时做最终决策

触发模型：

1. **硬条件（命中即触发）**
   - 认证、授权、权限边界、敏感信息处理
   - 数据迁移、schema 变更、删除与回填
   - 公共 API、跨层 contract、外部系统集成
   - 支付、消息队列、缓存一致性、并发状态
   - 共享核心模块且 blast radius 明显
   - 用户显式要求执行任务级多 CLI 审查

2. **软条件门槛（分层判断）**
   - **复杂度层**：改动文件数、改动行数、涉及模块/层数、异常路径数量
   - **影响面层**：公共模块、跨层边界、外部集成、blast radius
   - **可信度层**：测试覆盖不足、当前 CLI 不确定性高、AI 生成比例高、历史缺陷密度高

门槛判定：

- 任一层达到高门槛 → `required`
- 两层达到中门槛 → `recommended`
- 均未达到门槛 → `skip`

执行约束：

- 默认由用户手动调用 **2 个其他 CLI**；必要时可扩展到 **4 个 reviewer**
- 建议优先在 **3 轮**内收敛；若问题仍未收敛，必须显式提示用户判断是否继续下一轮
- 若当前轮没有新的有效修复建议，可提前关闭
- reviewer 建议互斥、越出任务边界，或超过建议轮次后用户未明确要求继续时，必须停止自动推进并要求人工介入
- 进入该层前，目标 reviewer CLI 必须已具备 `multi-cli-review` 能力；当前 CLI 必须已具备 `multi-cli-review-action` 能力。若缺失，先补齐对应 skill，再进入补充审查
- reviewer 只消费当前 CLI 已创建好的目录结构；**reviewer 不负责创建目录**

交接协议：

- 当前 CLI 负责先创建 `tmp/multi-cli-review/<task-id>/review-round-<N>/`
- 当前 CLI 输出给其他 CLI 的**完整标准命令**
- 其他 CLI 使用 `multi-cli-review` 写入各自的 `{reviewer-id}.md`
- 当前 CLI 使用 `multi-cli-review-action` 统一读取、去重、冲突标记、执行修复
- 修复后，当前 CLI 必须重新跑该任务的质量检查 / 验证，之后才能关闭该任务

推荐输出物：

- `$TASK_DIR/review-gate/review-gate-round-<N>.md`：记录本轮为何触发 `required / recommended / skip`
- `$TASK_DIR/review-gate/reviewer-commands-round-<N>.md`：给其他 CLI 的标准命令模板
- `tmp/multi-cli-review/<task-id>/review-round-<N>/{reviewer-id}.md`：各 reviewer 报告
- `tmp/multi-cli-review/<task-id>/summary-round-<N>.md`：聚合结果
- `tmp/multi-cli-review/<task-id>/action.md`：当前 CLI 决策与执行记录
- `tmp/multi-cli-review/<task-id>/.processed.json`：由 `multi-cli-review-action` 维护的内部消费状态

#### 5.1.x.3 双模态执行原则

根据任务风险级别采用不同执行策略：

| 风险级别 | 任务类型 | 执行策略 |
|---------|---------|---------|
| **低风险** | 样板代码生成、测试用例生成、文档同步 | AI 激进执行 → 快速 Review → 自动门禁 |
| **中风险** | 功能实现、逻辑编写 | AI 主导 → 关键节点人工确认 |
| **高风险** | 核心算法、安全相关、数据迁移 | 人工主导 → AI 辅助 → 完整人工评审 |

#### 5.1.x.4 上下文污染检测升级

检测到污染迹象时的强化处理：
```
① 立即停止当前会话
② 基于最新决策摘要（而非完整历史）开启新会话
③ 标记本次会话异常供后续复盘
```

**时间盒规则**：
- 简单任务（L0，单文件修改）：根据复杂度合理设定
- 标准任务（L1，跨模块）：根据复杂度合理设定
- 复杂任务（L2，架构级）：拆分为多个子时间盒，确保单次可完成

**退出条件（按优先级）**：
① 证据通过（DoD满足）→ 正常退出
② 时间盒耗尽但进度>70% → 延长一个半时间盒，仍未完成进入5.2
③ 同类错误连续2轮未收敛 → 立即进入5.2，不等时间盒结束
④ 以"证据通过"作为退出条件，不使用主观置信度阈值

### 5.2 人工介入机制
- 触发条件1：同类缺陷连续2轮未解决
- 触发条件2：任务耗时超估算50%仍未达到DoD
- 触发条件3：出现安全高危、数据一致性风险、线上阻断风险
- 触发条件4：模型输出不稳定（同输入多次结果差异过大）
- 触发条件5：任务级多 CLI 审查超过建议轮次仍未收敛且用户未明确要求继续，或 reviewer 建议互斥 / 越界

介入动作：
- 先做根因分类（需求问题/设计问题/实现问题/评估问题）
- 调整策略（重写验收、拆任务、换模型、改Prompt、改工具链）
- 产出"介入结论记录"，作为后续同类任务模板

### 5.3 阶段性检查点
- 交付效率指标（建议周维度复盘）：
  - 部署频率
  - 变更前置时间（从提交到上线）
  - 变更失败率
  - 平均恢复时间（MTTR）
  - 返工率（rework）
- AI效率指标：
  - 首轮通过率（一次实现即通过DoD的比例）
  - 评估通过率（评估集通过比例）
  - 单任务Token成本（按被验收任务计）
  - 缓存命中率（`cached_tokens`占比）
  - 降级/回退触发率
- 要求：
  - 任一核心指标连续两周恶化，必须触发流程复盘

### 5.4 人工代码校验（风险分级必选）

- 低风险任务：抽样校验
- 中风险任务：必须人工审查关键逻辑与异常路径
- 高风险任务：必须完整人工审查、威胁建模、回归验证
- 单人项目执行方式：至少间隔30分钟后进行"冷启动复审"，避免同一上下文盲点

> 这一阶段体现了"实现后强制自检"和"优化-评审"的闭环思想。

### 5.5 代码与文档同步机制

#### 5.5.1 同步原则

- **任务粒度同步**：每个任务完成后，必须同步更新相关文档
- **上下文变更同步**：每次上下文有重大变更（如需求变更、架构调整），需同步更新文档
- **即时优先**：文档更新应尽可能在代码变更后立即完成，避免事后遗忘

#### 5.5.2 同步内容清单

| 变更类型 | 必须更新的文档 | 更新时机 |
|---------|--------------|---------|
| 新增功能 | 需求文档、功能规格说明、接口文档 | 功能验收通过后 |
| 代码重构 | 技术架构文档、模块说明 | 重构完成通过测试后 |
| 数据库变更 | 数据库设计文档、ER图、数据字典 | 迁移/变更生效后 |
| 接口变更 | 接口文档、集成说明 | 接口联调完成后 |
| Prompt调整 | 提示词模板库、评估集 | 评估基线回归通过后 |

#### 5.5.3 同步执行规范

```
代码与文档同步流程：

① 触发条件检测
    ├── 代码提交后检查是否有文档依赖变更
    ├── 上下文更新后检查文档时效性
    └── 任务验收通过后触发同步检查

② 文档受影响范围分析
    ├── 识别本次变更影响的文档范围
    ├── 区分必须更新/建议更新/无需更新
    └── 标记文档更新优先级

③ 执行文档同步
    ├── 按受影响范围逐项更新
    ├── 更新内容必须与代码/上下文一致
    └── 添加版本记录和变更说明

④ 同步验证
    ├── 检查文档与代码的一致性
    ├── 验证文档完整性
    └── 确认无遗漏项
```

#### 5.5.4 文档版本管理

- 所有文档采用与代码关联的版本号
- 重大变更需在文档中记录变更日志
- 保留历史版本以便追溯

### 5.6 单次对话任务闭环规范

#### 5.6.1 闭环定义

单次对话上下文内完成一个独立任务，包含完整的工作流程：

```
单次对话任务闭环流程：

① 任务理解与确认
    ├── 明确任务目标与验收标准
    ├── 确认DoR满足条件
    └── 与用户确认理解一致

② 代码实现
    ├── 按任务规格进行代码编写
    ├── 遵循项目编码规范
    └── 确保代码可运行

③ 代码质量检测（按项目自动化检查矩阵执行）
    ├── 静态分析/规范检查（如 ESLint、Ruff、Checkstyle）
    ├── 类型检查/编译校验（如 Type Check、mypy、Build）
    ├── 质量平台/安全扫描（如 SonarQube、依赖漏洞、敏感信息、注入风险）
    └── 单元/集成测试（核心逻辑覆盖）

④ 代码优化
    ├── 性能优化（减少不必要的计算/请求）
    ├── 可读性优化（命名、注释、结构）
    └── 重复代码提取

⑤ Bug修复（如有）
    ├── 根据测试反馈定位问题
    ├── 分析根因
    └── 修复并验证

⑥ 回归验证
    ├── 确保不破坏现有功能
    ├── 运行绑定评估集
    └── 确认满足DoD标准
```

#### 5.6.2 任务拆分原则

- 单次对话任务应在30-90分钟可完成
- 复杂任务需拆分为多个可闭环的子任务
- 跨对话任务必须有明确的状态传递机制

#### 5.6.3 上下文健康管理

- 每次任务开始前检查上下文清洁度
- 任务完成后导出关键结论到外部存储
- 新任务开始时优先从外部存储恢复上下文，而非依赖对话历史

### 6.6 Bug反馈学习机制

每个Bug修复完成后，需要进行反馈学习，避免同类问题再次出现。

#### 6.6.1 问题分类与根因分析

对修复的Bug进行分类和根因分析，形成知识沉淀：

| 字段 | 说明 |
|------|------|
| Bug编号 | BUG-001 |
| 问题分类 | 需求理解/设计缺陷/编码错误/测试遗漏/技术债务/其他 |
| 根因类型 | 遗漏/误解/疏忽/经验不足/流程问题/其他 |
| 根因描述 | 详细描述问题产生的根本原因 |
| 5Why分析 | 逐层追问为什么，直到找到根本原因 |

**示例 - 5Why分析：**
- 问题：用户登录后显示空页面
  - Why 1: 为什么显示空页面？→ 页面数据未加载
  - Why 2: 为什么数据未加载？→ API返回空数据
  - Why 3: 为什么API返回空数据？→ 数据库查询条件错误
  - Why 4: 为什么查询条件错误？→ 代码中使用了错误的字段名
  - Why 5: 为什么使用错误字段名？→ 需求文档和数据库设计文档字段名不一致，未核对
- 根本原因：需求文档和数据库设计文档字段名不一致，开发时未进行核对

- 单次偶发环境问题，且无稳定复现路径
- 纯个人偏好，不具备复用价值
- 已有同类记录，但本次没有新增证据或新结论

  - 尾款未到账时，不交付源码仓库权限、生产环境密钥、管理员账号、最终部署控制权
- **试运行授权轨**
  - 尾款未到账时，只交付已明确披露限制条件的试运行版本
  - 交付材料必须写清：授权有效期、续期方式、到期行为、永久授权触发条件
  - 到期行为不得删改数据，不得造成不可恢复损失

#### 7.2.2 尾款到账后的最终移交

仅在开发者明确确认尾款到账后，才执行最终移交：

- 源码仓库或源码包
- 永久授权文件或移除试运行限制的版本
- 构建脚本、部署脚本、CI/CD 配置
- 生产环境变量、密钥、证书、第三方平台配置
- 服务器、域名、数据库、对象存储等管理员权限
- 最终运维文档、回滚说明、交接记录

建议把以上移交项做成 checklist，由交付双方逐项签收。

进入正式交付前，建议先运行：

```bash
python3 <WORKFLOW_DIR>/commands/shell/delivery-control-validate.py --phase delivery --task-dir <task-dir>
python3 <WORKFLOW_DIR>/commands/shell/ownership-proof-validate.py --phase delivery --task-dir <task-dir>
```

前一个命令用于检查 `assessment.md`、`task_plan.md` 与 `delivery/` 交付事件文档中的双轨交付控制信息是否一致；后一个命令用于检查源码水印与归属证明产物是否齐全。任一验证失败，都不应进入正式交付。

- 形成团队知识库

#### 7.3.1 工作流经验反馈优化机制

为避免"项目里发现了流程问题，但工作流本身没有被改进"，本工作流要求把流程层经验通过 AI 起草、人工确认的方式沉淀到：

```text
docs/workflows/新项目开发工作流/learn/
```

**整体流程**：

```text
开发中发现问题
  → AI 在 tmp/ 目录下起草反馈文档
  → 提示用户将文件移动到 learn/ 目录
  → 人工在 learn/ 目录中决定是否回流到工作流文档
  → 确认后修改工作流总纲/命令/模板
  → 清理 tmp/ 中的已处理文件
```

##### AI 起草规则

当用户在开发过程中表达流程问题信号时（如"这个流程有坑""这一步老容易漏""这个命令说明看完还是会错""先把这次踩坑记一下"），AI 应在当前项目的 `tmp/` 目录下生成一份经验反馈草稿。

**文件命名约定**：

```text
tmp/workflow-feedback-YYYY-MM-DD-短名.md
```

示例：
- `tmp/workflow-feedback-2026-04-01-record-session-closure.md`
- `tmp/workflow-feedback-2026-04-02-phase-router-misroute.md`

**文件头部状态标记**：

```markdown
<!-- 状态: 待移动到 learn/ -->
```

用户将文件移动到 `learn/` 目录后，应将状态标记改为：

```markdown
<!-- 状态: 已移动到 learn/，待人工决策 -->
```

**最小要素要求**：

AI 起草时至少引导用户提供以下 4 个要素（可以一句话写在一起）：

| 要素 | 说明 | 示例 |
|------|------|------|
| **现象** | 发生了什么（具体行为，不是感受） | "plan 阶段不知道该不该先跑 test-first" |
| **阶段** | 在哪个阶段/命令操作中遇到 | "plan 阶段，命令映射里没有写清依赖关系" |
| **影响** | 导致了什么后果 | "来回试了两次，浪费 20 分钟" |
| **初步判断** | 是流程问题、文档问题还是命令问题 | "命令映射文档缺依赖关系说明" |

**追问逻辑**：

- 如果用户只说了一句模糊的话（如"这个流程有问题"），AI 应追问上述 4 个要素中缺失的部分，补齐后再写入文件
- 如果 4 个要素都已具备，AI 可直接生成完整草稿，无需追问
- 追问不超过一轮，避免打断开发节奏

**草稿内容结构**：

AI 生成的草稿应基于 `learn/TEMPLATE.md`。根据场景选用两种格式：

**快速捕获**（开发中踩坑，30 秒内记完）：

```markdown
<!-- 状态: 待移动到 learn/ -->

# <一句话描述>

踩坑位置：<阶段/命令>
现象：<一句话>
可能原因：<一句话或"待分析">

<!-- 复盘时再展开为完整格式 -->
```

**标准起草**（AI 辅助用户完整记录时使用）：

```markdown
<!-- 状态: 待移动到 learn/ -->

# 经验反馈记录：<主题>

## 现象
- 发生了什么：<用户描述的具体行为>
- 发生在哪个阶段/命令：<阶段名称或命令>
- 影响：<导致的后果，如返工/误判/漏检/卡住>
- 初步判断：<流程问题 / 文档问题 / 命令问题>

## 详细描述（可选）
<按真实开发顺序补充：当时发生了什么 → 为什么卡住 → 怎么发现并修正的>

## 建议改哪里
<具体的工作流修改建议，如"在命令映射中补充 plan 的依赖说明">

---
<!-- 以下由人工复盘时填写 -->
## 人工结论
- [ ] 改工作流 → 修改了哪些文件：
- [ ] 不改 → 原因：
```

##### 用户交接规则

AI 生成反馈文档后，必须明确提示用户：

```text
已将工作流反馈草稿写入：tmp/workflow-feedback-YYYY-MM-DD-短名.md

如果确认需要记录，请将此文件移动到：
  docs/workflows/新项目开发工作流/learn/

移动后，在复盘时决定是否回流到工作流文档。
```

##### 清理规则

- 用户将文件移动到 `learn/` 目录后，AI 应删除 `tmp/` 中的原文件
- 若用户在 delivery 阶段确认不需要保留某条反馈，AI 应直接删除 `tmp/` 中的对应文件
- `tmp/` 目录下不应积累超过 5 份未处理的反馈文件；超过时 AI 应主动提醒用户处理

##### 触发方式

- **自然触发**：开发中直接出现"这个流程有坑""这一步老容易漏""先把这次踩坑记一下"这类反馈时，就应触发经验反馈记录；不必等到项目结束
- **命令触发**：
  - `/trellis:start`：当用户表达"想记录这次流程问题/复盘这次踩坑/后面要优化这里"时，Phase Router 应识别为经验反馈信号
  - `/trellis:delivery`：项目收尾时，在 Step 9 统一做显式复盘与沉淀

##### 实践要求

- 记录应该写成"真实开发里哪里卡住了"，而不是形式化周报
- 优先记录导致返工、误判、漏检、来回切换上下文的点
- 如果只是一次性失误，先记事实，不急着升级成工作流改动

##### 边界要求

- AI 可以主动读取 `learn/` 目录、总结候选项、提出优化草案
- AI 不应仅凭单条经验记录自动把临时结论升级为正式工作流规范
- 若多个 learn 记录结论互相冲突，必须进入人工决策，不得自动合并
- AI 起草的反馈文件只作为人工决策的输入，不自动生效

### 7.4 会话记录与元数据提交

- 代码、文档或配置变更已由人工测试并提交后，再执行会话记录
- 这里只有**当前执行任务已完成后的最终收尾记录**，才允许进入 `record-session`
- 收尾顺序：先 `record-session`（需要当前任务上下文），再 `archive`（会清除 `.current-task`）；未完成任务不得因记录会话而误归档；**非当前任务**不得借本轮收尾顺手自动提交
- `archive` 后，`.trellis/tasks` 与 `.trellis/.current-task` 应已 clean
- 进入 `record-session` helper 前，staged 区未混入非目标变更；否则应中断
- `record-session` 应通过工作流 helper 执行，不应直接把 `add_session.py` 作为最终收尾入口
- 不允许仅依据"task archived"或"session added successfully"字样就判定完成，必须以 git 状态为准

收尾校验：

```bash
git status --short .trellis/tasks .trellis/.current-task
git status --short .trellis/workspace .trellis/tasks
```

- 预期输出为空；若仍有 `.trellis/tasks`、`.trellis/workspace` 或 `.trellis/.current-task` 变更，说明归档/记录未真正闭环
- 若自动提交失败，应视为 `record-session` 未完成，先处理 git 写入失败原因，再结束本轮工作
- 若目标不是当前任务，或 staged 区存在非目标变更，应视为流程前置条件不满足，先清理边界问题，再继续收尾

> 这一阶段体现了"错误沉淀与复用"的方法论价值。

## 🧪 附录一：AI评估驱动开发 (Evaluation-Driven Development)

在项目开始前，应建立AI评估体系作为基础。

### 附录1.1 评估指标定义
- **准确率/通过率**：AI生成代码通过测试的比例
- **处理时间**：系统级 + 组件级响应时间
- **系统性能**：吞吐量、并发能力
- **幻觉率**：AI产生错误输出的频率
- **上下文利用效率**：上下文窗口的有效利用率

### 附录1.2 测试用例设计

- **快乐路径用例**：核心功能场景
- **边界用例**：AI易失败的场景
- **对抗用例**：提示词注入、恶意输入
- 使用LLM生成扩展测试用例

**评估集规模建议（参考 OpenAI Evals 和 Anthropic 内部实践）**：
- 最小可信评估集：≥50条（低于此数量，统计波动过大，结果不可信）
- 生产级评估集：200~500条（核心路径覆盖率≥80%）
- 对抗用例比例：≥15%（否则鲁棒性测试形同虚设）
- 评估集增长触发：每次线上故障、每次重大Prompt变更，至少沉淀5~10条新用例

**评估结果解读门槛**：
- 单次评估波动在±3%以内属正常，不应触发回滚
- 连续2次下降且累计超过5% → 触发回归阻断

### 附录1.3 评估方法选择
- **确定性检查**：结构、路径、工具调用正确性
- **语义检查**：使用LLM-as-Judge进行语义等价判断
- **混合策略**：轻量检查 + LLM评估

### 附录1.4 评估自动化
- 建立持续评估pipeline
- 配置评估频率（每次commit/每日/每周）
- 设定回归检测阈值

### 附录1.5 持续评估门禁
- PR级快速评估：每次PR运行最小评估集（核心路径 + 最近高频缺陷）
- 夜间全量评估：每日运行完整评估集，跟踪趋势变化
- 回归阻断：关键指标低于阈值时阻断发布
- 评估集演进：每次线上故障与高优Bug必须沉淀为新评估用例

> 评估体系是AI Workflow的核心基础，没有评估就没有改进

## 🔒 附录二：AI代码与AI应用安全审查（OWASP LLM Top 10 + Agentic Applications 2026 映射）

### 审查重点：OWASP LLM Top 10

- Prompt Injection（提示词注入）
- Sensitive Information Disclosure（敏感信息泄露）
- Supply Chain（供应链风险）
- Data/Model Poisoning（数据与模型投毒）
- Improper Output Handling（输出处理不当）
- Excessive Agency（过度代理权限）
- System Prompt Leakage（系统提示词泄露）
- Vector/Embedding Weaknesses（向量与嵌入弱点）
- Misinformation（错误信息）
- Unbounded Consumption（无界资源消耗）

### 审查重点：OWASP Top 10 for Agentic Applications 2025/2026（新增）

| 风险项 | 描述 | 防护建议 |
|-------|------|---------|
| **Tool Misuse** | 工具被错误调用或超出预期范围使用 | 工具入参必须白名单校验，权限最小化 |
| **Identity & Memory Management** | Agent 身份混淆、内存/上下文信息泄露 | 严格隔离不同任务上下文，会话状态加密 |
| **Autonomous Decision Errors** | Agent 过度自主决策导致错误结果 | 关键决策必须人工确认，输出必须可追溯 |
| **Goal Hierarchy Failure** | Agent 目标层级混乱，优先级误判 | 明确目标优先级，建立冲突仲裁机制 |
| **Unbounded Execution** | Agent 执行无外部边界限制 | 设置执行预算上限，超限强制中断 |

### agentic 工作流防护（新增）

1. **自主操作限制**：
   - 所有涉及外部系统写操作、敏感数据访问、费用产生等高影响行为，必须插入人工审批节点
   - Agent 不得在无确认情况下执行不可逆操作

2. **人类最终责任**：
   - 明确宣示：AI 生成内容的所有最终责任归人类决策者
   - 关键输出必须经过人类审核后方可生效
   - 建立"人类在环"（Human-in-the-Loop）机制作为强制门禁

3. **执行边界约束**：
   - 单次任务最大执行轮次设定上限
   - 累计资源消耗（Token、时间、费用）设定告警阈值
   - 连续失败自动触发人工介入流程

4. **审计追踪**：
   - 所有AI的工具调用（尤其是写操作）必须留有可追溯日志
   - 日志必须包含：输入摘要、调用时间、模型版本、输出结果、执行用户

**最低安全要求（可落地版）**：

1. **Prompt Injection防护**：
   - 用户输入必须与系统指令在结构上隔离（使用角色分隔或XML标签，不拼接到system prompt）
   - 禁止将外部内容（如用户上传文件、爬取内容）直接注入高权限上下文

2. **输出处理**：
   - AI输出不得未经验证直接执行（eval、exec、shell注入等）
   - 结构化输出必须经过Schema验证，验证失败必须拒绝而非降级处理

3. **敏感信息防护**：
   - 系统Prompt不得包含可被提取的密钥、内部路径或用户PII
   - 生产环境API Key必须通过环境变量注入，不得硬编码在Prompt模板中

4. **访问控制**：
   - 工具调用的权限范围必须最小化（只读工具不赋写权限）
   - Agent自主操作外部系统前，必须有人工审批节点或可撤销确认机制

## 📑 附录三：意图文档维护机制（新增）

### 意图文档定义

项目级 Intent.md 是高层次意图声明，包含：
- 项目业务目标与核心价值
- 关键设计决策及理由
- 重要约束与边界条件
- 已确认的验收证据基线

### 维护流程

```
意图文档生命周期：

① 项目启动
    └── 基于确认的 BRD 生成初始 Intent.md

② 每次 AI 迭代后
    └── AI 自动同步更新以下内容：
        - 本次迭代产生的新的设计决策
        - 需求理解的澄清或修正
        - 验收标准的补充或调整
        - 与原意图的偏差说明（如有）

③ 人类复核
    └── 开发者在下一会话开始前复核更新
    └── 确认签字后生效

④ 长期项目维护
    └── 每季度或每重要里程碑后全面复核
    └── 清理过时假设，确保意图一致性
```

### Intent.md 模板

```markdown
# Intent.md - [项目名称]

## 业务目标
- 核心价值主张
- 关键成功指标

## 设计意图
- 架构决策及理由
- 技术选型约束
- 已排除的替代方案

## 验收证据基线
- 已确认的核心测试用例
- 性能/质量基线
- 关键风险及应对

## 意图变更日志
| 日期 | 变更内容 | 变更理由 | 确认人 |
|------|---------|---------|-------|
```
