# 需求变更管理执行卡

> 本文档是 `§2.5 需求变更管理` 的**可执行载体**。
> 所有"进入 §2.5"的跳转点都应链接到本文档，而不是各自描述变更流程。

## 适用场景

`§2.4 需求冻结` 之后，`§3` 到 `§7` 任一阶段出现以下情况时触发：

- 新增需求
- 修改需求（改变范围、接口契约、验收标准、成本、工期）
- 删除需求

**不触发本流程的情况**：纯澄清（不改变范围、接口契约、验收标准、成本、工期），留在当前阶段即可。

## 4 步轻量闭环

### Step 1: 变更识别

**目标**：确认这是变更还是纯澄清。

| 判断项 | 纯澄清 | 需要走变更流程 |
|--------|--------|--------------|
| 范围不变 | 是 | 否 |
| 接口契约不变 | 是 | 否 |
| 验收标准不变 | 是 | 否 |
| 成本不变 | 是 | 否 |
| 工期不变 | 是 | 否 |

任一判断项为"否"，即进入变更流程。

### Step 2: 影响评估

**目标**：量化变更影响，为用户提供决策依据。

评估维度：

- **已完成工作**：哪些已完成任务需要返工或调整
- **交付边界**：对当前交付物范围的影响
- **成本与工期**：变更所需的额外工作量和时间
- **测试范围**：受影响的测试用例和验证项
- **双 PRD 同步**：`customer-facing-prd.md` 与 `developer-facing-prd.md` 需要更新的内容

输出：在当前任务目录下形成变更评估记录，明确写出"变更内容 / 影响范围 / 建议方案 / 预估成本"。

### Step 3: 用户确认

**目标**：由用户决定是否批准变更。

- 展示变更评估结果
- 等待用户明确确认（批准 / 拒绝 / 需要调整）
- **未确认前不并入当前阶段**

### Step 4: 回到受影响最早阶段

**变更获批后**：

1. 同步更新 `docs/requirements/customer-facing-prd.md` 和 `docs/requirements/developer-facing-prd.md`
2. 回到受影响的最早阶段（brainstorm / design / plan / start）
3. 对该阶段的基线进行更新后再继续推进

补充：

- 若变更发生在 `project-audit` 或 `review-gate` 阶段，且仅影响实现细节或修复策略，通常回到 `start`
- 若变更已影响设计基线、接口契约或需求范围，则回到 `design` 或 `brainstorm`

**变更未获批**：

- 维持冻结基线
- 不把变更混入当前交付
- 记录变更请求和拒绝原因，供后续参考

## 典型变更口语示例

- "这个功能再加一个"
- "这个页面流程改一下"
- "这个接口字段要调整"
- "这个验收标准我想改"
- "这个功能先不做了"

## 典型纯澄清口语示例

- "这里我确认一下是什么意思"
- "这个描述对应的是不是原来那个流程"
- "我不是加功能，只是想确认理解"
