# 工作流嵌入执行规范

> 当前 workflow 在目标项目中的唯一嵌入执行协议。
> 适用 CLI：Claude Code / OpenCode / Codex。
> 最后更新: 2026-04-20

---

## 文档定位

本文件只负责三件事：

1. 定义如何根据**本文件绝对路径**推导 workflow 根目录与执行脚本路径
2. 定义何时允许执行首次嵌入
3. 定义如何判定当前目标项目是否已经完整有效嵌入

本文件不负责：

- 自动修复、自动补装、自动重置、自动清理
- 兼容升级迁移方案
- 平台背景介绍或长篇平台教程

若本文件与同目录其他文档口径冲突，以本文件为准。

---

## 硬规则

- 只有**纯净初始态**目标项目才允许执行首次嵌入。
- 只要检测到当前 workflow 的任何历史嵌入痕迹，无论其状态是成功、失败、残缺还是不明，都必须停止后续嵌入操作。
- 本协议禁止在异常状态上继续运行 `install-workflow.py` 试图覆盖修好。
- 本协议禁止使用手工复制 `commands/`、`skills/`、`agents/` 作为嵌入替代方案。
- 本协议禁止调用 `upgrade-compat.py --merge`、`upgrade-compat.py --force`、`uninstall-workflow.py` 作为自动修复步骤。
- 如果嵌入过程中任一步失败，目标项目应被视为**非初始态**；后续必须由用户手动处理后，再从初始态重新执行完整嵌入。

## 路径推导

在目标项目中使用本 workflow 时，应先拿到本文件的绝对路径。

约定：

- `WORKFLOW_SPEC_ABS_PATH` = 本文件绝对路径
- `WORKFLOW_ROOT_DIR` = `dirname(WORKFLOW_SPEC_ABS_PATH)`
- `TARGET_PROJECT_ROOT` = 目标项目根目录

由此推导：

- 状态检测脚本：`<WORKFLOW_ROOT_DIR>/commands/detect-embed-state.py`
- 安装脚本：`<WORKFLOW_ROOT_DIR>/commands/install-workflow.py`
- 校验脚本：`<WORKFLOW_ROOT_DIR>/commands/upgrade-compat.py`

不要依赖当前 shell 工作目录来猜测这些路径。

## 状态模型

本协议只允许以下三种状态：

### `INITIAL_BASELINE_READY`

含义：目标项目处于允许首次嵌入的纯净 Trellis 初始态。

只有此状态允许继续执行嵌入。

### `ALREADY_VALID_EMBEDDED`

含义：目标项目已经完整且有效嵌入当前 workflow。

必须停止，不得重复嵌入。

### `BLOCKED_NON_INITIAL_STATE`

含义：目标项目不是允许首次嵌入的初始态。

这包括但不限于：

- 历史安装成功
- 历史安装失败
- 半嵌入
- 残留 patch / skills / helper / install record / attempt record
- 存在当前 workflow 的托管痕迹但无法证明其完整有效

必须停止，并提示用户手动处理。

---

## 初始态判定

`INITIAL_BASELINE_READY` 必须同时满足：

- 目标项目是 Git 仓库
- `origin` 至少配置 2 个 push URL
- 已执行 `trellis init`
- 存在 `.trellis/.version`
- 若是新建仓库且尚无本地提交历史，则当前本地分支必须是 `main`
- 不存在 `.trellis/workflow-installed.json`
- 不存在 `.trellis/workflow-embed-attempt.json`
- 不存在当前 workflow 的托管痕迹，例如：
  - `AGENTS.md` 中的 `workflow-nl-routing` 区段
  - `.trellis/workflow.md` 中的 workflow patch
  - 当前 workflow 新增阶段命令 / skills
  - 当前 workflow 注入的 `start` / `finish-work` / `record-session` patch
  - 当前 workflow 分发的 helper scripts
  - 当前 workflow 托管的 managed agents

只要以上任一条件不成立，就不能视为初始态。

## 嵌入尝试记录

正式安装开始后，安装器会先写入：

```text
.trellis/workflow-embed-attempt.json
```

用途：

- 标记“当前项目已经发生过 workflow 嵌入尝试”
- 若安装中途失败，为后续状态检测提供硬证据

只要该文件存在，状态就不能判为 `INITIAL_BASELINE_READY`。

记录结构：

```json
{
  "status": "in_progress | failed",
  "workflow_version": "string",
  "workflow_schema_version": "string",
  "workflow_spec_path": "string",
  "workflow_root": "string",
  "target_project_root": "string",
  "started_at": "ISO 8601 timestamp",
  "updated_at": "ISO 8601 timestamp",
  "cli_types": ["claude", "opencode", "codex"],
  "last_step": "string",
  "failed_at": "ISO 8601 timestamp, optional when status=failed",
  "error": "string, optional when status=failed"
}
```

说明：

- `status` 由安装器正式写入时只允许为 `in_progress` 或 `failed`
- 只读诊断脚本在读取损坏或不可解析的 attempt 记录时，可能临时合成 `unknown` 状态用于诊断输出；这不是安装器的正式写入值
- `last_step` 用于指出安装器最后完成或尝试执行到的阶段
- `error` 仅用于帮助用户理解失败原因，不表示系统会自动修复

成功闭环条件：

1. `install-workflow.py` 完成目标写入
2. 安装器自身触发一次 `upgrade-compat.py --check`
3. 自检通过后，删除 `.trellis/workflow-embed-attempt.json`
4. 当前项目才可被视为已完成成功嵌入闭环

补充：

- 安装器在自己的装后自检阶段会临时设置 `WORKFLOW_IGNORE_EMBED_ATTEMPT=1`，仅用于让 `upgrade-compat.py --check` 在 attempt 文件尚未清理时继续执行完整性校验
- 这是安装器与校验器之间的内部合同，不是给用户手动设置的操作入口

## 允许执行的命令

本协议只允许以下命令进入嵌入主链：

```bash
/ops/softwares/python/bin/python3 <WORKFLOW_ROOT_DIR>/commands/detect-embed-state.py --project-root <TARGET_PROJECT_ROOT>
/ops/softwares/python/bin/python3 <WORKFLOW_ROOT_DIR>/commands/install-workflow.py --project-root <TARGET_PROJECT_ROOT> --dry-run
/ops/softwares/python/bin/python3 <WORKFLOW_ROOT_DIR>/commands/install-workflow.py --project-root <TARGET_PROJECT_ROOT>
/ops/softwares/python/bin/python3 <WORKFLOW_ROOT_DIR>/commands/upgrade-compat.py --project-root <TARGET_PROJECT_ROOT> --check
```

禁止在本协议主链中使用：

```bash
upgrade-compat.py --merge
upgrade-compat.py --force
uninstall-workflow.py
```

## 标准执行顺序

### Step 1：先判定状态

在目标项目中先执行：

```bash
/ops/softwares/python/bin/python3 <WORKFLOW_ROOT_DIR>/commands/detect-embed-state.py \
  --project-root <TARGET_PROJECT_ROOT>
```

若返回：

- `INITIAL_BASELINE_READY`：允许继续
- `ALREADY_VALID_EMBEDDED`：停止
- `BLOCKED_NON_INITIAL_STATE`：停止

### Step 2：执行安装预演

仅当状态是 `INITIAL_BASELINE_READY` 时，允许执行：

```bash
/ops/softwares/python/bin/python3 <WORKFLOW_ROOT_DIR>/commands/install-workflow.py \
  --project-root <TARGET_PROJECT_ROOT> \
  --dry-run
```

### Step 3：执行正式安装

仅当预演通过且目标项目仍处于初始态时，允许执行：

```bash
/ops/softwares/python/bin/python3 <WORKFLOW_ROOT_DIR>/commands/install-workflow.py \
  --project-root <TARGET_PROJECT_ROOT>
```

### Step 4：确认完整有效嵌入

正式安装完成后，必须再确认：

```bash
/ops/softwares/python/bin/python3 <WORKFLOW_ROOT_DIR>/commands/upgrade-compat.py \
  --project-root <TARGET_PROJECT_ROOT> \
  --check
```

若该检查未通过，则不得将目标项目视为已完整有效嵌入。

## 完整有效嵌入判定

要判定为 `ALREADY_VALID_EMBEDDED`，至少需要同时满足：

- `detect-embed-state.py` 不再返回初始态
- `.trellis/workflow-installed.json` 存在
- `.trellis/workflow-embed-attempt.json` 不存在
- `.trellis/workflow.md` patch 存在
- `AGENTS.md` 中 `workflow-nl-routing` 区段存在
- `.trellis/scripts/workflow/*.py` helper scripts 已分发
- `pack.requirements-discovery-foundation` 的导入结果可见
- Claude / OpenCode / Codex 各自的 workflow 托管资产已按原生承载方式落盘
- `upgrade-compat.py --check` 通过

补充：

- Claude Code / OpenCode 的用户入口仍是项目命令目录
- Codex 的用户入口仍是 `AGENTS.md + hooks + skills + subagents`
- Codex 不应被当作支持项目级 `/trellis:xxx` 命令目录的 CLI

## 阻断口径

只要命中以下任一情况，必须立即停止后续嵌入：

- `detect-embed-state.py` 返回 `ALREADY_VALID_EMBEDDED`
- `detect-embed-state.py` 返回 `BLOCKED_NON_INITIAL_STATE`
- `install-workflow.py` 任一步失败
- 安装器留下 `.trellis/workflow-embed-attempt.json`
- `upgrade-compat.py --check` 未通过

统一结论口径：

> 当前项目不是允许首次嵌入的初始态。  
> 本协议禁止继续执行嵌入。  
> 请用户手动处理后，再从初始态重新执行完整嵌入。

## 联动更新要求

本文件属于当前 workflow 的嵌入协议单一事实源。

以后每次修改以下任一内容时，都必须先判断本文件是否需要同步修改：

- 嵌入前置条件
- 初始态判定
- 非初始态阻断规则
- 完整/有效嵌入判定
- `workflow-embed-attempt.json` 结构或语义
- `workflow-installed.json` 的嵌入闭环语义
- `install-workflow.py`
- `upgrade-compat.py --check`
- `workflow_assets.py`
- Claude Code / OpenCode / Codex 的原生入口承载方式

联动评估结论只能是两种之一：

- 已同步更新 `工作流嵌入执行规范.md`
- 已检查，本次变更不影响 `工作流嵌入执行规范.md`
