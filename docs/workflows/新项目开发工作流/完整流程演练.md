# 双轨交付控制完整流程演练

> 目标：给出一条可直接照做的 walkthrough，演示外部项目在 `trial_authorization` 轨道下，如何从可行性评估一路走到交付文档与移交 checklist。

---

## 适用范围

本演练适用于以下场景：

- 外包、定制开发、新客户项目
- 需要先试运行，再在尾款到账后完成正式授权和控制权移交
- 需要验证 `delivery-control`、`authorization-management`、`transfer-checklist` 这组三件套是否能贯穿 workflow

本演练**聚焦双轨交付控制主链**：

`feasibility -> brainstorm -> design -> plan -> delivery`

本演练**不覆盖**：

- 真实业务代码实现
- `/trellis:start` 的编码阶段
- `/trellis:self-review`、`/trellis:check`、`/trellis:finish-work` 的代码审查链路

从 `brainstorm` 完成并冻结 PRD 开始，本演练默认还遵循一条跨阶段分流规则：

- 后续任一阶段若出现**新增 / 修改 / 删除需求**，先进入 `§2.5 需求变更管理`
- 若只是**纯澄清**，且不改变范围、接口契约、验收标准、成本、工期，则留在当前阶段处理
- 下方样例文档会一并体现这条规则

## 前置条件

1. 工作流命令已嵌入项目。
2. 目标项目可使用 `trellis-library` CLI 装配资产。
3. 你已经明确这是一个**外部项目**，而不是内部项目。

## 双轨速查

| 项目 | `hosted_deployment` | `trial_authorization` |
|---|---|---|
| feasibility 必填字段 | `delivery_control_track` `delivery_control_handover_trigger` `delivery_control_retained_scope` | 同左，且补 `trial_authorization_terms.*` |
| design 必选资产 | `delivery-control` + `transfer-checklist` | `delivery-control` + `transfer-checklist` + `authorization-management` |
| 条件资产 | 正式移交涉及密钥/配置时，再加 `secrets-and-config` | 同左 |
| delivery 事件 | 可存在 retained-control delivery | 可存在 retained-control delivery，也可在触发条件满足后转 final control transfer |

在双轨交付控制场景下，PRD 至少要把这些点冻结：

- 核心业务功能
- 试运行授权状态如何展示
- 到期后只读与导出行为
- 尾款后正式授权和最终移交边界

同时建议在 PRD 中显式写明冻结后规则：

- 纯澄清留在当前阶段处理
- 新增 / 修改 / 删除需求统一走 `§2.5 需求变更管理`
- 变更获批后，再回到受影响的最早阶段更新基线

对客户说明时，可直接使用类似话术：

> 这版文档确认的是本轮交付范围和验收标准。后续如果你有新的想法，我会先帮你判断是“澄清”还是“正式变更”。澄清我会直接处理；如果是新增、修改或删除，我会先评估对时间和费用的影响，你确认后我再并入执行，这样不会影响当前已确认的交付。

假设客户在看设计稿时说：

> “总部看板里我还想多加一个按整改状态筛选。”

建议最小处理动作：

2. 判断它是否改变当前冻结范围

4. 使用轻量变更单记录，而不是直接改设计稿继续推进

## 阶段 5：Delivery

对于双轨交付控制，`delivery` 阶段不再只是假设“最终一次性交付”，而是按**交付事件**组织文档。

本次演练的交付目录：

- [delivery/test-report.md](../../../.trellis/tasks/03-30-dual-track-e2e-simulation/delivery/test-report.md)
- [delivery/acceptance.md](../../../.trellis/tasks/03-30-dual-track-e2e-simulation/delivery/acceptance.md)
- [delivery/deliverables.md](../../../.trellis/tasks/03-30-dual-track-e2e-simulation/delivery/deliverables.md)
- [delivery/transfer-checklist.md](../../../.trellis/tasks/03-30-dual-track-e2e-simulation/delivery/transfer-checklist.md)
- [delivery/retrospective.md](../../../.trellis/tasks/03-30-dual-track-e2e-simulation/delivery/retrospective.md)

### 这一步要验证的重点

- 当前交付事件是 retained-control delivery，还是 final control transfer
- 若是 final control transfer，是否已满足 `final_payment_received`
- 若走 `trial_authorization`，`trial_authorization_terms` 是否已附档
- 到期只读与导出是否已经验证
- 验收阶段若客户提出新增 / 修改 / 删除需求，是否已先进入 `§2.5 需求变更管理`，而不是直接混入当前交付

### 极简演示：验收阶段客户临时提改动

假设客户在验收时说：

> “这轮先过，但导出页我还想多一个门店对比列。”

建议最小处理动作：

1. 先确认这不是对原验收标准的纯澄清
2. 由于它会改变导出内容与验收范围，判定为**正式变更**
3. 明确说明“不混入当前验收通过版本”
4. 记录到轻量变更单，评估对时间和费用的影响
5. 客户确认后，作为下一轮或补充变更执行

如果客户说的是：

> “我只是确认一下，只读模式还能不能导出历史数据？”

则应按纯澄清处理，可参考：

- [clarification-example.md](../../../.trellis/tasks/archive/2026-03/03-30-dual-track-e2e-simulation/clarification-example.md)

## 关键结论

这条演练链路验证了四件事：

1. 新的 `delivery_control_track` / `trial_authorization_terms` 契约已经能从 feasibility 贯穿到 delivery。
2. `authorization-management` 已从“外部项目无条件必选”收敛为“试运行授权轨条件必选”。
3. `transfer-checklist` 已从“只服务最终一次性交接”收敛为“服务每次正式交付事件”。
4. 从 PRD 冻结开始，后续阶段对“纯澄清”和“正式变更”可以采用不同出口，避免把冻结后的新增 / 修改 / 删除重新混入当前阶段。
