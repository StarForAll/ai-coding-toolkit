# 双轨交付控制完整流程演练

> 目标：给出一条可直接照做的 walkthrough，演示外部项目在 `trial_authorization` 轨道下，如何从可行性评估一路走到交付文档与移交 checklist。

> 首次阅读请先看：[多CLI通用新项目完整流程演练](./多CLI通用新项目完整流程演练.md)。本文保留为外部项目双轨交付控制的专项案例，不承担通用新项目主链入口职责。

---

## 适用范围

本演练适用于以下场景：

- 外包、定制开发、新客户项目
- 需要先试运行，再在尾款到账后完成正式授权和控制权移交
- 需要验证 `delivery-control`、`authorization-management`、`transfer-checklist` 这组三件套是否能贯穿 workflow

本演练**聚焦双轨交付控制主链**：

`feasibility -> brainstorm -> design -> plan -> delivery`

本演练**不覆盖**：

- 真实业务代码实现
- `/trellis:start` 的编码阶段
- `/trellis:check`、`/trellis:review-gate`、`/trellis:finish-work` 的代码审查链路

从 `brainstorm` 完成并冻结 PRD 开始，本演练默认还遵循一条跨阶段分流规则：

- 后续任一阶段若出现**新增 / 修改 / 删除需求**，先进入 [需求变更管理执行卡](./需求变更管理执行卡.md)
- 若只是**纯澄清**，且不改变范围、接口契约、验收标准、成本、工期，则留在当前阶段处理
- 下方样例文档会一并体现这条规则

## 双轨速查

| 项目 | `hosted_deployment` | `trial_authorization` |
|---|---|---|
| feasibility 必填字段 | `delivery_control_track` `delivery_control_handover_trigger` `delivery_control_retained_scope` | 同左，且补 `trial_authorization_terms.*` |
| design 必选资产 | `delivery-control` + `transfer-checklist` | `delivery-control` + `transfer-checklist` + `authorization-management` |
| 条件资产 | 正式移交涉及密钥/配置时，再加 `secrets-and-config` | 同左 |
| delivery 事件 | 可存在 retained-control delivery | 可存在 retained-control delivery，也可在触发条件满足后转 final control transfer |

### 4.1 在 `§3.7` 同步适配收尾命令

在这条演练链路里，`design` 阶段不是只补 `.trellis/spec/`，还要把收尾命令的项目化基线一起定下来。

这里默认你已经按 `工作流总纲 §3.7` 的前 3 个动作完成了：先从 `trellis-library` 导入适配当前项目的 spec，再对当前项目 `.trellis/spec/` 做项目化完善，再写清自动化检查矩阵。下面只展开演练其中与收尾命令直接相关的后半段动作，不重新替代 `§3.7` 的主定义。

**`/trellis:finish-work` 在这里一次定准：**

- 根据已经确认的技术栈、部署方式和自动化检查矩阵，写清真实要跑的 lint / test / build / scan / delivery gate
- 明确哪些变更必须同步更新 spec、checklist、delivery 文档
- 不继续保留通用模板里的空泛说法

**`/trellis:record-session` 在这里先定基线：**

- 明确最终记录入口是否要求 workflow helper
- 明确收尾顺序：先 `record-session`（需要当前任务上下文），再 `archive`（会清除 `.current-task`）
- 明确 `.trellis/tasks`、`.trellis/.current-task`、staged 区的门禁
- 明确本项目 session 至少要记录哪些设计/计划/交付决策

这一步的重点是“先把项目基线定下来”，不是增加复杂机制。

边界：

- 这里定的是**当前项目后续收尾**的项目化基线，不要求回填历史任务、旧 session 或已归档目录
- 若历史记录与新基线不一致，应单独评估是否需要变更，不在当前 walkthrough 里顺手修账

### 5. 极简演示：设计阶段客户临时提改动

假设客户在看设计稿时说：

> “总部看板里我还想多加一个按整改状态筛选。”

建议最小处理动作：

1. 先接住诉求，不直接拒绝
2. 判断它是否改变当前冻结范围
3. 由于它会影响筛选能力与验收口径，判定为**正式变更**
4. 使用轻量变更单记录，而不是直接改设计稿继续推进
5. 客户确认后，再回到设计/计划更新基线

## 阶段 4：Plan

`task_plan.md` 的重点不再是承载执行矩阵，而是把双轨交付控制相关 task 和任务图摘要写清楚。

本次演练拆出的关键任务：

- 试运行版交付任务
- 授权状态与到期行为验证
- 永久授权切换任务
- 源码移交任务
- 控制权移交任务
- 密钥配置移交任务

参考样例：

- [task_plan.md](../../../.trellis/tasks/03-30-dual-track-e2e-simulation/task_plan.md)

其中必须体现：

- `delivery_control_track`
- `delivery_control_handover_trigger`
- 尾款前保留控制范围
- 支付后移交任务不得提前进入执行候选
- 冻结后若新增 / 修改 / 删除需求，不直接在当前 `task_plan.md` 顺手吸收，而是先完成变更评估与审批

### 计划后的轻量校正点

到了 `plan` 阶段，如果任务拆解比 `design` 阶段更清楚，只允许对 `record-session` 做一次**轻量校正**，不要重做首次适配。

本次演练里，通常只需要补清三类信息：

- 哪些任务完成后应立即 archive
- 哪些任务虽然产物已齐，但必须等交付事件完成后才允许进入最终 `record-session`
- 试运行版交付、永久授权切换、最终控制权移交是否分别记录 session，还是合并记录

不应该在这里做的事：

- 不应第一次才思考 `record-session` 的入口和元数据门禁
- 不应回头大改 `finish-work`，除非 `plan` 阶段新增了前面没有识别出的强制检查命令

## 阶段 5：Delivery

对于双轨交付控制，`delivery` 阶段不再只是假设“最终一次性交付”，而是按**交付事件**组织文档。

### 这一步要验证的重点

- 当前交付事件是 retained-control delivery，还是 final control transfer
- 若是 final control transfer，是否已满足 `final_payment_received`
- 若走 `trial_authorization`，`trial_authorization_terms` 是否已附档
- 到期只读与导出是否已经验证
- 验收阶段若客户提出新增 / 修改 / 删除需求，是否已先进入 [需求变更管理执行卡](./需求变更管理执行卡.md)，而不是直接混入当前交付

### 极简演示：验收阶段客户临时提改动

假设客户在验收时说：

> “这轮先过，但导出页我还想多一个门店对比列。”

建议最小处理动作：

1. 先确认这不是对原验收标准的纯澄清
2. 由于它会改变导出内容与验收范围，判定为**正式变更**
3. 明确说明“不混入当前验收通过版本”
4. 记录到轻量变更单，评估对时间和费用的影响
5. 客户确认后，作为下一轮或补充变更执行

如果客户说的是：

> “我只是确认一下，只读模式还能不能导出历史数据？”

### 最小收尾样例：`finish-work` → `record-session`

虽然本演练不展开真实业务代码实现，但在这条链路下，完成 workflow / spec / design / plan / delivery 文档更新后，仍然建议按项目化后的命令做一次最小收尾。

**`/trellis:finish-work` 最小证据清单**

至少要能回答四个问题：

- 本轮真实跑了哪些验证命令
- 这些命令是否通过
- 本轮是否补齐了当前项目 spec / checklist / delivery 文档
- 本轮是否还有未闭环的任务或交付边界

对于本次演练，一个足够小的证据集合可以是：

```bash
git diff --name-only

python3 docs/workflows/新项目开发工作流/commands/shell/design-export.py \
  --validate .trellis/tasks/03-30-dual-track-e2e-simulation/design

python3 docs/workflows/新项目开发工作流/commands/shell/plan-validate.py \
  .trellis/tasks/03-30-dual-track-e2e-simulation
```

如果当前项目把双轨交付控制也纳入 `finish-work` 门禁，还应补充：

- 本轮交付事件属于 retained-control delivery 还是 final control transfer
- 若要进入最终移交，`final_payment_received` 是否已满足
- `transfer-checklist`、授权条款、密钥/配置移交边界是否已更新到当前文档基线

**`/trellis:record-session` 最小入口样例**

在本项目已经把 `record-session` 基线适配到 helper 入口的前提下，最小收尾流程可以写成：

```bash
python3 ./.trellis/scripts/workflow/record-session-helper.py \
  --title "workflow: close dual-track walkthrough alignment" \
  --commit "<human-commit-hash>" \
  --summary "Update workflow/design/plan/delivery guidance and close current task"

python3 ./.trellis/scripts/task.py archive <current-task>

git status --short .trellis/tasks .trellis/.current-task
```

收尾顺序：**先 record-session，再 archive**。record-session 需要当前任务上下文；archive 会清除 `.current-task`。

预期：

- `record-session-helper.py` 前后检查都通过
- `archive` 后，`.trellis/tasks` 与 `.trellis/.current-task` 已 clean
- `record-session` 完成后，`.trellis/workspace` 与 `.trellis/tasks` 不再残留脏变更
- 以上动作只作用于**当前完成的活动任务**，不借此批量整理旧任务或重写已归档记录

如果这些前置条件不满足，不应把本轮工作判定为真正收尾完成。

## 关键结论

这条演练链路验证了五件事：

1. 新的 `delivery_control_track` / `trial_authorization_terms` 契约已经能从 feasibility 贯穿到 delivery。
2. `authorization-management` 已从“外部项目无条件必选”收敛为“试运行授权轨条件必选”。
3. `transfer-checklist` 已从“只服务最终一次性交接”收敛为“服务每次正式交付事件”。
4. 从 PRD 冻结开始，后续阶段对“纯澄清”和“正式变更”可以采用不同出口，避免把冻结后的新增 / 修改 / 删除重新混入当前阶段。
5. 技术架构确认后的 `§3.7` 不只是补项目 spec，还应同步定准 `/trellis:finish-work`，并为 `/trellis:record-session` 建立项目化基线；`§4 plan` 后只做轻量校正。
6. `工作流总纲 §3.7` 是这套门禁的唯一主定义；`design` / `plan` / walkthrough 对它的描述都只是承接摘要，不应各自扩写成不同版本。
