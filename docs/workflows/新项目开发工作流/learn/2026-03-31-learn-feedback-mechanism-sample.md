## 问题现象 / 经验描述

我本来以为只要把 `learn/` 目录、筛选标准和人工确认边界补上，这个机制就已经完整了。

但回头从真实开发视角看，第一版还是太像“制度说明”：

- 说清了为什么要记，却没有说开发中什么时候会自然触发
- 说了要放到 `learn/`，但没有把 `/trellis:start` 和 `/trellis:delivery` 这两个最实际的入口挂起来
- 模板字段完整，但写出来的感觉更像流程报告，不像开发者复盘刚踩过的坑

真正卡住的不是“有没有 learn 机制”，而是“开发过程中意识到有坑时，能不能顺手触发并记下来”。

## 触发条件

- 在补完 `learn/README.md` 和 `TEMPLATE.md` 后，从“实际开发怎么用”角度复盘整套机制
- 用户继续追问“需要补充自然触发和命令触发两种方式”“不能太形式化”

## 影响

- 如果只停留在第一版，维护者还是容易把这件事推迟到项目结束才做
- 开发中途明明已经意识到流程有问题，也可能因为没有明确入口而不记录
- 模板容易被写成复盘八股文，导致后续记录价值下降

如果下次换个人来用，仍然有较大概率踩到同一个问题。
