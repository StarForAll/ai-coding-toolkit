# 经验反馈记录：<主题>

> 本模板提供两种格式，按场景选用：
>
> - **快速捕获**（30 秒）：开发中踩坑时快速记下，事后复盘再展开
> - **完整记录**（复盘时用）：结构化填写，包含根因分析和回流跟踪
>
> learn 文件是一次性的：复盘后改了工作流，文件使命就完成了。

---

## 快速捕获格式（开发中踩坑时用）

适合：AI 主动提示或用户说"记一下"时，30 秒内记完。

```markdown
# <一句话描述>

踩坑位置：<阶段/命令>
现象：<一句话>
可能原因：<一句话或"待分析">

<!-- 复盘时再展开为完整格式 -->
```

示例：

```markdown
# finish-work 前忘 archive 导致报错

踩坑位置：/trellis:finish-work
现象：直接跑 finish-work 报错，因为 .trellis/tasks 还没 clean
可能原因：finish-work 文档没有在 Step 1 明确写前置条件

<!-- 复盘时再展开为完整格式 -->
```

> 快速捕获不需要填完整模板的 4 个子字段。先记事实，复盘时再决定是否展开。

## 完整记录格式（复盘时用）

> 以下为完整模板，适合项目复盘或阶段性回顾时使用。

---

## 起草阶段（AI 或人工填写）

### 现象

- **发生了什么**：<具体行为描述，不要先下结论>
- **发生在哪个阶段/命令**：<如 plan 阶段、delivery 命令、record-session 等>
- **影响**：<导致了什么后果，如返工/误判/漏检/卡住/浪费时间>
- **初步判断**：<流程问题 / 文档问题 / 命令问题 / 其他>

可直接按下面这种口吻写：

- "我本来以为……，结果……"
- "真正卡住我的不是实现，而是……"
- "这一步文档里提到了，但实际执行时还是容易漏……"

### 详细描述（可选）

按真实开发顺序补充：当时发生了什么 → 为什么卡住 → 怎么发现并修正的。

能一句话说清的地方不要硬写成长篇。

### 建议改哪里

<具体的工作流修改建议，如"在命令映射中补充 plan 的依赖说明">

这里允许很朴素，不必写成正式提案。比如：

- "把这句话挪到前面"
- "在 Step 9 明说要补 learn 记录"
- "start 路由里加一个触发词就够了"
