# 目标项目工作流兼容升级方案指导

> 用于指导“已经嵌入本 workflow 的目标项目”在 Trellis 官方升级后，如何分析并执行 workflow 兼容升级。
>
> 这份文档是**分析与决策流程**，不是“一键自动升级固定程序”。

---

## 定位

这份文档只解决一个问题：

- 当目标项目已经使用了 `docs/workflows/新项目开发工作流`，且目标项目先执行了 Trellis 官方升级后，如何判断 workflow 相关文件应该做 `新增 / 替换 / 合并 / 删除 / 忽略`

它不负责：

- 代替当前仓库维护 workflow 源资产时的兼容判断
- 代替目标项目处理 Trellis 官方基线冲突
- 代替人工判断结构性 breaking change

---

## 默认决策规则

默认情况下，目标项目升级**先走兼容升级分析流程**，不要一开始就进入“结构性迁移”。

但这里有一个硬前置：

- 目标项目必须已经升级到**当前最新 Trellis 版本**
- 在满足这个条件之前，**禁止**执行 `A/B/C` 三态分析
- 在满足这个条件之前，**禁止**执行 `upgrade-compat.py --check/--merge/--force`

只有当兼容升级分析明确表明当前升级已经不是普通的 `add / replace / merge / delete` 文件级处理时，才进入结构性迁移。

默认顺序：

1. 先完成当前仓库 workflow 源资产兼容升级
2. 目标项目先完成 Trellis 官方升级，只处理官方基线冲突
3. 再执行 `A/B/C` 三态分析
4. 若分析结果仍可通过文件级动作处理，则继续兼容升级
5. 仅当分析结果显示“结构性 break”时，才进入结构性迁移

结构性迁移设计见：[结构性迁移设计](./结构性迁移设计.md)

---

## 硬前置

在使用这套方案前，必须先满足以下前置条件：

1. 当前项目已经完成 workflow 源资产兼容升级。
2. 目标项目已经完成 Trellis 官方升级，且这一步只处理 Trellis 官方基线冲突。
3. 目标项目当前 `.trellis/.version` 必须等于当前最新 Trellis 版本。

如果这些条件不成立：

- 不要执行只读 `A/B/C` 分析
- 不要执行 `upgrade-compat.py`
- 先回到 Trellis 官方升级步骤，把目标项目升级到当前最新版本后再回来

---

## 缺失 workflow 版本字段时的处理

若目标项目当前的 `.trellis/workflow-installed.json` 中**没有**以下字段：

- `workflow_version`
- `workflow_schema_version`

不要把它视为升级阻塞条件。

处理规则：

1. 将该目标项目视为 `legacy/unknown` 状态
2. 在目标项目已经升级到当前最新 Trellis 的前提下，当前这一次升级仍然按 `A/B/C` 三态分析执行
3. 不尝试仅靠缺失字段去反推历史版本或历史结构
4. 若分析结果仍可落在普通兼容升级范围内，就继续兼容升级
5. 若分析结果命中结构性 break，再进入结构性迁移

也就是说：

- 缺失版本字段时，当前方案依然可用
- 版本字段在这里的主要作用不是“补历史账”，而是为**这次升级完成之后的下一次升级**提供锚点

因此建议在本次兼容升级或结构性迁移完成后，将确认后的：

- `workflow_version`
- `workflow_schema_version`

回写到 `.trellis/workflow-installed.json`，避免后续继续以 `legacy/unknown` 状态重复分析。

## 三态模型

目标项目的 workflow 兼容升级，不直接拿“当前仓库 vs 目标项目”做二方 diff，而是使用三态模型：

- `A`：纯净基线
  - 在临时项目中执行与目标项目相同版本的 `trellis init`
  - 只保留 Trellis 官方基线，不安装当前 workflow
- `B`：期望状态
  - 在 `A` 的基础上安装当前仓库中已经完成兼容升级的最新 workflow
- `C`：真实状态
  - 目标项目在完成 Trellis 官方升级、并只处理官方基线冲突后的当前状态

三态职责：

- `A` 用来回答“新版 Trellis 官方基线长什么样”
- `B` 用来回答“当前仓库希望把 workflow 安装成什么样”
- `C` 用来回答“目标项目现在实际上是什么样”

## 托管范围

默认只分析 workflow 托管资产，不碰目标项目业务代码。

### 主分析范围

- Claude / OpenCode 的补丁型基线命令
  - `start`
  - `finish-work`
  - `record-session`
- Claude / OpenCode 的 workflow 分发命令
  - `brainstorm`
  - `check`
  - `feasibility`
  - `design`
  - `plan`
  - `test-first`
  - `review-gate`
  - `delivery`
- Codex 的 workflow skills
  - 同名阶段 skills
  - `finish-work` skill
- 共享 helper scripts
  - `.trellis/scripts/workflow/*.py`

### 默认不纳入主分析范围

- 目标项目业务代码
- Trellis 未被当前 workflow 托管的原生命令
- `AGENTS.md` NL 路由表
- `todo.txt`
- `pack.requirements-discovery-foundation` 导入结果
- bootstrap task 清理结果
- Codex 的 `hooks.json` / `session-start.py`

这些内容可以作为附加检查项，但不建议和主分析链混在一起。

## 脚本角色

当前目录下的相关脚本角色应这样理解：

- `commands/install-workflow.py`
  - 用于把 workflow 映射到临时项目，生成 `B` 期望状态
  - 它仍然是 source-of-truth 到目标布局的标准映射器
- `commands/analyze-upgrade.py`
  - 用于读取 `A/B/C` 三态并输出动作分类
  - 只分析，不直接修改目标项目
- `commands/upgrade-compat.py`
  - 用于低风险漂移修复、重部署、补丁恢复
  - 不应作为结构性升级或 breaking change 的主入口
- `commands/uninstall-workflow.py`
  - 用于回滚或清理，不是常规升级主链的一部分

## 推荐步骤

### Step 1：准备 `A` 纯净基线

在临时目录创建纯净 Trellis 项目：

```bash
mkdir -p /tmp/workflow-upgrade-a
cd /tmp/workflow-upgrade-a
# 在此执行与目标项目相同版本的 trellis init
```

得到的项目根目录记为 `A_ROOT`。

### Step 2：准备 `B` 期望状态

基于 `A_ROOT` 再准备一份期望状态项目，例如复制出 `B_ROOT`：

```bash
cp -R /tmp/workflow-upgrade-a /tmp/workflow-upgrade-b
```

然后在 `B_ROOT` 上安装当前仓库最新版 workflow：

```bash
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/install-workflow.py \
--project-root /tmp/workflow-upgrade-b
```

### Step 3：准备 `C` 真实状态

目标项目先执行 Trellis 官方升级，并只处理官方基线冲突。  
这一步完成后的项目根目录就是 `C_ROOT`。

同时必须确认：

- `C_ROOT/.trellis/.version` 已经是当前最新 Trellis 版本
- 若还不是当前最新版本，当前流程到此停止，不允许继续执行 Step 4，也不允许先做只读分析

### Step 4：执行三态分析

只有在 `C_ROOT` 已完成最新 Trellis 官方升级后，才能运行分析脚本。

运行分析脚本：

```bash
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/analyze-upgrade.py \
--baseline-root /tmp/workflow-upgrade-a \
--expected-root /tmp/workflow-upgrade-b \
--target-root /path/to/target-project \
--cli claude,opencode,codex \
--report /tmp/workflow-upgrade-report.md
```

如需机器可读输出：

```bash
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/analyze-upgrade.py \
--baseline-root /tmp/workflow-upgrade-a \
--expected-root /tmp/workflow-upgrade-b \
--target-root /path/to/target-project \
--cli claude,opencode,codex \
--json
```

### Step 5：按动作分类处理

分析脚本会把资产分类为：

- `keep`
  - `C` 已与 `B` 一致，无需处理
- `add`
  - 目标项目缺少该 workflow 资产，应补充
- `replace`
  - 目标项目仍停留在基线或缺少补丁，可按最新 workflow 替换/恢复
- `merge`
  - 目标项目内容既不等于 `A`，也不等于 `B`，必须人工判断
- `delete`
  - 最新 workflow 已不再托管该资产，目标项目仍保留旧内容

### Step 5.5：判断是否进入结构性迁移

满足以下任一情况时，不再把当前升级视为普通兼容升级，而应转入结构性迁移：

- 核心补丁型命令的注入锚点已经失效，无法再按旧补丁模型恢复
- workflow 自己发生阶段拆分、阶段合并、命令重命名、文件布局重构
- 目标项目里的 workflow 托管文件存在大面积私有改写，分析结果中 `merge` 项占主导
- 当前 workflow 的托管范围、安装副作用或运行入口协议发生变化
- `.backup-original` 已不能作为可靠恢复基线

一旦进入该分支，应停止直接使用 `upgrade-compat.py` 做主处理，转而依据 [结构性迁移设计](./结构性迁移设计.md) 输出迁移方案。

如果目标项目缺少 `workflow_version / workflow_schema_version`，这本身**不自动等于**结构性迁移，只表示当前项目仍处于 `legacy/unknown` 状态，仍需由 `A/B/C` 分析来决定下一步。

### Step 6：只对低风险项使用 `upgrade-compat.py`

当分析结果表明：

- 没有结构性 breaking change
- 目标项目没有重要私有 workflow 改动
- 主要是补丁丢失、命令漂移、helper script 漂移

可以再使用：

```bash
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/upgrade-compat.py \
--check --project-root /path/to/target-project
```

必要时再进入：

```bash
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/upgrade-compat.py \
--merge --project-root /path/to/target-project
```

`--force` 只适用于：

- `.backup-original` 仍可用
- 只是旧结构内的恢复问题
- 已明确不是 Trellis 或 workflow 自身的结构性 break

## 动作判定建议

| 情况 | 建议动作 |
|------|----------|
| `C == B` | `keep` |
| `A != B` 且 `C == A` | `replace` |
| `A` 中无该资产，`B` 有，`C` 无 | `add` |
| `C != A` 且 `C != B` | `merge` |
| `B` 无该资产，`C` 仍存在 | `delete` |

---

## 如果跳过了“私有改动盘点”且已经执行 `trellis update`

不要直接进入 `--merge` 或 `--force`。

正确补救方式：

1. 先把目标项目当前状态冻结为 `C`
2. 再准备 `A` 和 `B`
3. 执行三态分析
4. 先处理 `merge` 项
5. 最后只把低风险漂移交给 `upgrade-compat.py`

也就是说，漏掉的“盘点私有改动”并没有消失，只是要在 `trellis update` 之后补做。

---

## breaking change 边界

这套方案能降低 breaking change 风险，但不能消灭 breaking change。

以下情况仍必须人工判断：

- Trellis 官方命令骨架、注入锚点或标题结构发生变化
- 当前 workflow 自己发生阶段结构调整、命令重命名或文件布局变化
- 目标项目长期手改了 workflow 托管文件
- `.backup-original` 已丢失或不再对应当前官方基线

出现这些情况时：

- 先修当前仓库里的 workflow 源资产
- 再重新生成 `B`
- 然后重新分析目标项目，不要把 `upgrade-compat.py` 当作主要迁移机制
- 若确认已是结构性 break，则进入 [结构性迁移设计](./结构性迁移设计.md) 定义的流程，而不是继续把问题压成普通兼容升级

---

## 输出要求

最终建议至少形成一份升级报告，包含：

- 分析使用的 `A/B/C` 路径
- 涉及的 CLI 范围
- 每个托管资产的动作分类
- 哪些项可自动处理
- 哪些项必须人工合并
- 是否存在 breaking risk
- 最终执行顺序

这份报告才是后续对目标项目实际执行升级的依据。
