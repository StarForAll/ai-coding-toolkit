# 工作流总纲 → Trellis 命令映射

> 将《AI Workflow 项目开发工作流 V1.1.8》映射为可执行的 Trellis 命令体系。
> 基于 trellis-meta 规范，源文件在工作流子目录下，通过安装脚本部署到目标项目。
> 最后更新: 2026-04-01

---

## 多 CLI 同装原则

当前 workflow 的默认安装模型不是“按单个 CLI 分别装一遍”，而是：

- 在目标项目已完成 `trellis init` 后
- 自动检测该项目里已存在的 CLI 配置目录
- 将当前 workflow 支持的 CLI 适配层**同时**嵌入同一个目标项目

默认会一起部署当前检测到的：

- Claude Code 命令层
- OpenCode 命令层
- Codex skills / hooks / AGENTS 适配层

`--cli claude,opencode,codex` 仅用于**过滤**本次安装目标，不改变“多 CLI 同装”是默认主叙事这一事实。

> 关键区别：**同装不等于同一入口协议**。Claude Code 和 OpenCode 以项目命令为主，Codex 以 `AGENTS.md + hooks + skills + subagents` 为主，不提供项目级 `/trellis:xxx` 命令目录。

## 渐进性披露与配置分层

为了同时支持 Claude Code、OpenCode、Codex，而又不把默认上下文写得过重，这套 workflow 对 MCP / skills 采用渐进性披露：

| 层级 | 主要文件 | 是否默认应进入主上下文 | 作用 |
|------|---------|----------------------|------|
| 主规则层 | `工作流总纲.md` | 是 | 定义能力路由原则、降级规则、配置边界 |
| 映射层 | `命令映射.md` | 按阶段需要引用 | 定义阶段 × 场景 × 工具 / skill × 配置层矩阵 |
| 平台展开层 | `commands/claude/README.md` `commands/opencode/README.md` `commands/codex/README.md` | 否，按需查阅 | 解释各 CLI 的原生配置入口、承载方式和验证方法 |

### 配置层矩阵

| 配置层 | 应放内容 | Claude Code | OpenCode | Codex |
|-------|---------|------------|----------|-------|
| 项目稳定规则 | 工具优先级、联网门禁、`[Evidence Gap]` 口径 | `AGENTS.md` | `AGENTS.md` | `AGENTS.md` |
| workflow 阶段规则 | 在什么阶段优先调用哪类能力 | `工作流总纲.md` + `命令映射.md` + `.claude/commands/trellis/*.md` | `工作流总纲.md` + `命令映射.md` + `.opencode/commands/trellis/*.md` | `工作流总纲.md` + `命令映射.md` + `skills` / `AGENTS` / hooks 注入内容 |
| CLI 原生挂接 | hooks、权限、命令发现、instructions、skills、agents | `.claude/settings.json` `.claude/settings.local.json` `.claude/hooks/` `.claude/agents/` | `opencode.json.instructions` `.opencode/commands/` `.opencode/agents/` 与平台原生配置 | `.codex/config.toml` `.codex/hooks.json` `.agents/skills/` `.codex/agents/` |
| 复用能力资产 | skills / agents / 通用脚本本体 | skills / `.claude/agents/` / `.trellis/scripts/workflow/` | skills / `.opencode/agents/` / `.trellis/scripts/workflow/` | `.agents/skills/` / `.codex/agents/` / `.trellis/scripts/workflow/` |

约束：

- workflow 文档只定义“该用什么能力”，不负责保存凭据或 MCP server 注册细节
- 具体 MCP provider、凭据、本机权限应留在各 CLI 的原生配置面
- 平台展开层可以写最小配置示例，但不应要求主上下文默认加载整份 README

### 冻结后全局变更分流

> 当前工作流尚未定义独立 `/trellis:change-request` 命令。
> 因此，冻结后的需求变更以 **`§2.5 需求变更管理` 全局分支** 的方式执行，再回到主链路。

| 触发类型 | 路由 | 说明 |
|---------|------|------|
| 纯澄清 | 留在当前阶段 | 仅限不改变范围、接口契约、验收标准、成本、工期的澄清 |
| 新增需求 | 进入 `§2.5 需求变更管理` | 先评估影响、成本、工期、测试范围，未确认前不并入当前阶段 |
| 修改需求 | 进入 `§2.5 需求变更管理` | 与新增需求同级处理，不按“顺手改一下”留在当前阶段 |
| 删除需求 | 进入 `§2.5 需求变更管理` | 需评估已完成工作、交付边界、费用与排期影响 |
| 变更获批 | 回到受影响的最早阶段 | 例如回到 brainstorm / design / plan / start，对基线进行更新后再继续 |
| 变更未获批 | 保持当前阶段 | 维持冻结基线，不把变更混入当前交付 |

典型变更口语示例：

- “这个功能再加一个”
- “这个页面流程改一下”
- “这个接口字段要调整”
- “这个验收标准我想改”
- “这个功能先不做了”

典型纯澄清口语示例：

- “这里我确认一下是什么意思”
- “这个描述对应的是不是原来那个流程”
- “我不是加功能，只是想确认理解”

补充约束：
- `§1 feasibility` 的默认输出必须包含 `assessment.md`，并明确记录"是否允许进入 brainstorm"
- 新项目若缺少有效 `assessment.md`，不应直接进入 `§2 brainstorm`
- `§2.4` 冻结完成后，`§3` 到 `§7` 任一阶段若命中新增 / 修改 / 删除需求，应先走 `§2.5 需求变更管理`，而不是直接在当前阶段继续推进
- `§2.4` 冻结完成后，只有“纯澄清且不改变范围、接口契约、验收标准、成本、工期”才允许留在当前阶段
- 外部项目若缺少明确的交付控制轨道，不应直接进入正式交付；应先回到 `feasibility` 或 `plan` 补齐"托管部署 / 试运行授权"的约定
- 外部项目默认优先选择 `托管部署`；`试运行授权` 仅作为明确披露的备选轨，不得使用隐藏失效机制
- 进入 `record-session` 前，已完成任务应先归档；未完成任务不要误归档；非当前任务不得借本轮收尾顺手自动提交
- 元数据闭环只允许用于**当前执行任务完成后的最终收尾**
- 进入 `record-session` 前，`.trellis/tasks` 与 `.trellis/.current-task` 必须已经 clean；否则先处理 archive 或其它任务元数据问题
- `record-session` 进入 helper 前，staged 区不得混入非目标变更；否则应硬阻断
- `record-session` 执行后若 `.trellis/workspace` 或 `.trellis/tasks` 仍然 dirty，则不算记录完成
- `record-session` 的标准执行入口应使用 workflow helper，而不是直接调用 `add_session.py`
- `§3 design` 的 `3.7 技术架构确认后的项目 Spec 对齐` 是 `/trellis:finish-work` 的**主适配阶段**：必须把真实检查矩阵同步到对应项目命令
- `§3 design` 的 `3.7 技术架构确认后的项目 Spec 对齐` 也是 `/trellis:record-session` 的**基线适配阶段**：必须先写清记录入口、archive 前置条件、元数据边界
- `§4 plan` 完成后，仅允许对 `/trellis:record-session` 做一次**轻量校正**，用于补齐任务归档节点、里程碑粒度与记录粒度；不应把首次适配拖到 `§4` 之后
- 一般不需要在 `§4 plan` 后再次大改 `/trellis:finish-work`，除非计划阶段新增了新的强制检查门禁

### 双轨交付控制速查

| 阶段 | 必须冻结/检查的字段 | 关键动作 |
|---|---|---|
| `§1 feasibility` | `delivery_control_track` `delivery_control_handover_trigger` `delivery_control_retained_scope` | 决定走 `hosted_deployment` 还是 `trial_authorization` |
| `§3 design` | 先完成 `工作流总纲 §3.7` 的项目级 spec 对齐门禁，再进入 plan | 先导入适配 spec，再做项目化修订，再明确检查矩阵；外部项目基础必选 `delivery-control` + `transfer-checklist`；`trial_authorization` 再加 `authorization-management`；`finish-work` 在此阶段定准，`record-session` 在此阶段定基线 |
| `§4 plan` | `delivery_control_track` `delivery_control_handover_trigger` `trial_authorization_terms.*`；必要时轻量校正 `record-session` | 移交任务不得早于触发条件；把试运行交付、永久授权切换、最终控制权移交拆成单独任务；若任务粒度更明确，只轻量补 `record-session`，不重做首次适配 |
| `§6+§7 delivery` | 当前交付事件类型 `delivery_control_handover_trigger` 触发状态 | 区分"继续保留控制权的正式交付"与"最终控制权移交"；验证尾款到账后方可最终移交 |

#### 双轨验证命令

```bash

# 验证特定阶段

python3 docs/workflows/新项目开发工作流/commands/shell/delivery-control-validate.py --phase delivery --task-dir <task-dir>
```

> 进入 `§5.1.y` 前，当前 CLI 与目标 reviewer CLI 需预先具备对应 skill 能力；若缺失，先补齐 skill，再进入补充审查层。

`§2 brainstorm` 的任务生成前置规则：

- 先判断需求描述是否准确，再决定是否生成具体任务
- 复杂度统一采用 `L0/L1/L2`
- `L0` 可直接进入 `/trellis:start` 做单任务闭环
- `L1/L2` 进入 `/trellis:plan`；其中 `L2` 先补信息、再拆多个子任务
- 默认不回写历史任务基线；若旧任务需要调整，走 `§2.5 需求变更管理`

外部项目附加说明：

- `§1 feasibility` 负责确定是否采用双轨交付控制，以及选择 `托管部署` 还是 `试运行授权`
- `§4 plan` 负责把“试运行版交付 / 托管部署 / 永久授权切换 / 源码移交 / 控制权移交”拆成明确任务
- `§6+§7 delivery` 负责执行尾款前后的交付门禁与最终移交 checklist

## 下一步推荐输出规范

每个命令末尾的标准输出格式（由 AI 执行命令后生成）：

```markdown
## 下一步推荐

**当前状态**: <一句话描述>

| 你的意图 | 推荐命令 | 说明 |
|---------|---------|------|
| <意图> | `/trellis:xxx` | **默认推荐**。<说明> |
| <意图> | `/trellis:yyy` | <说明> |
| 不确定下一步 | `/trellis:start` | 用 Phase Router 自动检测 |
```

**设计原则**：
- 不只是线性链路，要给出多种可能路径
- 包含回退路径（回到上一步修改）
- 包含跳过路径（跳过某些阶段）
- 包含特殊情况路径（并行、Bug修复、规范更新）
- 始终包含「不确定 → start」兜底

## Trellis 版本升级兼容

当 `trellis init/upgrade` 更新了 Trellis 框架后：

```bash
# 1. 检测冲突（从项目内部运行，或用 --project-root 指定）
python3 docs/workflows/新项目开发工作流/commands/upgrade-compat.py --check
python3 docs/workflows/新项目开发工作流/commands/upgrade-compat.py --check --project-root /path/to/project

# 2. 自动合并（推荐）
python3 docs/workflows/新项目开发工作流/commands/upgrade-compat.py --merge

# 3. 强制覆盖（仅当自动合并失败）
python3 docs/workflows/新项目开发工作流/commands/upgrade-compat.py --force
```

### 升级处理流程

```
trellis init/upgrade
    │
    ▼
upgrade-compat.py --check
    │
    ├── 版本一致 + 部署完整 → 无需处理
    │
    ├── 版本一致 + 有漂移 → 运行 --merge 修复
    │
    ├── 版本不一致 + 无冲突 → 建议运行 --merge 重新部署并刷新版本标记
    │
    └── 版本不一致 + 有冲突
        ├── --merge: 重新部署命令/脚本，并自动注入 Phase Router
        ├── --force: 先从 .backup-original 恢复 start.md，再重新注入
        └── 失败: 手动合并 + 参考备份
```

### 冲突范围

| 文件 | 冲突风险 | 说明 |
|------|---------|------|
| `start.md` | **高** | 唯一被修改的官方文件 |
| 其他官方命令 | 无 | 未修改 |
| 7 个新命令 | 无 | 独立文件 |
| 4 个 helper scripts | 中 | 命令会直接引用，缺失会导致运行失败 |
