# 工作流总纲 → Trellis 命令映射

> 将《AI 辅助开发实战工作流 V0.1.24》映射为可执行的 Trellis 命令体系。
> 基于 trellis-meta 规范，源文件在工作流子目录下，通过安装脚本部署到目标项目。
> 最后更新: 2026-04-15

**原生适配范围**：Claude Code / OpenCode / Codex。仓库中若仍保留 Cursor、Gemini 等其他 CLI 文档，仅作为可选参考，有则使用，没有则忽略。

> 首次阅读请先看：[多CLI通用新项目完整流程演练](./多CLI通用新项目完整流程演练.md)。本文负责解释阶段到命令、skills、CLI 承载方式的映射，不替代通用主链 walkthrough。

## 多 CLI 同装原则

当前 workflow 的默认安装模型不是“按单个 CLI 分别装一遍”，而是：

- 目标项目本身已经是 Git 仓库
- `origin` 已至少配置两个 push URL
- 在目标项目已完成 `trellis init` 后
- 自动检测该项目里已存在的 CLI 配置目录
- 将当前 workflow 支持的 CLI 适配层**同时**嵌入同一个目标项目

默认会一起部署当前检测到的：

- Claude Code 命令层
- OpenCode 命令层
- Codex skills / hooks / AGENTS 适配层

`--cli claude,opencode,codex` 仅用于**过滤**本次安装目标，不改变“多 CLI 同装”是默认主叙事这一事实。

> 关键区别：**同装不等于同一入口协议**。Claude Code 和 OpenCode 以项目命令为主，Codex 以 `AGENTS.md + hooks + skills + subagents` 为主，不提供项目级 `/trellis:xxx` 命令目录。

## 安装执行顺序

当前 workflow 的正确安装顺序是：

1. 目标项目先完成 `trellis init`
2. 再运行 `docs/workflows/新项目开发工作流/commands/install-workflow.py`
3. 由安装脚本把 workflow 支持的 CLI 适配层按各 CLI 官方原生格式直接嵌入目标项目
4. 安装脚本自动导入 `pack.requirements-discovery-foundation` 并删除 `00-bootstrap-guidelines`

标准命令：

```bash
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/install-workflow.py \
--project-root <target-project>
```

注：

- 上图中 `check → finish-work` 是默认主链
- 上图中 `check → review-gate` 仅表示条件触发分支，不是默认路径

这一步完成后，目标项目里会直接获得对应的命令、skills、hooks 和补丁增强，初始需求发现基础资产也会通过脚本落到 `.trellis/`，再按各 CLI 的原生入口使用。

安装完成后，实际交流和执行依然可以采用**渐进性披露**：

- workflow 先按当前阶段进入对应入口
- 再按需要展开该阶段命令、映射、平台配置和能力说明

因此，下面的“渐进性披露”描述的是**安装后的使用层**，不是安装动作本身。

## 原生入口与推荐输出口径

下文所有“自然语言路由”“下一步推荐”“阶段入口示例”都应遵守同一口径：

| CLI | 原生入口 | 推荐输出时的写法 |
|-----|---------|----------------|
| Claude Code | 项目命令 | `/trellis:<phase>` |
| OpenCode | TUI 项目命令 / CLI 命令 | TUI 写 `/trellis:<phase>`；CLI 可补 `trellis/<phase>` |
| Codex | 自然语言 + 对应 skill | 推荐“继续某阶段的自然语言意图”，并补同名 skill，如 `brainstorm` skill |

约束：

- 不把 `/trellis:xxx` 直接当作 Codex 的唯一推荐入口
- 同一条“下一步推荐”若面向多 CLI，必须把 Codex 的 skill / 自然语言入口单独说清楚
- “不确定下一步”时：
  - Claude / OpenCode：推荐 `/trellis:start`
  - Codex：推荐描述当前意图，或显式触发 `start` skill

## 渐进性披露与配置分层（安装后的使用层）

为了同时支持 Claude Code、OpenCode、Codex，而又不把默认上下文写得过重，这套 workflow 对 MCP / skills 采用渐进性披露：

| 层级 | 主要文件 | 是否默认应进入主上下文 | 作用 |
|------|---------|----------------------|------|
| 主规则层 | `工作流总纲.md` | 是 | 定义能力路由原则、降级规则、配置边界 |
| 映射层 | `命令映射.md` | 按阶段需要引用 | 定义阶段 × 场景 × 工具 / skill × 配置层矩阵 |
| 平台展开层 | `commands/claude/README.md` `commands/opencode/README.md` `commands/codex/README.md` | 否，按需查阅 | 解释各 CLI 的原生配置入口、承载方式和验证方法 |

### 配置层矩阵

| 配置层 | 应放内容 | Claude Code | OpenCode | Codex |
|-------|---------|------------|----------|-------|

| workflow 阶段规则 | 在什么阶段优先调用哪类能力 | `工作流总纲.md` + `命令映射.md` + `.claude/commands/trellis/*.md` | `工作流总纲.md` + `命令映射.md` + `.opencode/commands/trellis/*.md` | `工作流总纲.md` + `命令映射.md` + `skills` / `AGENTS` / hooks 注入内容 |
| CLI 原生配置层 | hooks、权限、命令发现、instructions、skills、agents | `.claude/settings.json` `.claude/settings.local.json` `.claude/hooks/` `.claude/agents/` | `opencode.json.instructions` `.opencode/commands/` `.opencode/agents/` 与平台原生配置 | `.codex/config.toml` `.codex/hooks.json` `.agents/skills/` `.codex/agents/` |

约束：

- workflow 文档只定义“该用什么能力”，不负责保存凭据或 MCP server 注册细节
- 具体 MCP provider、凭据、本机权限应留在各 CLI 的原生配置面
- 平台展开层可以写最小配置示例，但不应要求主上下文默认加载整份 README

## 分支路径速查表

每个阶段完成后，用户可能想做什么 → 推荐什么命令：

| 当前阶段 | 默认下一步 | 回退修改 | 跳过前进 | 其他可能 |
|---------|-----------|---------|---------|---------|
| §1 feasibility | brainstorm（仅 `assessment.md` 明确允许，且首次入口已完成法律/合规门禁与 `main` 分支门禁时） | 补信息后重跑 feasibility | —（首次不可跳过；若已完成且仍有效，则复用 assessment 结果继续 brainstorm；若强行跳到 implementation，仍会被后续粗估门禁拦截） | 终止项目 / 谈判后再评估 |
| §2 brainstorm | design（默认，前提：已补齐 `docs/requirements/*.md`，并完成项目级粗估门禁；若这是首次入口还需先通过 `main` 分支门禁，并明确质量平台门禁） | — | plan（`L1/L2`） / start（仅 `L0` 单任务闭环，不强制双 PRD，但仍需粗估） | — |
| §3 design | plan（仅块 A/B/C/D 全部完成 + design 退出清单完成 + 用户明确确认后） | brainstorm | start / test-first（仅显式先测） | check-cross-layer；若存在前端视觉落地链路，需先拆 `UI -> 首版代码界面` task，并约束该 task 不得由 Codex 作为主执行器 |
| §4 plan | start（仅用户明确确认进入某个叶子 task 的 implementation / test-first 后） | design | test-first（仅显式先测，仍需用户确认并设置 `execution_authorized=true`） | 项目级审查任务在全部代码相关 task 完成后才成为候选入口 |

| §5 start | check（单任务修复回路） / project-audit（多任务/跨模块/发版前/高blast radius 条件触发） | brainstorm | — | break-loop / update-spec |

| §5.1.y review-gate | finish-work | start (修复) | — | 人工裁决 / multi-cli-review-action |

| §7 delivery | record-session | start (修复) | — | break-loop / update-spec / archive |

### 冻结后全局变更分流

> 当前工作流尚未定义独立 `/trellis:change-request` 命令。
> 因此，冻结后的需求变更以 [需求变更管理执行卡](./需求变更管理执行卡.md) 的方式执行，再回到主链路。

| 触发类型 | 路由 | 说明 |
|---------|------|------|
| 纯澄清 | 留在当前阶段 | 仅限不改变范围、接口契约、验收标准、成本、工期的澄清 |
| 新增需求 | 进入 [需求变更管理执行卡](./需求变更管理执行卡.md) | 先评估影响、成本、工期、测试范围，未确认前不并入当前阶段 |
| 修改需求 | 进入 [需求变更管理执行卡](./需求变更管理执行卡.md) | 与新增需求同级处理，不按”顺手改一下”留在当前阶段 |
| 删除需求 | 进入 [需求变更管理执行卡](./需求变更管理执行卡.md) | 需评估已完成工作、交付边界、费用与排期影响 |
| 变更获批 | 回到受影响的最早阶段 | 例如回到 brainstorm / design / plan / start，对基线进行更新后再继续；并同步回写 `docs/requirements/customer-facing-prd.md` 与 `docs/requirements/developer-facing-prd.md` |
| 变更未获批 | 保持当前阶段 | 维持冻结基线，不把变更混入当前交付 |

典型变更口语示例：

- “这个功能再加一个”
- “这个页面流程改一下”
- “这个接口字段要调整”
- “这个验收标准我想改”
- “这个功能先不做了”

典型纯澄清口语示例：

- “这里我确认一下是什么意思”
- “这个描述对应的是不是原来那个流程”
- “我不是加功能，只是想确认理解”

补充约束：
- `§1 feasibility` 的默认输出必须包含 `assessment.md`，并明确记录"是否允许进入 brainstorm"
- `§1 feasibility` 的默认输出必须包含 `project_engagement_type`；所有项目都必须先完成项目类别判断
- 当前 workflow 首次进入时，若目标项目为新建仓库，必须先检查本地主分支和初始分支是否为 `main`；若目标项目已有本地提交历史，仅记录现状，不强制切换
- 新项目若缺少有效 `assessment.md`，不应直接进入 `§2 brainstorm`；若已有有效 assessment，则属于复用已完成阶段结果，而不是跳过 `§1 feasibility`
- 判断 assessment 是否仍有效时，至少要确认：`总体决策` 不是 `暂停/拒绝`、`是否允许进入 brainstorm = 是`、`法律/合规风险结论` 已明确，且当前前提未发生足以推翻评估结论的变化
- 若 `project_engagement_type = external_outsourcing`，则 `kickoff_payment_ratio`、`kickoff_payment_received` 与 `delivery_control_*` 字段都属于强制门禁，不是可选建议
- `§2 brainstorm` 完成后，进入下一阶段前至少必须补齐项目级正式需求文档 `docs/requirements/customer-facing-prd.md`，并同步落盘不可跳过的项目级粗估：`task_dir/prd.md` 中的 `## 项目级粗估` 与 `customer-facing-prd.md` 中的 `## 项目级粗估摘要`；`docs/requirements/developer-facing-prd.md` 等到 design 阶段技术架构确认后再正式生成
- `§2 brainstorm` 完成并准备进入 `§3 design` 时，必须明确项目自动化检查矩阵：采用 Sonar 的项目必须写真实命令，未采用时必须写替代门禁和原因
- `task_dir/prd.md` 只作为当前任务工作底稿，不能替代上述项目级正式需求文档
- `§2.4` 冻结完成后，`§3` 到 `§7` 任一阶段若命中新增 / 修改 / 删除需求，应先走 [需求变更管理执行卡](./需求变更管理执行卡.md)，而不是直接在当前阶段继续推进
- `§2.4` 冻结完成后，只有“纯澄清且不改变范围、接口契约、验收标准、成本、工期”才允许留在当前阶段
- 外部项目若缺少明确的交付控制轨道，不应直接进入正式交付；应先回到 `feasibility` 或 `plan` 补齐"托管部署 / 试运行授权"的约定
- 外部项目默认优先选择 `托管部署`；`试运行授权` 仅作为明确披露的备选轨，不得使用隐藏失效机制
- 收尾顺序：先 `record-session`（需要当前任务上下文），再 `archive`（会清除 `.current-task`）；未完成任务不要误归档；非当前任务不得借本轮收尾顺手自动提交
- 元数据闭环只允许用于**当前执行任务完成后的最终收尾**
- `archive` 后，`.trellis/tasks` 与 `.trellis/.current-task` 应已 clean
- `record-session` 进入 helper 前，staged 区不得混入非目标变更；否则应硬阻断
- `record-session` 执行后若 `.trellis/workspace` 或 `.trellis/tasks` 仍然 dirty，则不算记录完成
- `record-session` 的标准执行入口应使用 workflow helper，而不是直接调用 `add_session.py`
- 阶段推进、矩阵状态切换、archive 与 record-session 默认都只作用于**当前活动任务**；不得借收尾或路由顺手回填旧任务、历史矩阵或已归档记录
- 当前阶段只允许按 `.current-task -> 当前叶子任务 -> workflow-state.json` 判定；若缺少这条链路，不允许自动跨阶段推进
- `§3 design` 的 `3.7 技术架构确认后的项目 Spec 对齐` 是 `/trellis:finish-work` 的**主适配阶段**：必须把真实检查矩阵同步到对应项目命令
- `§3 design` 的自动化检查矩阵必须有明确质量平台门禁；采用 Sonar 的项目必须写真实命令，未采用时必须写替代门禁和原因
- `§3 design` 的 `3.7 技术架构确认后的项目 Spec 对齐` 也是 `/trellis:record-session` 的**基线适配阶段**：必须先写清记录入口、archive 前置条件、元数据边界
- `§4 plan` 完成后，仅允许对 `/trellis:record-session` 做一次**轻量校正**，用于补齐任务归档节点、里程碑粒度与记录粒度；不应把首次适配拖到 `§4` 之后
- 一般不需要在 `§4 plan` 后再次大改 `/trellis:finish-work`，除非计划阶段新增了新的强制检查门禁

## 下一步推荐输出规范

每个命令末尾的标准输出格式（由 AI 执行命令后生成）：

> 这里定义的是阶段完成后的推荐输出契约，用于帮助当前 CLI 或协作者说明下一步；它不是框架层自动跳转保证。

```markdown
## 下一步推荐

**当前状态**: <一句话描述>

| 你的意图 | Claude / OpenCode 推荐入口 | Codex 推荐入口 | 说明 |
|---------|---------------------------|----------------|------|
| <意图> | `/trellis:xxx` | 自然语言继续该阶段，或显式触发 `xxx` skill | **默认推荐**。<说明> |
| <意图> | `/trellis:yyy` | 自然语言转入对应阶段，或显式触发 `yyy` skill | <说明> |
| 不确定下一步 | `/trellis:start` | 描述当前意图，或显式触发 `start` skill | 用 Phase Router / skill 路由做阶段检测 |
```

**设计原则**：
- 不只是线性链路，要给出多种可能路径
- 包含回退路径（回到上一步修改）
- 包含跳过路径（跳过某些阶段）
- 包含特殊情况路径（并行、Bug修复、规范更新）
- 始终包含「不确定 → start」兜底
- 对 Codex 必须使用 skill / 自然语言入口表达，不复用 `/trellis:xxx` 作为唯一写法
- 若当前阶段是 `plan`，默认推荐必须写成“确认进入某个叶子 task 的 implementation / test-first”，不能写成“直接开始做”

## 自然语言路由表

> 当用户用自然语言描述意图时，AI 可根据本表收敛候选阶段入口与推荐命令。
> 本表由 `install-workflow.py` 部署到目标项目的 session context（有 hooks 的 CLI）或 AGENTS.md（无 hooks 的 CLI）。
> 它用于缩小候选范围，不表示所有 CLI 都存在确定性的自动命令路由；若上下文不足、前置条件缺失或候选接近，仍应先确认。

### 路由规则

1. **状态机优先**：当前阶段只允许按 `.current-task -> 当前叶子任务 -> workflow-state.json` 判定；若缺少这条链路，则停止自动跨阶段推断
2. **上下文优先**：如果当前有活跃任务且处于某阶段，优先匹配该阶段或同阶段内的子块入口
3. **精确优先**：精确匹配关键词 > 模糊语义匹配
4. **兜底路由**：匹配不到 → `/trellis:start`（由 Phase Router 做阶段检测）
5. **歧义确认**：当 top-2 候选优先级接近时，向用户确认意图

### 歧义消解优先级

当多个命令匹配同一输入时，按以下规则消解：

| 优先级 | 规则 | 示例 |
|--------|------|------|

| 3 | 当前已确认阶段优先 | 当前仍在 design 且等待确认 → 继续 `/trellis:design`，而不是自动跳 `/trellis:plan` |

| 5（兜底） | 无法确定 | → `/trellis:start`（由 Phase Router 做阶段检测） |

## Trellis 版本升级兼容

当 `trellis init/upgrade` 更新了 Trellis 框架后：

### 当前仓库维护 workflow 源内容时：先拿到纯净 Trellis 基线

当需要在**当前仓库内**判断这套 workflow 的命令、补丁、脚本和兼容逻辑该如何跟随 Trellis 升级时，先按仓库通用规范获取“初始 Trellis 基线”：

1. 在 `/tmp` 下创建临时项目
2. 对临时项目执行 `trellis init`
3. 以该临时项目中的纯净 `.trellis/`、CLI 基线命令 / skills / hooks 作为参考基线

这一步的目的不是安装 workflow，也不是直接作为目标项目执行升级兼容时的判定标准，而是先回答：**当前 Trellis 版本原始提供的框架结构到底是什么**。

不要直接把当前仓库内的 `.trellis/`、`.claude/`、`.opencode/`、`.codex/` 目录当作“初始 Trellis 基线”。这些目录已经承载了本项目自己的 workflow 补丁与平台适配，只能作为“已落地实现参考”。

完成纯净基线获取后，先把两份对象放在一起判断：

- `/tmp` 临时项目的纯净 Trellis 基线
- 当前 workflow 需要注入的命令、补丁、helper scripts

若后续还要评估已安装项目会受到什么影响，再**额外**查看真实目标项目当前已安装的 workflow 状态。

这里要特别区分：

- **当前仓库维护 workflow 内容时的规范**：先用 `/tmp` + `trellis init` 获取纯净基线，再对照 workflow 源资产判断 Trellis 原始框架变化
- **目标项目执行兼容升级时的推荐流程**：先做 `A/B/C` 三态分析，再决定哪些项可直接替换、哪些必须人工合并
- **目标项目执行 `upgrade-compat.py` 时的自动化实现**：读取目标项目 `.trellis/.version` 与 `.trellis/workflow-installed.json`，并检查本地部署资产是否漂移，用于低风险修复而不是替代分析

也就是说：

- `analyze-upgrade.py` 是目标项目兼容升级的**主分析入口**
- `upgrade-compat.py` 是目标项目部署后的**低风险自动化修复层**
- 二者都不能替代当前仓库维护 workflow 内容时的纯净基线对照
- 默认先走兼容升级分析；只有当分析结果明确显示“结构性 break”时，才进入《结构性迁移设计》定义的迁移流程

### 升级处理流程

```
在当前仓库维护 workflow 内容时，于 /tmp 临时项目执行 trellis init，获取纯净 Trellis 基线
    │
    ▼
对照 workflow 源资产，判断 Trellis 原始框架变化
    │
    ▼
必要时更新当前仓库内的命令、补丁、脚本和说明文档
    │
    ▼
当下游目标项目需要实际执行升级时：
    │
    ▼
先在目标项目执行 Trellis 官方升级，只处理官方基线冲突
    │
    ▼
确认目标项目 `.trellis/.version` 已等于当前最新 Trellis 版本
    │
    ├── 否 → 当前步骤停止，禁止执行只读 A/B/C 分析和 upgrade-compat.py
    └── 是 → 继续
    │
    ▼
准备 A/B/C 三态：
  A = 纯净 Trellis 基线
  B = 安装当前 workflow 后的期望状态
  C = 目标项目真实状态
    │
    ▼
analyze-upgrade.py
    │
    ├── keep → 无需处理
    ├── add / replace → 可按分析结果执行
    ├── merge → 人工判断、手动合并
    └── delete → 按当前 workflow 是否已放弃托管决定是否移除
    │
    ▼
仅对低风险漂移使用 upgrade-compat.py
    │
    ├── --check: 校验补丁/命令/helper script 漂移
    ├── --merge: 重部署命令/脚本，并恢复 `start` / `finish-work` / `record-session` 补丁
    └── --force: 仅在 `.backup-original` 有效且确认不是结构性 break 时使用
```

示例命令：

```bash
# 0. 只有目标项目已经升级到当前最新 Trellis 时，才允许继续下面两步

# 1. 三态分析（推荐主入口）
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/analyze-upgrade.py \
--baseline-root /tmp/workflow-upgrade-a \
--expected-root /tmp/workflow-upgrade-b \
--target-root /path/to/project \
--cli claude,opencode,codex \
--report /tmp/workflow-upgrade-report.md

# 2. 低风险修复（仅在分析后确认可自动处理时使用）
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/upgrade-compat.py \
--check --project-root /path/to/project
```

### 冲突范围

| 文件 | 冲突风险 | 说明 |
|------|---------|------|
| `start.md` | **高** | 保留 Trellis 基线后再注入 Phase Router |
| `finish-work.md` / `finish-work` skill | **高** | 保留 Trellis 基线后再注入项目化补丁 |
| `record-session.md` | **高** | 保留 Trellis 基线后再注入元数据闭环补丁 |
| 8 个 workflow 分发命令 | 中 | 其中 `brainstorm` / `check` 为同名合并命令，`feasibility` / `design` / `plan` / `test-first` / `review-gate` / `delivery` 为纯新增命令；当前 `--check` 会校验它们与源资产的一致性 |
| 7 个 helper scripts | 中 | 命令会直接引用，缺失或内容漂移都会导致运行失败 |
