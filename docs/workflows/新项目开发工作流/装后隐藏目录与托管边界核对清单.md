# 装后隐藏目录与托管边界核对清单

> 用于在目标项目执行 `trellis init` + `install-workflow.py` 之后，核对隐藏目录、托管边界与历史漂移风险。
> 本文档只回答三件事：哪些资产由 `trellis init` 提供、哪些由当前 workflow 安装器托管、哪些仍需人工维护。

## 适用边界

本文档适用于两类场景：

1. 当前仓库在维护 `docs/workflows/新项目开发工作流/` 时，需要确认装后真实落盘结果
2. 某个目标项目已经嵌入本 workflow，需要做装后核对或升级后复查

本文档不替代：

- [工作流总纲](./工作流总纲.md) 中的主流程定义
- [CLI原生适配边界矩阵](./CLI原生适配边界矩阵.md) 中的按 CLI 边界说明
- [目标项目兼容升级方案指导](./目标项目兼容升级方案指导.md) 中的 `A/B/C` 三态分析

## 先分清三层来源

```text
trellis init baseline
  -> 目标项目的纯净 Trellis 隐藏目录与基线命令 / skills / hooks
workflow installer managed assets
  -> 当前 workflow 注入的命令、skills、共享 helper、workflow.md patch、AGENTS.md 路由块、安装记录
manual project-owned assets
  -> 项目自己维护的 settings / hooks / instructions / config / 非托管 agents
```

判断漂移前，先不要把这三层混在一起。

补充一个维护边界：

- `docs/workflows/新项目开发工作流/commands/workflow_assets.py` 是当前 workflow 托管资产常量的单一事实源
- `install-workflow.py`、`upgrade-compat.py`、`analyze-upgrade.py`、`test_workflow_installers.py` 都以这里的常量为准
- 若文档、安装器、升级器、测试之间出现"到底托管哪些命令 / helper / agents / patch"的口径冲突，先回到 `workflow_assets.py` 对照

## Step 1：先跑自动化核对

从当前仓库执行：

```bash
/ops/softwares/python/bin/python3 \
docs/workflows/新项目开发工作流/commands/upgrade-compat.py \
--project-root <target-project> \
--check
```

这一步负责检查：

- workflow 分发命令 / skills 是否与当前 source contract 一致
- `start` / `finish-work` / `record-session` patch 是否仍在
- `parallel` 禁用覆盖是否漂移
- shared helper scripts 与 `.trellis/workflow.md` patch 是否仍在
- `AGENTS.md` 中的 `workflow-nl-routing` 区段是否缺失或内容漂移

边界：

- 若 `--check` 通过，命令内容一致性、shared helper scripts、一部分 patch 存在性可视为已完成自动化核对，后续人工步骤以抽检和边界补充为主
- 若 `--check` 失败，应先根据输出修复对应漂移，再继续人工核对
- `--check` 仍**不会**替代以下人工核对：`origin` 双 push URL、`.claude/settings*.json`、`opencode.json.instructions`、`.codex/hooks.json`、`.codex/hooks/session-start.py` 等项目自维护资产

若这一步失败，不要直接跳过去人工脑补“应该没问题”。

## Step 2：逐项核对隐藏目录

### A. `trellis init` 基线产物

| 资产 | 期望状态 | 归属 |
|------|----------|------|
| `AGENTS.md` 内 `<!-- TRELLIS:START ... TRELLIS:END -->` | 必须存在 | `trellis init` 托管 |
| `.trellis/.version` | 必须存在 | `trellis init` 托管 |
| `.trellis/workflow.md` | 必须存在；之后允许被 workflow 安装器追加 patch | `trellis init` 基线 + workflow patch |
| `.claude/commands/trellis/start.md` `finish-work.md` `record-session.md` | Claude baseline 命令存在 | `trellis init` 基线 |
| `.opencode/commands/trellis/start.md` `finish-work.md` `record-session.md` | OpenCode baseline 命令存在 | `trellis init` 基线 |
| `.agents/skills/start/SKILL.md` `finish-work/SKILL.md` `record-session/SKILL.md` | Codex 共享 baseline skills 存在 | `trellis init` 基线 |
| `.codex/skills/parallel/SKILL.md` | 可能存在，也可能不存在；存在时需纳入影响面 | `trellis init` 可能创建的额外目录 |

### B. 当前 workflow 安装器托管资产

| 资产 | 期望状态 | 归属 |
|------|----------|------|
| `.trellis/workflow-installed.json` | 必须存在，且字段完整；至少可见 `initial_pack`、`bootstrap_task_removed`、`bootstrap_cleanup_status` | workflow 安装器托管 |
| `.trellis/library-lock.yaml` | 若安装器已成功导入 `pack.requirements-discovery-foundation`，应能看到对应导入记录 | workflow 安装器导入结果 |
| `AGENTS.md` 内 `<!-- workflow-nl-routing-start ... workflow-nl-routing-end -->` | 必须存在 | workflow 安装器托管 |
| `.trellis/workflow.md` 中 workflow patch | 必须存在 | workflow 安装器托管 |
| `.trellis/scripts/workflow/*.py` | helper scripts 已分发 | workflow 安装器托管 |
| `.claude/commands/trellis/*.md` | 阶段命令已分发；`start` / `finish-work` / `record-session` / `parallel` patch 正常 | workflow 安装器托管 |
| `.opencode/commands/trellis/*.md` | 阶段命令已分发；`start` / `finish-work` / `record-session` / `parallel` patch 正常 | workflow 安装器托管 |
| `.agents/skills/<phase>/SKILL.md` | 阶段 skills 已同步 | workflow 安装器托管 |
| `.codex/skills/<phase>/SKILL.md` | 若该目录存在，则同一批阶段 skills 也应同步存在 | workflow 安装器托管 |
| `.agents/skills/start/SKILL.md` `.agents/skills/finish-work/SKILL.md` | 活动 skills 目录中的 Phase Router / finish-work patch 正常 | workflow 安装器托管 |
| `.agents/skills/parallel/SKILL.md` `.codex/skills/parallel/SKILL.md` | 若文件存在，则应为禁用覆盖版本 | workflow 安装器托管 |

补充说明：

- `bootstrap_task_removed` 是向后兼容保留的布尔字段，只表示"本次安装是否实际执行过删除"
- `bootstrap_cleanup_status` 提供更细的状态，例如 `removed`、`absent`、`dry-run-removed`、`unknown`
- 判断 bootstrap task 的真实现状时，应结合 `.trellis/tasks/00-bootstrap-guidelines` 是否仍存在一起看，不要单看其中任一字段

### C. 仍需人工维护的项目资产

| 资产 | 期望状态 | 归属 |
|------|----------|------|
| `.claude/settings.json` `.claude/settings.local.json` | 与项目 MCP / 权限策略一致 | 项目人工维护 |
| `.claude/hooks/*.py` | 能继续做会话 / 子代理上下文注入 | 项目人工维护 |
| `opencode.json.instructions` | 与当前主入口文档挂接关系一致 | 项目人工维护 |
| `.opencode/plugins/*.js` `.opencode/package.json` | 能继续注入上下文 | 项目人工维护 / Trellis baseline |
| `.codex/config.toml` | `AGENTS.md` fallback 与项目配置正常 | 项目人工维护 |
| `.codex/hooks.json` `.codex/hooks/session-start.py` | 仍能注入 Trellis 会话上下文 | 项目人工维护 |
| 非 `research / implement / check` 的 agents | 与项目自定义约束一致 | 项目人工维护 |

## Step 3：Codex 双 skills 目录单独核对

这是最容易遗漏、也最容易产生历史漂移的点。

### 事实边界

- `.agents/skills/` 是共享 / 通用 workflow skills 的主承载面
- `.codex/skills/` 是 `trellis init` 之后**可能出现**的额外影响面，通常承载 Codex 特有或本地侧 skills
- 当前 workflow 安装器会向**所有存在的 skills 目录**同步写入阶段 skills
- **活动 skills 目录** = `resolve_codex_skills_dir(root)` 解析出的主目录，当前 `trellis init` 下通常是 `.agents/skills/`
- `start` / `finish-work` patch 只打在**活动 skills 目录**
- `record-session` 不存在 Codex 专属 patch；它仍依赖 Trellis baseline skill + `.trellis/scripts/workflow/record-session-helper.py`

### 最小核对命令

在目标项目根执行：

```bash
# 前 4 条是 workflow 通用前提与共享 patch 核对，不是 Codex 特有检查
git remote get-url --push --all origin | wc -l
rg -n "workflow-nl-routing-start|workflow-nl-routing-end" AGENTS.md
rg -n "<!-- workflow-projectization-patch -->" .trellis/workflow.md
rg -n "pack.requirements-discovery-foundation" .trellis/library-lock.yaml

# 下面才是 Codex skills 目录专项核对
ls -d .agents/skills/ .codex/skills/ 2>/dev/null
test -f .agents/skills/start/SKILL.md
test -f .agents/skills/finish-work/SKILL.md
ls .agents/skills/parallel/SKILL.md 2>/dev/null
ls .codex/skills/parallel/SKILL.md 2>/dev/null
rg -n "workflow-parallel-disabled" .agents/skills .codex/skills 2>/dev/null
```

核对规则：

- `git remote get-url --push --all origin | wc -l` 的结果应至少为 `2`
- `AGENTS.md` 中同时要能看到 `TRELLIS` managed block 与 `workflow-nl-routing` 区段；后者是 workflow 安装器托管面
- `.trellis/workflow.md` 应包含 `<!-- workflow-projectization-patch -->`
- `.trellis/library-lock.yaml` 是需求发现基础资产导入结果的直接证据；若该文件不存在，先回看安装器输出和 `trellis-library/cli.py assemble --auto` 的执行结果，不要直接把问题归因成命令 patch 漂移
- 若 `.trellis/library-lock.yaml` 存在但没有 `pack.requirements-discovery-foundation`，先对照 `.trellis/workflow-installed.json` 中的 `initial_pack`，再人工重跑一次 `trellis-library/cli.py assemble --target <project> --pack pack.requirements-discovery-foundation --auto`
- 非活动目录中的 `start` / `finish-work` 不属于当前 workflow patch 托管面，不应按“缺 patch = 漂移”处理
- 但只要某个目录里存在 `parallel/SKILL.md`，就必须核对它是否已经被禁用覆盖
- 只要 `.codex/skills/` 存在，就必须被算进装后影响面，而不是忽略

### 双 skills 目录核对矩阵

| 资产类型 | `.agents/skills/` | `.codex/skills/` |
|----------|-------------------|------------------|
| 阶段 skills（`feasibility` 到 `delivery`） | 若目录存在，安装器同步写入 | 若目录存在，安装器同步写入 |
| `start` / `finish-work` patch | 只在活动 skills 目录检查与增强 | 非活动目录默认不要求存在 patch |
| `parallel` 禁用覆盖 | 若文件存在，则必须核对 | 若文件存在，则必须核对 |
| `record-session` | 依赖 Trellis baseline skill + shared helper | 同左；当前 workflow 不追加 Codex 专属 patch |

## Step 4：收尾链路单独核对

当前 workflow 的 close-out 不是“只看 `record-session` 文件还在不在”。

至少确认：

- `.trellis/scripts/workflow/record-session-helper.py` 存在
- `record-session` 的说明仍强调“先 `record-session`，再 `archive`”
- `archive` 仍来自目标项目 Trellis baseline，而不是由当前 workflow 重新分发
- `record-session-helper.py` 由安装器分发到 `.trellis/scripts/workflow/`，并被 `record-session` 的 workflow patch 引用；若 helper 缺失，当前 workflow 的元数据闭环无法成立

可用命令：

```bash
test -f .trellis/scripts/workflow/record-session-helper.py
rg -n "record-session-helper.py|task.py archive" .trellis/workflow.md .claude/commands/trellis/record-session.md .opencode/commands/trellis/record-session.md .agents/skills/record-session/SKILL.md 2>/dev/null
```

## 通过标准

装后核对通过，至少应满足：

- `upgrade-compat.py --check` 通过
- `AGENTS.md` 的 `TRELLIS` managed block 与 `workflow-nl-routing` 区段都存在
- `.trellis/workflow-installed.json`、`.trellis/workflow.md`、`.trellis/library-lock.yaml` 状态正确
- `.agents/skills/` 与 `.codex/skills/` 的影响面都被实际检查过
- 手动维护资产未被误写成“安装器托管”

若以上任何一项无法证实，应明确记为“未验证”或“有漂移风险”，不要默认视为健康。
