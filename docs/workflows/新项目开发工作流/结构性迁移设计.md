# 结构性迁移设计

> 用于定义当目标项目的 workflow 升级已经超出普通兼容升级能力时，如何进入“结构性迁移”处理。

---

## 定位

结构性迁移不是默认升级主链。

默认升级主链仍然是：

1. 当前仓库完成 workflow 源资产兼容升级
2. 目标项目完成 Trellis 官方升级，只处理官方基线冲突，并确认 `.trellis/.version` 已是当前最新 Trellis 版本
3. 使用 `A/B/C` 三态分析判断 `add / replace / merge / delete`
4. 对低风险漂移再使用 `upgrade-compat.py`

只有当这条主链已经无法解释或安全处理当前升级时，才进入结构性迁移。

若目标项目还没有升级到当前最新 Trellis 版本，则连 `A/B/C` 分析都不应启动；当前步骤必须停留在 Trellis 官方升级阶段。

---

## 什么时候触发

满足以下任一条件时，应将当前升级判定为“结构性迁移”而非普通兼容升级：

### 1. 补丁模型失效

- `start` / `finish-work` / `record-session` 的注入锚点或标题结构已变化
- 当前 workflow 不能再沿用“保留基线 + 注入补丁”的模型

### 2. 命令体系变化

- 阶段命令重命名
- 一个命令拆成多个命令
- 多个命令合并成一个命令
- 某个同名命令从“覆盖/合并/使用原生”中的一个结论切换到另一个结论

### 3. 文件布局变化

- workflow 托管文件路径整体调整
- helper scripts 的目录或调用契约变化
- Codex / Claude / OpenCode 的部署布局变化

### 4. 安装契约变化

- 安装器导入的 pack、路由注入、hook 依赖、初始化副产物变化
- 目标项目已无法按当前安装契约解释 workflow 已安装状态

### 5. 目标项目私有漂移过重

- 大量 workflow 托管文件都落入 `merge`
- 目标项目已把 workflow 托管文件当成本地私有实现长期维护
- 已无法通过“保留少量人工回贴”解决

### 6. 恢复基线失效

- `.backup-original` 缺失
- `.backup-original` 已明显不对应当前可接受的官方基线
- `upgrade-compat.py --force` 会把目标项目拉回过时结构

---

## 目标

结构性迁移要解决的不是“补一两个文件”，而是以下问题：

- 当前 workflow 的新结构到底是什么
- 目标项目旧结构和新结构之间如何映射
- 哪些目标项目私有适配要保留
- 哪些旧文件要停用、删除、改名或拆分
- 如何在迁移失败时回滚

---

## 非目标

结构性迁移设计不负责：

- 自动完成所有迁移
- 消灭所有人工判断
- 绕过 Trellis 官方升级本身的冲突处理
- 替代当前仓库对 workflow 源资产的兼容升级设计

---

## 输入

结构性迁移至少需要以下输入：

1. `A`：纯净 Trellis 基线
2. `B`：当前 workflow 最新期望状态
3. `C`：目标项目真实状态
4. 当前兼容升级分析报告
5. 当前仓库针对本次 Trellis 升级的源资产兼容结论

推荐额外输入：

- 目标项目中 workflow 托管文件的历史 diff
- 目标项目是否存在项目私有 workflow 规则说明
- 迁移前完整备份或升级分支

### 进入本设计前的版本门禁

在准备上述输入之前，必须先确认：

- 目标项目已经完成 Trellis 官方升级
- 目标项目 `.trellis/.version` 等于当前最新 Trellis 版本

若这个门禁未满足：

- 不执行 `A/B/C`
- 不执行 `upgrade-compat.py`
- 不进入结构性迁移动作设计

先完成 Trellis 官方升级，再回到本设计继续判断。

### 缺失版本字段时的解释

若目标项目没有在 `.trellis/workflow-installed.json` 中记录：

- `workflow_version`
- `workflow_schema_version`

则应将该目标项目视为 `legacy/unknown`，含义是：

- 当前无法仅靠安装记录判断它属于哪一代 workflow 结构
- 当前不能只靠版本字段选择迁移路径
- 当前仍需依赖 `A/B/C` 分析和目标项目真实状态来做判断

这类缺失不会让结构性迁移“自动成立”，它只说明：

- 当前这次迁移的判断成本更高
- 本次迁移完成后，应补齐版本字段，避免后续继续处于 `legacy/unknown`

---

## 输出

结构性迁移的最小输出不应只是“改文件”，而应包括：

1. 迁移结论
   - 为什么当前升级已不属于普通兼容升级
2. 迁移范围
   - 涉及哪些 CLI、命令、skills、scripts、附属资产
3. 迁移动作矩阵
   - 重命名
   - 拆分
   - 合并
   - 删除
   - 回贴目标项目私有改动
4. 回滚方案
5. 验证方案

---

## 迁移流程

### Step 1：冻结迁移对象

- 为目标项目创建独立升级分支
- 保存迁移前完整快照
- 明确本次迁移只处理 workflow 托管层，不顺便修改业务代码

### Step 2：确认新结构定义

在当前仓库内先回答：

- 新版 workflow 的阶段结构是什么
- 托管范围是否变化
- 哪些旧命令已废弃
- 哪些新命令/新文件成为正式入口

这一步必须先于目标项目迁移。

### Step 3：建立旧结构到新结构的映射

至少整理一张映射表：

| 旧资产 | 新资产 | 动作 | 说明 |
|--------|--------|------|------|
| 旧命令/旧文件 | 新命令/新文件 | rename / split / merge / drop | 为什么这么处理 |

### Step 4：分离目标项目私有适配

把目标项目私有改动分为三类：

- 应上收为通用能力
- 只属于该目标项目
- 无价值漂移

只有完成这一步，后面的迁移动作才有边界。

### Step 5：执行迁移

按以下顺序执行：

1. 清理已废弃资产
2. 建立新结构所需文件
3. 应用当前 workflow 的新托管内容
4. 恢复必要的目标项目私有适配
5. 再做一次局部 diff 复核

### Step 6：验证

迁移完成后，至少验证：

- workflow 主入口是否还能进入正确阶段
- 关键命令/skills 是否仍可用
- helper scripts 路径是否仍有效
- 目标项目私有适配是否还保留在正确位置

---

## 与 `upgrade-compat.py` 的关系

结构性迁移不是 `upgrade-compat.py --force` 的加强版。

区别：

- `upgrade-compat.py`
  - 适合低风险漂移修复
  - 依赖当前补丁模型和历史备份
  - 默认假设托管结构没有根本变化
- 结构性迁移
  - 处理托管结构本身已经变化的场景
  - 不默认依赖旧备份作为最终基线
  - 需要显式迁移映射与人工判断

因此：

- 结构性迁移可以使用 `upgrade-compat.py` 的局部能力
- 但不能把 `upgrade-compat.py` 视为结构性迁移主入口

---

## 未来可落地的工具化方向

当前先以“文档设计 + 分析驱动”方式落地即可。

若后续要提高可重复性，可考虑按以下顺序演进：

### Phase 1：版本化记录

在 `.trellis/workflow-installed.json` 中增加：

- `workflow_version`
- `workflow_schema_version`

作用：

- 区分“目标项目使用的是哪一代 workflow 结构”
- 为后续迁移规则选择提供依据

补充规则：

- 对历史目标项目，缺失这两个字段时默认按 `legacy/unknown` 处理
- 不要求先补字段才能开始本次升级
- 但要求在本次兼容升级或结构性迁移完成后补写这两个字段

### Phase 2：迁移映射文件

为结构性迁移增加明确映射文件，例如：

- `docs/workflows/新项目开发工作流/migrations/<migration-id>.md`
- 或 `docs/workflows/新项目开发工作流/migrations/<migration-id>.json`

记录：

- 旧资产到新资产映射
- 删除列表
- 需要人工确认的项

### Phase 3：半自动迁移执行器

在映射文件稳定后，再考虑实现专门的迁移执行器，例如：

- `migrate-workflow.py`

但前提是：

- 已有稳定版本号
- 已有稳定迁移映射格式
- 已经验证结构性迁移不只是一次性的特例

---

## 设计原则

结构性迁移设计应坚持：

1. 先分析，后执行
2. 先定义新结构，再处理目标项目
3. 不把结构性 break 强行压成普通兼容升级
4. 明确哪些是 workflow 问题，哪些是目标项目私有问题
5. 保留回滚路径

---

## 推荐结论格式

当一次升级被判定为结构性迁移时，建议输出如下结论：

```markdown
## 迁移结论

- 当前升级已超出普通兼容升级范围
- 触发原因：<trigger list>
- 迁移范围：<affected assets>
- 推荐动作：进入结构性迁移

## 迁移步骤

1. <step>
2. <step>
3. <step>

## 风险

- <risk>

## 回滚

- <rollback path>
```

这份结论应作为后续实际迁移执行的依据。
