# 阶段状态机与强门禁协议

> 本文档是当前 workflow 的阶段执行纪律单一事实源。所有阶段命令、路由规则、walkthrough 与 helper 脚本都应以本文件为准。

---

## 1. 核心原则

当前 workflow 采用 **强门禁模式**：

- 每个阶段都必须有明确的进入条件、阶段内允许动作、退出清单与禁止动作
- 每个阶段结束后，必须先完成退出清单，再由用户明确确认，才允许切换到下一阶段
- AI 只能推荐下一步，不能把“下一步推荐”解释成已经获批的阶段切换
- `/trellis:start` 只能重入**当前已确认阶段**，不能自动跨阶段推进

---

## 2. 当前阶段的唯一判定链

当前阶段只允许按以下链路判定：

```text
.trellis/.current-task
  -> 当前执行中的叶子任务
  -> $TASK_DIR/workflow-state.json
```

禁止做法：

- 在多个 active task 之间自行猜测当前阶段
- 仅凭 `design/`、`task_plan.md`、`check.md` 等文档是否存在来推断已切换阶段
- 看到“下一步推荐”就默认把状态推进到下一阶段

### `.current-task` 约束

- `.trellis/.current-task` 不能为空
- 它必须明确指向当前正在执行的任务
- 为空时：
  - 不允许识别当前阶段
  - 不允许 `/trellis:start` 自动重入任何阶段
  - 只能进入“恢复当前任务 / 明确当前任务”分支

### 叶子任务约束

- 只有**当前执行中的叶子任务**允许持有 `workflow-state.json`
- 父任务只负责汇总，不负责驱动当前阶段状态
- 如果一个任务已经有 children，则继续执行前应先切换 `.current-task` 到当前要推进的叶子任务

---

## 3. `workflow-state.json` 契约

推荐路径：

```text
$TASK_DIR/workflow-state.json
```

推荐最小字段：

```json
{
  "version": 1,
  "stage": "design",
  "stage_status": "in_progress",
  "current_block": "option-research",
  "completed_blocks": ["input-review"],
  "allowed_next_stages": [],
  "awaiting_user_confirmation": false,
  "last_confirmed_transition": null,
  "notes": [],
  "checkpoints": {
    "architecture_confirmed": false,
    "execution_authorized": false
  },
  "updated_at": "2026-04-15T00:00:00+00:00"
}
```

### 字段含义

| 字段 | 含义 |
|------|------|
| `stage` | 当前已确认阶段 |
| `stage_status` | 当前阶段状态：`in_progress` / `blocked` / `awaiting_user_confirmation` / `completed` |
| `current_block` | 当前阶段正在执行的子块；无子块时可为 `null` |
| `completed_blocks` | 已完成的子块列表 |
| `allowed_next_stages` | 用户尚未确认切换前，仅作候选说明，不等于已切换 |
| `awaiting_user_confirmation` | 当前是否停在待用户确认点 |
| `last_confirmed_transition` | 最近一次已确认的阶段切换记录；存在时至少包含 `from` / `to` / `confirmed_at` |
| `checkpoints` | 当前阶段的关键确认点，例如 `architecture_confirmed`、`execution_authorized` |

### 建议 helper

```bash
python3 .trellis/scripts/workflow/workflow-state.py init <task-dir> --stage brainstorm
python3 .trellis/scripts/workflow/workflow-state.py show <task-dir>
python3 .trellis/scripts/workflow/workflow-state.py validate <task-dir>
python3 .trellis/scripts/workflow/workflow-state.py set <task-dir> --stage design --stage-status in_progress
```

在维护源 workflow 仓库时，可直接使用：

```bash
python3 docs/workflows/新项目开发工作流/commands/shell/workflow-state.py ...
```

### 版本升级策略

- 当前 `workflow-state.json` 版本号为 `1`
- 向后兼容的新增字段允许直接扩展；旧 helper 应忽略未知字段，不得因为多余字段直接破坏读取
- 破坏性 schema 变更（字段删除、字段语义变化、必填字段重构）必须：
  - 提升 `version`
  - 提供迁移脚本或迁移步骤
  - 在升级说明里明确旧状态文件如何迁移
- `workflow-state.py validate` 应拒绝未知或暂不支持的 `version`

## 4. `/trellis:start` 的边界

`/trellis:start` 在当前 workflow 中只承担两件事：

1. 读取 `.current-task` 与 `workflow-state.json`
2. 重入当前已确认阶段的内部流程

它**不负责**：

- 自动把 `brainstorm` 推到 `design`
- 自动把 `design` 推到 `plan`
- 自动把 `plan` 推到 `implementation`
- 因为缺某个文档就替你跨阶段补做

如果缺少必要状态，`/trellis:start` 必须停在恢复分支，而不是猜测：

- `.current-task` 缺失或为空
- 当前 task 不存在
- `workflow-state.json` 缺失
- 当前 task 已经不是叶子任务
- 当前状态与目标项目正式文档边界冲突

## 5. 阶段切换协议

所有阶段切换统一遵循以下顺序：

1. 当前阶段完成阶段内允许动作
2. 当前阶段退出清单全部满足
3. AI 输出：
   - 已完成
   - 未完成
   - 缺失项
   - 候选下一阶段
4. 设置 `stage_status = awaiting_user_confirmation`，并保持 `checkpoints.execution_authorized = false`
5. 等用户明确确认
6. 只有确认后，才更新 `stage` / `current_block` / `last_confirmed_transition`
7. 若确认的是进入 `implementation` / `test-first`，还必须显式把 `checkpoints.execution_authorized = true`

未确认前：

- 不能进入下个阶段
- 不能把候选阶段写成已执行阶段
- 不能因为“看起来差不多”而自动推进

补充约束：

- `allowed_next_stages` 只是候选说明，不等于已获准执行
- `plan` 阶段完成后，进入 `implementation` / `test-first` 前必须保留“用户已明确确认”的状态证据
- 未明确确认前，`execution_authorized` 必须保持为 `false`
- `design` 及后续阶段的入口还要通过 `workflow-state.py validate` 的文档门禁检查；其中包含 `brainstorm` 收口前必须落盘的项目级粗估要求

## 6. design 阶段特殊文档边界

在技术架构确认前：

- 可同步到目标项目正式文档的只有产品信息与功能需求
- `customer-facing-prd.md` 可以继续更新
- 若准备离开 `brainstorm` 进入 `design`，`customer-facing-prd.md` 必须已经补齐 `## 项目级粗估摘要`
- `developer-facing-prd.md` 不应正式生成
- 技术架构、设计结论、spec 对齐、检查矩阵、`finish-work` / `record-session` 适配都必须留在 task 工作底稿

补充门禁：

- `task_dir/prd.md` 必须已经补齐 `## 项目级粗估`
- 若上述粗估门禁未满足，`workflow-state.py validate` 必须拒绝 design 与后续阶段继续推进

在技术架构确认后：

- 才允许正式生成 `developer-facing-prd.md`
- 才允许正式落盘设计文档、项目级文档同步和工程化联动

### design 参考资产隔离规则

在 design 阶段和“可执行原型验证”步骤中，UI 原型只属于**参考资产**，不属于正式实现输入。

禁止直接带入正式实现的内容：

- 照片、截图、录屏、外链预览图
- Figma / Stitch / HTML / Storybook 等原型文件本体或分享链接
- 原型工具导出的组件代码、页面源码、切图占位代码
- 为验证原型临时拼接的网页源码、样式类名、演示数据结构

允许保留并转写进正式实现输入的结论：

- 页面信息架构、页面层级、区块划分
- 组件清单、关键元素位置、状态集合
- 主流程 / 支流程 / 异常流程与交互反馈
- 视觉方向、布局约束、可访问性结论、必要动效约束

约束：

- 允许保留的内容必须先沉淀为 task 工作底稿、`design/pages/*.md`、`design/specs/*.md` 或后续 `design/frontend-ui-spec.md`
- 正式实现阶段不得直接把原型资产或其导出代码当作实现输入
- 若原型与最终技术架构使用的语言、框架、组件体系不一致，仍只保留交互/视觉结论，不继承其代码形态

### design 后半段分块机制

design 后半段采用固定子块：

- 块 A：`developer-facing-prd.md`
- 块 B：设计文档落盘
- 块 C：项目级文档同步
- 块 D：工程化联动

约束：

- 块 A、块 B 完成后，都必须等待用户确认
- 块 C 与块 D 允许按项目情况调序
- 但块 C 与块 D 都必须留在 design 阶段内完成
- 进入 `plan` 前，必须满足：块 A/B/C/D 已完成 + design 退出清单完成 + 用户明确确认

---

## 7. plan 阶段硬禁令

`plan` 阶段只允许做规划动作，不允许做执行动作。

只允许：

- 拆真实 Trellis task / child task
- 生成或更新摘要型 `task_plan.md`
- 规划依赖、lane、门禁摘要、任务图摘要
- 给出候选下一阶段与候选叶子 task
- 停在 `awaiting_user_confirmation`

明确禁止：

- 生成项目基础代码、脚手架代码、页面初版、接口实现、迁移脚本
- 编写或修改任何属于具体 task 执行范围的实现代码
- 借“先做一点”“顺手补一个基础文件”为名提前进入 implementation
- 因重新进入 `plan` 阶段而自动恢复任何具体 task 的实现
- 在未获用户明确确认前，把 `.current-task` 切到实施叶子 task
- 在未获用户明确确认前，把任何 task 的 `workflow-state.stage` 改为 `implementation` / `test-first`

## 9. 当前版本结论

本 workflow 从“阶段建议链”收敛为“阶段状态机 + 强门禁协议”。

后续所有阶段文档都应显式说明：

- 当前阶段如何初始化 / 读取 `workflow-state.json`
- 当前阶段允许做什么
- 当前阶段禁止做什么
- 当前阶段何时进入 `awaiting_user_confirmation`
- 用户确认后才允许如何切换
