# 旧项目重构工作流 V1.0.0

基于《AI Workflow 项目开发工作流 V1.1.0》，针对"已有旧项目全面重构"场景的场景化适配版本。

**核心目标**：通过内部代码更新实现性能增强和安全性提高，同时保证原有功能完全不变。

## Trellis 升级兼容基线规则

若后续需要在**当前仓库内**维护这类以 Trellis 为基底的 workflow 内容，并判断 Trellis 升级后该如何兼容升级，初始 Trellis 框架代码必须来自：

1. 在 `/tmp` 下新建临时项目
2. 对该临时项目执行 `trellis init`
3. 以该临时项目中的纯净 `.trellis/`、CLI 基线命令 / skills / hooks 作为参考基线

不要直接把当前仓库内的 `.trellis/`、`.claude/`、`.opencode/`、`.codex/` 目录当作“初始 Trellis 基线”。这些目录已经混入本仓库自己的 workflow 补丁、共享 skills 与平台适配，只能作为当前实现参考。

这条规则只约束当前仓库内的 workflow 源内容维护，不直接作为目标项目执行升级兼容时的判定标准。

仓库级权威规则见：[自定义工作流制作规范](../自定义工作流制作规范.md)。

**核心差异**：
- 新项目开发是从0到1，重构是从1到1（功能不变）
- 增加"现状分析"阶段，逆向补全文档
- 测试重心从"功能验证"转向"回归验证 + 对比验证"

---

## 核心理念

### 功能不变原则

- 原有功能100%保留
- 用户可见行为必须一致
- 接口契约向后兼容
- 任何变更需通过回归测试验证

### 性能与安全导向

- 重构是手段，不是目的
- 每项代码修改需对应明确的性能/安全目标
- 无收益的架构变更不应进行
- 高危漏洞必须清零，中低危需风险接受与期限

## 阶段一：项目现状分析（重构特有）

### 1.1 代码可读性评估

**目的**：摸清家底，识别重构起点

**执行步骤**：

```
① 代码结构扫描
    ├── 目录结构分析（层级是否清晰）
    ├── 命名规范检查（变量/函数/文件命名）
    ├── 注释覆盖率统计
    ├── 重复代码检测
    └── 代码圈复杂度分析

② 技术栈识别
    ├── 依赖版本清单
    ├── 框架版本
    ├── 第三方库清单
    └── 过时/废弃依赖标记

③ 业务专家访谈
    ├── 关键业务流程与隐性规则
    ├── 历史行为差异与遗留约束
    └── 高风险变更点与不可动红线

④ 数据库逻辑深度排查
    ├── 触发器（Triggers）
    ├── 存储过程（Stored Procedures）
    └── 定时任务/作业（Jobs/Cron）

⑤ 关键结论输出
    ├── 可重构性评分（1-5分）
    ├── 重点改造模块清单
    └── 文档缺失清单
```

**交付物**：《代码现状评估报告》

### 1.2 技术债务盘点

**目的**：量化技术债务，为重构优先级提供依据

**评估维度**：

| 债务类型 | 评估指标 | 风险等级 |
|---------|---------|---------|
| 代码质量 | 重复率、圈复杂度、代码覆盖率 | 高/中/低 |
| 依赖健康 | 废弃依赖、安全漏洞、版本过旧 | 高/中/低 |
| 安全风险 | 认证方式、加密算法、数据传输 | 高/中/低 |
| 性能瓶颈 | 数据库查询、缓存策略、计算复杂度 | 高/中/低 |
| 架构问题 | 耦合度、扩展性、技术选型 | 高/中/低 |

**交付物**：《技术债务清单》

### 1.3 架构现状分析

**目的**：理解现有架构，为新架构设计提供基线

**分析内容**：

```
架构现状分析：

├── 系统架构图（逆向绘制）
│   ├── 模块划分
│   ├── 依赖关系
│   └── 数据流向
│
├── 接口契约盘点
│   ├── 内部接口清单
│   ├── 外部接口清单
│   └── 接口使用方映射
│
├── 数据模型现状
│   ├── 数据库表结构
│   ├── 缓存策略
│   └── 数据流转路径
│
└── 部署架构
    ├── 部署拓扑
    ├── 资源配置
    └── 运维依赖
```

**交付物**：《架构现状文档》（逆向工程版）

### 1.4 重构风险评估

**目的**：提前识别风险，制定应对策略

**风险矩阵**：

| 风险项 | 影响范围 | 发生概率 | 风险等级 | 应对策略 |
|-------|---------|---------|---------|---------|
| 回归遗漏 | 业务功能 | 中 | 严重 | 全量回归测试 + 对比测试 |
| 性能劣化 | 用户体验 | 中 | 高 | 性能基线对比 |
| 安全漏洞 | 数据/系统 | 低 | 严重 | 安全审计 + 渗透测试 |
| 兼容性断裂 | 集成方 | 中 | 高 | 接口契约冻结 + 灰度发布 |

**风险等级评估方法**：

| 风险等级 | 评估标准 | 量化指标 |
|----------|----------|----------|
| 严重 | 影响核心业务功能或导致数据丢失 | 回归用例失败率 > 5% |
| 高 | 影响用户体验或导致系统不稳定 | 性能下降 > 20% |
| 中 | 影响部分功能但可降级 | 回归用例失败率 1-5% |
| 低 | 影响轻微，易于修复 | 回归用例失败率 < 1% |

**发生概率评估方法**：

| 概率等级 | 评估标准 | 参考依据 |
|----------|----------|----------|
| 高 | 历史上频繁发生或技术实现复杂 | 重构经验少、依赖复杂 |
| 中 | 偶尔发生或有一定难度 | 有部分经验、依赖可控 |
| 低 | 极少发生且实现相对简单 | 经验丰富、依赖清晰 |

**风险应对策略模板**：

```
风险应对措施清单：

□ 回归遗漏风险
  - 预防：建立完整回归用例集
  - 检测：执行全量回归测试
  - 应对：预留回归测试缓冲时间
  - 负责人：XXX

□ 性能劣化风险
  - 预防：建立性能基线
  - 检测：性能对比测试
  - 应对：性能优化专项
  - 负责人：XXX
```

### 1.5 基础设施依赖盘点

**目的**：识别对运行环境的隐性依赖，避免迁移与重构后不可预期故障

**检查内容**：

```
基础设施依赖清单：

- [ ]环境变量依赖
  - 关键环境变量列表与默认值
  - 缺失时的失败模式

- [ ]本地文件与路径依赖
  - 绝对路径/固定目录
  - 读写权限与挂载要求

- [ ]系统特性与内核依赖
  - 特定内核参数/系统调用
  - 旧版系统组件兼容性

- [ ]外部运行时依赖
  - 本地服务/守护进程
  - 计划任务/系统服务
```

**交付物**：《基础设施依赖清单》

## 阶段二：重构目标确认

### 2.1 业务价值分析

**核心原则**：重构是手段，不是目的

**价值锚定**：

```
重构价值声明：

- [ ]性能提升目标
  - 响应时间：从 XXXms → 目标YYYms
  - 吞吐量：从 XXX TPS → 目标YYY TPS
  - 资源占用：CPU/内存降低XX%
  - 验收阈值：采样窗口=XXX（如7天/高峰期）；置信区间=XX%（如95%）；对比环境一致

- [ ]安全增强目标
  - 认证升级：从 XXX → 目标YYY
  - 加密增强：从 XXX → 目标YYY
  - 漏洞修复：N个高危/中危漏洞
  - 验收阈值：采样窗口=XXX（如1个完整扫描周期）；置信区间=XX%（如95%）；复测通过率≥XX%

- [ ]可维护性目标
  - 循环依赖：从 XXX → 目标YYY
  - 单元测试覆盖率：从 XX% → 目标YY%
  - 废弃业务逻辑分支：清理N条
  - 模块边界：依赖方向清晰、无逆向引用

- [ ]功能不变约束
  - 原有功能100%保留
  - 用户可见行为一致
  - 接口契约向后兼容
```

**补充要求**：
- 性能目标需绑定具体业务场景与 SLI/SLO，明确验收阈值（采样窗口、置信区间）、统计口径与对比环境一致性
- 安全目标需补充威胁建模/攻击面清单、合规要求，以及 SAST/DAST/SCA/配置审计的验收阈值（采样窗口、置信区间）
- 可维护性目标需量化到可衡量指标（依赖环数量、覆盖率、复杂度/重复率等）

**可维护性量化指标表（可执行）**：

| 指标 | 基线 | 目标阈值 | 采样口径 | 采样窗口 |
|------|------|----------|----------|----------|
| 循环依赖数量 | X | ≤Y | 依赖分析工具输出（按模块/包统计） | 最近一次全量扫描 |
| 单元测试覆盖率 | X% | ≥Y% | 行覆盖率（排除自动生成代码） | 最近一次CI全量 |
| 圈复杂度（均值/峰值） | X / X | ≤Y / ≤Y | 静态分析规则一致 | 最近一次全量扫描 |
| 重复率 | X% | ≤Y% | 统一重复检测阈值 | 最近一次全量扫描 |
| 废弃逻辑分支数量 | X | 0 或 ≤Y | 依据“僵尸逻辑判定标准” | 最近一次业务确认 |
| 模块逆向依赖 | X | 0 或 ≤Y | 依赖方向检查 | 最近一次全量扫描 |

### 2.2 重构范围定义

**范围划分原则**：

```
重构范围矩阵：

├── 核心重构（必须做）
│   ├── 性能瓶颈模块
│   ├── 安全风险模块
│   └── 架构问题模块
│
├── 辅助重构（尽量做）
│   ├── 代码质量优化
│   ├── 依赖版本升级
│   └── 文档补全
│
└── 暂不重构（保持现状）
    ├── 稳定运行模块
    └── 重构收益低的模块
```

### 2.3 兼容性约束确认

**核心约束**：功能不变 → 接口契约冻结

**约束清单**：

```
兼容性约束：

- [ ]接口层面
  - 对外API必须保持兼容
  - 内部接口可优化但需同步更新调用方

- [ ]数据层面
  - 数据库字段不减少（可新增）
  - 缓存策略变化需考虑预热

- [ ]安全改动影响评估
  - 认证/授权/限流/加密升级的兼容性评估
  - 对外行为变化需灰度、豁免与回退预案

- [ ]行为层面
  - 用户操作结果必须一致
  - 错误提示文案保持一致

- [ ]部署层面
  - 部署方式是否允许变更
  - 灰度/回滚策略
```

### 2.4 验收标准制定

**关键变化**：从"功能验收"转向"回归验收 + 对比验收"

**验收标准模板**：

| 验收类型 | 验收内容 | 验证方法 | 通过条件 |
|---------|---------|---------|---------|
| 功能回归 | 原有功能完整保留 | 全量功能测试 | 回归用例100%通过 |
| 性能提升 | 性能指标达标 | 性能测试 | 达成目标值 |
| 安全增强 | 安全基线达标 | SAST/DAST/SCA + 配置审计 | 高危清零；中低危有风险接受与期限 |
| 行为一致 | 用户可见行为不变 | 对比测试 | 差异数≤预定义阈值（默认0） |

## 阶段三：设计阶段

### 3.1 新架构设计

**设计原则**：
- 不为重构而重构，每项设计需对应性能/安全目标
- 架构变更需有明确的收益声明

**设计内容**：

```
新架构设计框架：

- [ ]模块结构设计
  - 模块划分与职责边界
  - 依赖关系与层次结构
  - 接口契约定义

- [ ]技术方案详细设计
  - 技术选型与替代方案对比
  - 核心算法与数据结构
  - 异常处理与容错策略

- [ ]数据与接口约束
  - 数据模型设计
  - API 接口设计
  - 兼容性策略

- [ ]新旧架构差异对比
  - 关键差异点清单
  - 差异影响评估
  - 适配/兼容方案
```

### 3.2 迁移路径规划（重构特有）

**核心挑战**：如何在不中断业务的情况下完成重构

**迁移模式选择**：

| 模式 | 适用场景 | 风险 | 复杂度 |
|------|---------|------|-------|
| Big-Bang | 小项目、可停机 | 高 | 低 |
| Strangler Fig | 大项目、需平滑 | 低 | 高 |
| 枝叶先行 | 核心稳定、边缘改造 | 中 | 中 |
| AB并行 | 关键业务、需对比 | 低 | 高 |

**决策依据（快速选择表）**：

| 因素 | Big-Bang 倾向 | Strangler Fig 倾向 |
|------|--------------|--------------------|
| 项目规模 | 小、依赖少 | 大、依赖多 |
| 停机窗口 | 有明确停机窗口 | 不允许停机或仅短停机 |
| 团队能力 | 资源有限、迁移经验一般 | 人手充足、具备迁移与对比测试能力 |

**快速判断**：满足右侧条件≥2项时优先选择 Strangler Fig。

**迁移路径模板**：

```
迁移路径规划：

Phase 1: 基础设施准备
  └── 新架构基建、新环境部署

Phase 2: 边缘模块迁移
  └── 非核心模块先行

Phase 3: 核心模块迁移
  └── 核心业务迁移

Phase 4: 数据与流量切换
  └── 灰度引流、数据同步

Phase 5: 旧系统下线
  └── 验证通过后下线
```

### 3.3 回滚方案设计

**回滚触发条件**：

```
回滚触发条件清单：

- [ ]功能维度
  - 回归测试用例通过率 < 99%
  - 用户反馈重大功能异常

- [ ]性能维度
  - 性能指标未达基线
  - 出现明显性能劣化

- [ ]安全维度
  - 出现新的安全漏洞
  - 安全扫描未通过

- [ ]稳定性维度
  - 错误率显著上升
  - 系统可用性下降
```

**回滚方案模板**：

```
回滚方案：

├── 一级回滚：代码回滚
│   - 快速回滚到重构前版本
│   - 适用：代码层面问题
│
├── 二级回滚：流量回切
│   - 流量切回旧系统
│   - 适用：运行时问题
│
└── 三级回滚：数据回滚
    - 数据状态恢复
    - 适用：数据一致性问题
```

## 阶段四：任务分解与准备

### 4.0 僵尸代码/无效业务逻辑下线（先减负）

**目的**：在重构前移除无效逻辑，降低回归测试负担

**执行要点**：
- 必须经过人工确认其为“僵尸逻辑”后方可下线
- 通过日志/监控/审计确认真实调用与使用率
- 与业务确认废弃功能与可下线范围
- 增加灰度/开关与回退策略
- 清理后更新回归用例范围与风险评估

**交付物**：《下线清单与影响评估》

**僵尸逻辑判定标准（需人工确认）**：
- 业务负责人确认：无现存业务依赖，且无近期上线计划
- 过去N个采样窗口内无真实调用（默认N=2个发布周期）
- 监控/日志/审计三方佐证一致（至少两方一致）
- 关键报表/对账/风控链路不依赖该逻辑

### 4.2 回归用例设计（重构核心）

**用例来源**：
1. 逆向工程：从现有代码中提取功能点 → 转化为测试用例
2. 历史Bug：历史修复的Bug清单 → 转化为回归用例
3. 用户反馈：用户报告的问题 → 转化为回归用例
4. 边界场景：基于代码分析识别的边界点

**用例设计原则**：
- 每个用例需明确旧版本预期行为
- 需记录旧版本的输入/输出数据快照
- 边界条件必须有明确预期
- 异常场景需覆盖常见错误码

**用例来源优先级**：

| 来源 | 优先级 | 说明 |
|-----|-------|------|
| 核心业务功能 | P0 | 必须100%覆盖 |
| 历史修复Bug | P0 | 确保不再复发 |
| 边界场景 | P1 | 遗漏高风险场景 |
| 用户高频操作 | P1 | 体验保障 |
| 低频功能 | P2 | 按资源覆盖 |

```markdown
## 回归用例：REG-XXX

- 对应模块：XXX
- 优先级：P0/P1/P2

### 用例描述
[描述旧功能的预期行为]

### 验证方法

2. 步骤2：...
3. 预期结果：...

### 旧版本基准
- 旧版本执行结果：[通过/失败]
- 旧版本响应数据：[数据快照]

### 验收条件
- 新版本执行结果：通过
- 响应数据与旧版本一致
```

