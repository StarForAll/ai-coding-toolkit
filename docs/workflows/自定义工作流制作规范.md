# 自定义工作流嵌入 Trellis — 制作规范

> 从 ai-coding-toolkit 项目实践中提取的约束条件。
> 用于指导「为 Trellis 制作自定义工作流」的完整流程。
> 最后更新: 2026-04-08

---

## 项目定位

```
当前项目是「工作流制作工具」，不是嵌入工作流的目标项目。
install/uninstall/upgrade 脚本是模板，供其他项目使用。
源文件统一放在 docs/workflows/<工作流名>/commands/ 下。
当前项目的 .claude/ 和 .trellis/ 不应被修改。
```

---

## 条件总览

| # | 条件 | 一句话 |
|---|------|-------|
| 1 | 基于 trellis-meta | 遵循 YAML frontmatter + Markdown 格式 |
| 2 | 源文件在工作流子目录 | 不直接放 .claude/，通过脚本部署 |
| 3 | 复用现有 Skills | 每个命令声明 skill 引用关系 |
| 4 | 自然语言触发词 | description 和文档含用户实际说的话 |
| 5 | 分支式下一步推荐 | 每个命令末尾输出多路径推荐 |
| 6 | 版本升级兼容 | install + uninstall + upgrade-compat 三件套 |
| 7 | Python + 零依赖 | 仅 stdlib，兼容 /usr/bin/python3 |
| 8 | 自动发现项目根 | 不硬编码路径 |
| 9 | 跨 CLI 兼容 | Claude / Cursor / OpenCode / Codex / Gemini |
| 10 | 不修改当前项目 | 测试后 git diff 为空 |
| 11 | 不依赖 trellis-local | 安装记录存 .trellis/workflow-installed.json |
| 12 | 纯净 Trellis 基线 | 当前项目内维护 workflow 升级兼容时，参考基线必须来自 `/tmp` 临时项目执行 `trellis init` 的纯净产物 |
| 13 | 重名命令判定 | workflow 命令与 Trellis 原生命令重名时，每次兼容升级都必须逐个判断“覆盖 / 合并 / 使用原生” |

## 条件 1：基于 trellis-meta 规范

所有新增命令/skill 必须遵循 trellis-meta 的定义格式。

```
命令文件：YAML frontmatter（name, description）+ Markdown 内容
参考：~/.agents/skills/trellis-meta/references/how-to-modify/
```

## 条件 2：源文件放在工作流子目录

```
项目根/
└── docs/workflows/<工作流名>/
    ├── <工作流文档>.md
    ├── 完整流程演练.md      # 推荐：官方 walkthrough / 实例演练
    ├── 命令映射.md
    └── commands/
        ├── command-a.md          # 命令源文件
        ├── command-b.md
        ├── start-patch-*.md      # start.md 注入块（独立文件）
        ├── install-workflow.py   # 一键安装
        ├── uninstall-workflow.py # 一键卸载
        ├── upgrade-compat.py     # 版本升级兼容
        ├── shell/                # 辅助脚本
        │   └── helper.py
        ├── cursor/README.md      # Cursor 适配
        ├── opencode/README.md    # OpenCode 适配
        ├── codex/README.md       # Codex 适配
        ├── gemini/README.md      # Gemini 兼容层说明
        └── codex-gemini/README.md # 过渡说明，指向已拆分文档
```

目标项目安装后生成：

```
.claude/commands/trellis/         # 命令
.trellis/scripts/workflow/       # 辅助脚本
.trellis/workflow-installed.json # 安装记录
```

如果该工作流覆盖新项目启动、需求发现或 PRD 编制阶段，则嵌入完成后还必须立即补充 `trellis-library` 的需求发现基础资产：

- 默认：`pack.requirements-discovery-foundation`
- 最低集合：
  - `spec.universal-domains.product-and-requirements.problem-definition`
  - `spec.universal-domains.product-and-requirements.scope-boundary`
  - `spec.universal-domains.product-and-requirements.requirement-clarification`
  - `spec.universal-domains.product-and-requirements.acceptance-criteria`
  - `spec.universal-domains.product-and-requirements.prd-documentation-customer-facing`
  - `spec.universal-domains.product-and-requirements.prd-documentation-developer-facing`
  - `template.universal-domains.product-and-requirements.acceptance-criteria-template`
  - `template.universal-domains.product-and-requirements.customer-facing-prd-template`
  - `template.universal-domains.product-and-requirements.developer-facing-prd-template`
  - `checklist.universal-domains.product-and-requirements.acceptance-quality-checklist`
  - `checklist.universal-domains.product-and-requirements.customer-facing-prd-checklist`
  - `checklist.universal-domains.product-and-requirements.developer-facing-prd-checklist`

目的：先把需求文档基线补齐，再进入 `/trellis:feasibility`、`/trellis:brainstorm`、`/trellis:start` 等后续阶段，避免流程已安装但 PRD 规范缺位。

说明：以上 `spec.*` ID 指向 concern directory，实际会展开为 `overview.md`、`scope-boundary.md`、`normative-rules.md`、`verification.md` 四个子文件。`pack.requirements-discovery-foundation` 默认还包含 example 和 `solution-comparison` 等扩展资产，但它们不属于最低门禁集合。

推荐检测信号：

- `.trellis/workflow-installed.json` 存在
- `.trellis/library-lock.yaml` 不存在，或缺少最低集合中的关键 asset id

若目标项目尚未接入 `trellis-library` CLI，则先手动复制最低集合中的资产到 `.trellis/` 对应目录；当前规范不定义 `skip-library-gate` 一类的跳过开关。

### 推荐附加物：完整流程演练

若工作流包含 3 个及以上阶段，或带有条件门禁（例如外部项目、试运行授权、交付控制、回退分支），建议在同目录补一份：

```text
docs/workflows/<工作流名>/完整流程演练.md
```

用途：

- 给维护者一条“从输入到交付”的完整参考路径
- 验证命令、脚本、文档产物、spec 导入是否真的能串起来
- 降低后续只改命令、不补演练示例的漂移风险

可直接基于：

```text
docs/workflows/完整流程演练模板.md
```

最低要求：

- 至少一个真实样例任务目录，或明确写出为何当前只能提供模拟样例
- 至少一组真实执行过的验证命令
- 明确写出演练覆盖范围与不覆盖范围

## 条件 4：自然语言触发词

description 字段和文档中必须包含用户实际会说的话。

```yaml
---
name: my-command
description: 功能描述。触发词：用户会说的词1、词2、词3
---
```

文档中展开触发词表：

```markdown
| 用户说 | 触发命令 |
|-------|---------|
| "自然语言1" "自然语言2" | `/trellis:xxx` |
```

同时在 start.md 中维护全局触发词路由表。

## 条件 5：分支式下一步推荐

每个命令执行完毕后，AI 必须在末尾输出「下一步推荐」区块（不只线性路径）。

```markdown
## 下一步推荐

**当前状态**: <一句话描述当前完成的阶段和产出>

根据你的意图：

| 你的意图 | 推荐命令 | 说明 |
|---------|---------|------|
| 默认路径 | `/trellis:xxx` | **默认推荐**。<说明> |
| 回退修改 | `/trellis:yyy` | 回到上一步 |
| 跳过前进 | `/trellis:zzz` | 跳过中间步骤 |
| 其他操作 | `python3 script.py` | 或 Skill |
| 不确定下一步 | `/trellis:start` | 用 Phase Router 自动检测 |
```

规则：
- 始终包含「不确定 → `/trellis:start`」兜底
- 给出多种可能路径（回退、跳过、特殊情况）
- 标记**默认推荐**
- 可以推荐自定义命令、Trellis 官方命令、或 Skills

## 条件 6：版本升级兼容

必须提供三件套：`install` → `uninstall` → `upgrade-compat`

### 兼容升级基线规范（强制）

以后凡是**在当前仓库内维护基于 Trellis 的 workflow 内容**，需要判断 Trellis 升级后 workflow 源资产该如何兼容升级时，参考的初始 Trellis 框架代码必须来自**纯净基线**，而不是当前仓库里已经项目化修改过的资产。

这条规则的适用边界是：

- 适用于当前项目内对 workflow 命令、补丁、安装脚本、兼容脚本、说明文档做升级兼容设计
- 不直接作为“使用该 workflow 的目标项目”执行升级兼容时的判定标准

原因是目标项目已经有自己的历史状态、已安装补丁、备份、安装记录和部署漂移，不能拿“完全初始的 Trellis”直接要求目标项目按同一标准判定升级兼容。

强制流程：

1. 在 `/tmp` 下新建临时项目
2. 对该临时项目执行 `trellis init`
3. 如需验证某个升级后的 Trellis 版本，确保该临时项目使用的就是待兼容判断的 Trellis 版本
4. 以该临时项目中的 `.trellis/`、CLI 基线命令/skills/hooks 作为“初始 Trellis 基线”
5. 再将这份纯净基线与以下两类内容做对照：
   - 当前 workflow 需要注入的命令、补丁、helper scripts
   - 当前仓库内已经实现的 workflow 兼容逻辑与文档说明

若后续还要评估下游目标项目受影响的方式，再**额外**查看真实目标项目当前已安装的 workflow 状态；这属于目标项目侧影响评估，不改变本条规范的适用边界。

禁止事项：

- 禁止把当前仓库内的 `.trellis/`、`.claude/`、`.opencode/`、`.codex/` 直接当作“初始 Trellis 基线”
- 禁止用已经混入项目规则、workflow 补丁、共享 skills、平台适配的仓库资产反推 Trellis 原始框架

原因：

- 当前仓库本身就是工作流制作工具，内部 Trellis 相关目录已经承载了本项目自己的项目化修改
- 若直接拿本仓库做基线，将无法区分“Trellis 升级带来的基线变化”和“当前 workflow/项目历史定制”

三份参考对象的职责必须分清：

| 参考对象 | 用途 |
|---------|------|
| `/tmp` 临时项目执行 `trellis init` 后得到的纯净产物 | 判断 Trellis 当前版本的原始基线结构 |
| 当前 workflow 源资产（`docs/workflows/<工作流名>/commands/**` 等） | 判断当前仓库里的 workflow 源内容需要依赖和注入哪些补丁点 |
| 真实目标项目当前安装状态（可选、按需查看） | 仅用于评估下游项目已经安装后的影响，不作为本规范的直接判定标准 |

### 原生命令重名判定规范（强制）

当在当前仓库内维护 workflow 内容时，若发现某个 workflow 命令与当前 Trellis 纯净基线中的原生命令**同名**，每次做兼容升级都必须重新执行一次判定，不能沿用历史实现自动视为正确。

判定流程：

1. 先用 `/tmp` 临时项目执行 `trellis init`，确认当前 Trellis 版本里该原生命令的真实职责与内容
2. 再对照当前 workflow 同名命令承担的职责、上下游阶段位置和用户使用心智
3. 只能在以下三种结论中择一：
   - 继续覆盖：workflow 保持整份覆盖该原生命令
   - 合并：以 Trellis 原生命令为基底，只增补 workflow 特有规则或补丁
   - 使用原生：回归 Trellis 原生命令，workflow 侧不再继续覆盖
4. 判定结果必须显式记录到对应 workflow 文档中，并同步落实到安装、卸载、升级、测试逻辑

每个重名命令至少要记录以下内容：

| 字段 | 说明 |
|------|------|
| 命令名 | 与 Trellis 原生命令发生重名的命令 |
| 当前 Trellis 原生职责 | 来自 `/tmp` 纯净基线的真实职责 |
| 当前 workflow 职责 | 当前 workflow 对该命令赋予的职责 |
| 判定结果 | `覆盖` / `合并` / `使用原生` |
| 决策理由 | 为什么选择该结果 |
| 影响面 | 需要同步调整的 `install-workflow.py` / `uninstall-workflow.py` / `upgrade-compat.py` / 测试 / 文档 |

约束：

- 这条规则只适用于**当前仓库维护 workflow 源内容**时的兼容升级判断
- 它不直接等于目标项目执行 `upgrade-compat.py` 时的判定标准
- 后续每次 Trellis 升级，只要再出现同名命令，都必须重新跑这套判定，而不是默认复用上一次结论

### upgrade-compat.py

```
--check  检测冲突（版本比较 + start/命令/helper script 完整性检测）
--merge  自动合并（重新部署命令/脚本 + 注入 Phase Router）
--force  强制覆盖（从 .backup-original/start.md 恢复基线后重新注入）
```

### 关键设计

- 这类 workflow 默认是**嵌入在 `trellis init` 之后的目标项目上**，不是替换 Trellis 原生命令全集
- 当前仓库维护 workflow 升级兼容方案时，Trellis 原始基线应来自 `/tmp` 临时项目执行 `trellis init` 后的纯净产物
- 当前仓库内的 `.trellis/`、`.claude/`、`.opencode/`、`.codex/` 只能作为“已落地实现参考”，不能当作“初始 Trellis 基线”
- `start.md` 与 `record-session.md` 都可能是**保留 Trellis 基线后再注入补丁**的冲突点
- 其他新增阶段命令通常是独立文件，不与官方基线冲突
- Phase Router 注入块独立为 `start-patch-phase-router.md`
- 若需要增强 `record-session` 之类的基线命令，补丁块也应独立存放，并在安装/升级脚本里显式处理
- 安装记录存 `.trellis/workflow-installed.json`（JSON 格式）
- 目标项目侧 `upgrade-compat.py` 的版本记录与漂移检测，只是部署后升级兼容的**本地自动化检查层**，不能替代当前仓库维护 workflow 时的纯净基线对照

### 版本追踪

```json
// .trellis/workflow-installed.json
{
  "trellis_version": "0.4.0-beta.5",
  "installed": "2026-03-21T...",
  "commands": ["feasibility", "design", ...],
  "scripts": ["helper1.py", ...]
}
```

说明：

- `workflow-installed.json` 只记录“目标项目上次安装 workflow 时看到的 Trellis 版本”和当前 workflow 的部署清单
- 它可以帮助脚本判断“版本是否变化、部署是否漂移”
- 它属于目标项目侧部署记录，**不能单独充当**当前仓库维护 workflow 时的 Trellis 原始基线来源；原始基线仍应通过 `/tmp` 临时项目 + `trellis init` 获取

## 条件 7：Python 实现 + 零外部依赖

```
- #!/usr/bin/env python3
- 仅标准库：pathlib, shutil, subprocess, argparse, re, os, sys, datetime, json
- 不 pip install，不修改系统 Python
- 兼容 /usr/bin/python3（系统默认）
- chmod +x
- 颜色输出用 ANSI 转义码（硬编码常量）
- 绝不使用：requests, colorama, rich, click 等第三方库
```

## 条件 8：项目根目录自动发现

脚本不能硬编码路径。

```python
def find_root() -> Path:
    cur = Path(__file__).resolve().parent
    for _ in range(10):
        if (cur / ".claude").is_dir():
            return cur
        if cur.parent == cur:
            break
        cur = cur.parent
    sys.exit("未找到 .claude/ 目录，请在 Trellis 项目内运行")
```

## 制作检查清单

```
□ 源文件在 docs/workflows/<name>/commands/
□ 命令有 YAML frontmatter（name, description 含触发词）
□ 引用了相关 existing skills
□ 每个命令末尾有「下一步推荐」（分支式）
□ 脚本用 Python stdlib，零第三方依赖
□ 脚本自动发现项目根目录
□ 有 install-workflow.py（一键安装）
□ 有 uninstall-workflow.py（一键卸载）
□ 有 upgrade-compat.py（版本升级兼容）
□ 有 start-patch-phase-router.md（注入块独立文件）
□ 目标项目安装后 /trellis:start 自动路由
□ 若涉及新项目需求阶段，嵌入完成后已定义 PRD 规范基线补充动作（默认 `pack.requirements-discovery-foundation`）
□ 卸载后目标项目恢复原始状态
□ 跨平台适配器 README 已创建
□ 当前项目无运行时修改（git diff 为空）
□ 不依赖 trellis-local
```
