# 完整流程演练模板

> 用于给某个工作流补一份“从输入到交付”的官方 walkthrough。模板强调真实产物、真实验证和明确边界，避免只写概念说明。

---

## 适用范围

说明这份演练覆盖什么场景，不覆盖什么场景。

建议至少写清：

- 适用于哪些项目类型
- 演练聚焦哪个主链
- 不覆盖哪些阶段或哪些能力

---

## 演练场景

说明本次样例任务是什么，以及为什么选它。

建议包含：

- 业务背景
- 关键约束
- 风险点
- 样例目录链接

参考格式：

- [requirement.md](../../../.trellis/tasks/<task>/requirement.md)
- [assessment.md](../../../.trellis/tasks/<task>/assessment.md)
- [prd.md](../../../.trellis/tasks/<task>/prd.md)
- [task_plan.md](../../../.trellis/tasks/<task>/task_plan.md)
- [run-report.md](../../../.trellis/tasks/<task>/run-report.md)

---

## 前置条件

列出演练开始前必须成立的条件，例如：

1. 工作流命令已嵌入项目。
2. 相关 helper script 可以执行。
3. 相关 `trellis-library` 资产可被导入。
4. 已明确项目类型、交付模式或其他关键边界。

---

## 阶段 1：<名称>

### 1. 输入

- 输入文件 / 输入上下文

### 2. 执行动作

```bash
<command>
```

### 3. 预期产物

- [产物 A](<path>)
- [产物 B](<path>)

### 4. 本阶段检查点

- 必须冻结哪些字段
- 哪些风险必须在这一阶段暴露

---

## 阶段 2：<名称>

重复相同结构，按需要展开到完整主链。

推荐覆盖：

- 输入
- 执行动作
- 产物
- 检查点

---

## 实际验证命令

只列本次真实执行过的命令。

```bash
<command-1>
<command-2>
<command-3>
```

若某些命令未执行，必须明确写原因，不要假装覆盖。

---

## 关键结论

写清这次演练验证了什么，建议不超过 3 到 5 条。

例如：

1. 某个契约已经能从前置阶段贯穿到交付阶段。
2. 某个条件 spec 已收敛为条件必选，不再无条件导入。
3. 某个 checklist 已从一次性交付改成按交付事件执行。

---

## 后续扩展

写清下一步最值得补的内容，例如：

- 补真实代码实现任务
- 补自审 / check / finish-work 演练
- 补 hosted / trial 两条分支样例

---

## 编写检查清单

- [ ] 演练目标明确，不是泛泛介绍
- [ ] 样例目录可打开，路径有效
- [ ] 只写真实执行过的验证命令
- [ ] 明确覆盖范围与不覆盖范围
- [ ] 至少有一个能帮助维护者复现的主链
