# 需求风险评估参考手册（v1.0）

本文件用于减少主观漂移：评分，红线、踩坑信号、谈判条件均以此为准。

---

## 附录A：评分细则（0-5）

### A1 合规风险（权重30）

| 得分 | 标准 | 证据类型 |
|------|------|----------|
| 5 | 完全合规，授权与用途清晰 | 明确合法用途、完整授权、明确数据边界 |
| 4 | 轻微风险，可通过补充材料解决 | 需确认素材来源/补隐私政策/补授权声明 |
| 3 | 中等风险，需要明确条款与流程约束 | 涉及第三方/数据合规但可通过合同与流程落地 |
| 2 | 较高风险，缺口多且影响交付/上线 | 关键资质/授权/数据合规模糊，需强约束或暂停 |
| 1 | 高风险，短期难以补齐 | 合规要件缺失且客户无法承诺补齐 |
| 0 | 明显违规或不可触碰 | 违法用途、明确侵权、明显监管红线 |

### A2 可交付性与范围清晰度（权重20）

| 得分 | 标准 | 证据类型 |
|------|------|----------|
| 5 | 范围非常清晰，可直接拆解排期 | 详细功能清单 + out-of-scope + 验收标准 |
| 4 | 基本清晰，少量细节待澄清 | 核心模块与边界明确，验收基本可写 |
| 3 | 部分模糊，需补充后才能稳定推进 | 只明确了核心目标或部分模块，边界不足 |
| 2 | 比较模糊，目标导向表述偏多 | "类似XX"、"做个平台"、缺少验收/交付物 |
| 1 | 高度不确定，容易无限变更 | 多处缺口且客户倾向口头变更/不愿冻结 |
| 0 | 完全模糊，无法评估与承诺 | 无范围边界、无交付物、无验收口径 |

### A3 工期可行性（权重20）

| 得分 | 标准 | 证据类型 |
|------|------|----------|
| 5 | 工期宽松且依赖清晰 | 有缓冲、里程碑合理、依赖与客户配合明确 |
| 4 | 基本可行，需要一定并行或小幅加班 | 计划合理但缓冲有限 |
| 3 | 较紧，需要高效执行与严格控范围 | 里程碑紧凑，依赖可控但无冗余 |
| 2 | 非常紧，必须砍范围或增加资源 | 明显缺缓冲或依赖不确定 |
| 1 | 极不现实，几乎必然延期或牺牲质量 | 关键依赖在客户侧且不可控，仍要求短期交付 |
| 0 | 不现实，无法在给定期限交付 | 与工作量明显不匹配或依赖不可获得 |

### A4 价格与收益匹配（权重20）

| 得分 | 标准 | 证据类型 |
|------|------|----------|
| 5 | 价格合理，显著高于成本，有利润与风险缓冲 | 报价覆盖成本 + 风险溢价 + 付款节奏健康 + 责任可控 |
| 4 | 基本匹配，覆盖成本且有合理利润 | 里程碑付款合理，变更可收费，责任上限可谈 |
| 3 | 偏低，覆盖成本但利润薄 | 需强控范围与变更机制，控制责任/支持边界 |
| 2 | 较低，接近成本价 | 任何变更/延期/争议都可能亏损 |
| 1 | 极低，明显不匹配且客户预期高 | 存在免费变更预期或长期维保未计费 |
| 0 | 严重低于成本或付款结构极差 | 结果付款/尾款过大/长期拖欠风险高/责任无限 |

> 备注：该维度可包含"主体信用/回款风险"的证据（未结款、历史逾期、拒绝预付款等），因为它会直接改变真实收益与坏账概率。

### A5 协作与沟通风险（权重10）

| 得分 | 标准 | 证据类型 |
|------|------|----------|
| 5 | 沟通顺畅且决策链明确 | 决策人明确、响应及时、反馈链路短 |
| 4 | 基本顺畅，偶有延迟但机制清晰 | 有owner与验收负责人，会议节奏稳定 |
| 3 | 有挑战，链路较长或多方参与 | 需建立单一入口与冻结机制 |
| 2 | 风险较高，多头反馈/决策人缺位 | 口头变更多、责任边界模糊 |
| 1 | 高风险，沟通失效迹象明显 | 长期不回复、反复推翻结论、拒绝文档化 |
| 0 | 无法有效沟通或明显对抗 | 基本无法建立共识或持续扯皮 |

### A6 [应急]优先级冲突可承受度（权重+10，仅应急场景）

> 说明：该维度用于衡量“插队/加急”对现有项目的冲击**是否可承受**。分数越高表示越可承受（冲击越小、调整代价越低）。

| 得分 | 标准 | 证据类型 |
|------|------|----------|
| 5 | 无冲突，可无缝插入 | 团队有闲置资源/项目可延后不影响 |
| 4 | 轻微冲突，可调整解决 | 需小幅调整现有项目计划 |
| 3 | 中等冲突，需要协调 | 需延期其他项目或增加资源 |
| 2 | 较高冲突，代价较大 | 需牺牲其他项目进度或质量 |
| 1 | 高冲突，严重影响 | 现有多个项目将严重延期 |
| 0 | 不可接受 | 资源完全冲突，无法承接 |

### A7 [应急]应急溢价覆盖（权重+5，仅应急场景）

| 得分 | 标准 | 证据类型 |
|------|------|----------|
| 5 | 完全覆盖，有溢价空间 | 15%+加急费且在预算内 |
| 4 | 基本覆盖，略有盈余 | 10-15%加急费 |
| 3 | 刚好覆盖成本 | 5-10%加急费 |
| 2 | 部分覆盖，需要压缩成本 | <5%加急费或不包含 |
| 1 | 不覆盖，完全义务劳动 | 无加急费 |
| 0 | 倒贴 | 低于正常报价 |

### A8 [应急]质量牺牲度（权重-5，仅应急场景，惩罚项）

> 口径：该维度是**惩罚项**，分数越高表示为了赶工期牺牲的质量/安全/可维护性越多；`0` 最好，`5` 最差。

| 得分 | 标准 | 证据类型 |
|------|------|----------|
| 0 | 不牺牲质量或仅做轻微取舍 | 明确不降低安全/测试/回滚/观测等底线；可接受小范围UI/非关键功能延后 |
| 1 | 轻微牺牲，可控且可补偿 | 明确列出延期补做清单与窗口；不跳过关键验收与安全基线 |
| 2 | 中等牺牲，需要强约束与补偿计划 | 需要压缩测试/文档但可设“补齐里程碑”；可回滚 |
| 3 | 较大牺牲，后遗症明显 | 明确跳过部分测试/审计/可观测；技术债会显著累积 |
| 4 | 很大牺牲，高概率引发事故/返工 | 关键质量环节被砍；上线后靠人肉兜底/夜间救火 |
| 5 | 不可接受的牺牲 | 明确要求跳过安全/合规底线或“先上线再说”且无补救窗口 |

## 附录B：红线检查清单（任一命中直接拒绝或暂停）

### B1 法律合规
- [ ] 用途违法/灰产：赌博、诈骗、非法集资、洗钱、钓鱼等
- [ ] 侵权：版权素材/商标/品牌未经授权
- [ ] 数据授权不完整：数据来源不明或无法授权

### B2 行业监管（如适用）
- [ ] 金融类：缺少必要牌照/备案/合规流程
- [ ] 医疗类：缺少相关资质或合规要求无法满足
- [ ] 教育类：缺少资质或政策要求无法满足
- [ ] 内容类：缺少必要许可或审核机制

### B3 数据合规（如适用）
- [ ] 个人信息/敏感数据收集缺少合法授权与告知
- [ ] 隐私政策缺失或不可落地
- [ ] 跨境传输/第三方共享无法合规
- [ ] 数据存储、留存、删除策略不明且无法承诺

### B4 平台与安全红线（如适用）
- [ ] 违反应用商店/平台政策（诱导下载、违规收集等）
- [ ] 恶意/间谍软件特征或规避安全机制
- [ ] 明确要求绕过安全/风控/审核

## 附录C：典型踩坑信号（命中越多风险越高）

> 规则：命中项必须进入"风险登记表"，并至少影响置信度或触发谈判条件（不要只当作"吐槽"）。

### C1 合同类
- "先做出来，合同后补"
- "口头约定的都算"
- "后期再签正式合同"
- "不要写在合同里/不要留痕"

### C2 预算类
- "预算有限但要全面"
- "先做，后期不会亏待你"
- "长期合作，先低价"

### C3 工期类
- "工期必须很短"
- "质量还要高"
- "改动很小，先做了"

### C4 付款类
- "不接受预付款"
- "只接受结果付款"
- "验收后再给钱"

### C5 验收类
- "先不定验收标准"
- "做出来再看"
- "差不多就行"

### C6 变更类
- "先做，后面再改"
- "应该很简单，改一下"
- "先帮我看看，不一定做"

### C7 应急类（应急场景额外关注）
- "明天就要"
- "今天之内"
- "很急，来不及了"
- "先做了再说"

## 附录D：谈判条件清单（只选与风险强相关的）

> 原则：条款必须能映射到"失败链路/风险项"，并可验证、可执行。

### D1 需求与变更
- [ ] 明确需求冻结时间点
- [ ] 冻结后变更流程与计价方式
- [ ] 变更评估与签字确认机制
- [ ] 变更次数/影响上限（超限自动触发加价或延期）

### D2 付款与信用
- [ ] 预付款（建议30%+）
- [ ] 中期款（建议30%+）
- [ ] 验收尾款（建议<=30%，并设验收时限）
- [ ] 付款时限约定与逾期处理（逾期暂停/交付顺延）
- [ ] 结果付款改为里程碑付款（或至少设最低保底款）

### D3 验收与交付
- [ ] 明确验收清单与交付物清单
- [ ] 验收标准文档化（可测试/可复现）
- [ ] 验收时限（如7-30天）
- [ ] 默认通过条件（逾期未反馈视为通过）
- [ ] 验收分歧解决机制（以文档/测试为准）

### D4 成本与第三方依赖归属
- [ ] 第三方API费用
- [ ] 云资源费用
- [ ] 短信/邮件费用
- [ ] 版权素材费用
- [ ] 域名/证书费用
- [ ] 客户侧依赖交付清单（账号/接口/数据/白名单等）与时限

### D5 止损机制
- [ ] 逾期付款暂停机制
- [ ] 项目终止条件与已完成部分结算方式
- [ ] 客户未按期提供依赖导致顺延/追加费用
- [ ] "不可控风险"触发时的退出与结算（例如政策/平台变化）

### D6 法律与责任（强烈建议在高风险项目使用）
- [ ] 责任上限（例如：以已收款/合同金额为上限，排除间接损失）
- [ ] 赔偿范围限定（不承担业务损失/利润损失等）
- [ ] 知识产权/源代码归属与许可方式（交付边界明确）
- [ ] 第三方素材/开源合规：客户保证授权或费用自担；乙方仅按合规清单交付
- [ ] 保密条款与数据处理约定（数据范围、留存、删除、访问控制）
- [ ] 安全义务边界（乙方负责哪些安全基线；客户负责哪些账户/环境）
- [ ] 争议解决与管辖（明确法律适用与仲裁/法院）

### D7 应急场景专用
- [ ] 应急通道费/加急溢价（建议15-30%）
- [ ] 明确可牺牲范围（哪些功能可简化/不做）
- [ ] 原有项目保护（验收期不被打断）
- [ ] 团队超负荷预警机制
- [ ] 暂停/终止权（原有项目出现风险时的优先級）

## 附录E：Score Capping 规则（防止猜测）

### 信息缺失时的得分上限

| 缺失信息 | 受影响维度 | 最高得分 |
|----------|-----------|----------|
| 验收标准/交付物不明确 | 可交付性与范围清晰度 | 2/5 |
| 验收/交付物缺失 | 整体置信度 | 不能是"高" |
| 付款条款不明确 | 价格与收益匹配 | 3/5（除非已明确固定价格+里程碑） |
| 工期/里程碑缺失 | 工期可行性 | 3/5 |
| 数据类型/授权缺失 | 合规风险 | 3/5 |
| 关键依赖不明 | 工期可行性 | 3/5 |
| 决策人或验收负责人缺失 | 协作与沟通风险 | 3/5 |
| 范围边界缺失且仅目标愿景表述 | 工期可行性 | 2/5 |
| 责任上限/赔偿边界完全缺失且客户强势 | 价格与收益匹配 | 3/5（默认需要谈判条件） |

### 非线性决策门（硬性规则）

以下情况**不能**得出"接"的结论（最多"谈判后接"或"拒绝"）：
- 合规风险 ≤ 2/5
- 付款方式是"结果付款"且无预付款/里程碑
- 验收标准缺失 AND 工期可行性 ≤ 2/5
- 范围是"做一个和XX类似"且无边界无验收标准
- 客户拒绝任何书面确认（合同/订单/邮件确认等）且要求先开工
- 客户明确要求"责任无限/高额赔偿"且拒绝任何责任上限/免责条款（建议引入法务）

## 附录G：置信度校准（防止过度自信）

置信度不是"感觉"，按证据完整性与一致性给出。

### 高（High）
- 交付物、验收口径、范围边界、关键里程碑/工期、付款结构、关键依赖、数据合规要点均有**明确原文证据**（或用户提供的可审计脱敏短引）；
- 无重大冲突项（见附录I）；
- Score Capping 不触发（或只触发轻微项，且已补齐）。
> 约束：若处于“引用受限（只能复述/不能短引）”模式，即使信息齐全，置信度也不得为 High（最多 Medium）。

### 中（Medium）
- 大部分关键项有证据，但存在少量缺口需要补充；
- 或存在可解释的小冲突（通过澄清可消除）；
- 结论依赖少量关键假设（≤5 条）且已明示。

### 低（Low）
- 验收/交付物、付款结构、范围边界、数据授权/合规、关键依赖中任一属于"缺失/冲突"；
- 或文本存在多处冲突且无法用现有信息消除；
- 或结论需要大量推断/"暗示"信息；
- 或踩坑信号密集且无法用合同机制兜底。
- 或用户明确限制引用/提供证据，导致关键字段无法审计（只能口头概述且拒绝字段化）。

## 附录H：总分与三情景计算（避免漂移）

### 总分(/100)公式

权重总和为 100，维度得分范围 0-5：

`总分 = Σ(权重 * 维度得分 / 5)`

例如：合规风险 4/5，则贡献为 `30 * 4/5 = 24` 分。

### base / best / worst 的边界

- base：只用"明确"证据，不用推断；有缺口按附录E执行 capping
- best/worst：只能在"关键假设已明示"的前提下调整，且不得绕过附录E的硬门

建议在表格中写成：`base / best / worst`，例如 `3 / 4 / 2`。

### 应急调整分（仅应急场景）

> 口径：**base 总分(/100)** 仍只使用 A1-A5（权重合计=100）。A6-A8 作为“应急调整分”，单独加/减。

`应急调整分 = 10 * A6 / 5 + 5 * A7 / 5 - 5 * A8 / 5`

- A6：优先级冲突可承受度（0-5，越高越好；表示插队冲击越小、调整代价越低）
- A7：应急溢价覆盖（0-5，越高越好）
- A8：质量牺牲度（0-5，**越高越差**，惩罚项；0最好）

`应急总分 = base 总分 + 应急调整分`

展示建议：
- 对外展示可写：`应急总分: min(100, base+调整)/100`（并同时给出 `应急调整分`）
- 内部排序可保留未封顶的 `base+调整` 作为参考（但红线/硬门仍优先）

## 附录I：冲突检测清单（提升深度与鲁棒性）

命中任一项，必须：
1. 在"证据/缺口/冲突"列中明确指出；
2. 进入风险登记表；
3. 触发置信度下调（通常至少到"中"或"低"）。

### 冲突类型

- **范围 vs 工期**：工作量显著不匹配或缺少缓冲
  - 例："7大模块"但"3天交付"
  - 例："先上线再完善"但无里程碑

- **验收 vs 付款**：验收不清但尾款高/结果付款
  - 例：无验收标准但尾款占60%
  - 例："做出来再看"但要求验收后付款

- **目标 vs 可验证**：只有愿景没有可测指标/验收清单
  - 例：只说"做一个优秀的系统"无具体指标

- **合规 vs 数据**：提到个人信息/敏感数据但未说明授权/留存/删除/跨境
  - 例：需要用户数据但无隐私政策说明

- **依赖 vs 可得性**：关键依赖在客户侧但无法保证提供
  - 例："客户提供数据"但无时间承诺

- **责任 vs 收益**：责任/赔偿要求极高但报价偏低或付款结构差
  - 例：要求"无限赔偿 + 结果付款"，但预算"尽量低"

- **IP/授权 vs 交付物**：要求交付全部源代码/可复用资产，但拒绝承担第三方授权责任或费用
  - 例：要求"所有素材/依赖都算你们的"，但不提供授权证明

- **应急 vs 质量**：应急工期但要求高完整性
  - 例："明天就要"但"功能和正常项目一样"
  - 例："加急"但"不改价格"

## 附录K：风险登记表优先级（可选，不参与总分）

> 用途：只用于风险登记表的排序、资源聚焦、谈判条件优先级；**不参与 A1-A5 的总分计算**。

### 概率（P，1-5）

- 1：很少发生（需要多个偶发条件同时满足）
- 2：小概率（偶尔发生）
- 3：可能发生（项目中常见）
- 4：高概率（多数类似项目都会遇到）
- 5：几乎必然（已出现早期信号或结构性必然）

### 影响（I，1-5）

- 1：可忽略（不影响关键里程碑/成本）
- 2：轻度（少量返工/小幅延期可消化）
- 3：中度（需要额外人天/延期但可控，需要明确动作兜底）
- 4：重度（显著延期/超支/质量事故/客户争议）
- 5：重大（法律合规/重大声誉/不可逆损失，可能触发拒绝或强止损条款）

### 优先级（P*I）

- 1-5：低（可记录、轻量监控）
- 6-12：中（必须有缓解动作或条款）
- 13-25：高（必须进入“谈判条件/止损机制”，否则结论通常不应为“接”）

## 附录L：最小补充信息问题库（阶段6优先复用）

## 附录M：引用受限/脱敏场景处理（不转贴原文时的证据口径）

适用：用户明确提出“材料不可转发/不要引用原文/不要贴原文句子”，但仍希望做风险评估与决策建议。

### M0 敏感信息最小披露（通用硬规则）

无论是否“引用受限”，只要输出中出现可能的敏感信息，必须最小化披露：
- 不输出：手机号、邮箱、身份证/护照号、银行卡/对公账号、详细地址、可识别个人姓名（如非必要）、账号密码、API Key、Token、内部私有链接、可用于登录或找回的验证码/密保信息。
- 需要保留证据结构时：用占位符替换（例：`[客户A]`、`[联系人B]`、`[金额X]`、`[域名Y]`、`[Token已省略]`），仅保留能支撑判断的最小信息（例如“结果付款/尾款占比高/验收不清”等）。

### M1 优先级（从高到低）

1. **用户提供可审计的脱敏短引**（推荐）：保留关键信息、去掉主体/金额/敏感字段；长度仍按“短引”控制。
2. **用户提供字段化事实**：按主 Skill 的“输入最小集”填表，允许你在输出中引用字段值（不贴原文句子）。
3. **仅口头概述且拒绝字段化**：视为关键缺口；通常应输出 `暂停（补齐前拒绝承诺）`。

### M2 证据锚点写法

- `[直引] "..."`：用户原文短引或用户提供的脱敏短引（若含敏感字段先按 M0 脱敏）
- `[复述] ...（引用受限）`：用户不允许贴原文，但提供了字段化事实/可审计概述（必须可映射到用户输入）
- `[缺口] ...`：需要证据但无法定位到任何可审计输入（等价于“未找到原文证据”，必须下调置信度，必要时触发 `暂停`）

### M3 置信度与结论约束（硬规则）

- 处于“引用受限（只能复述/不能短引）”模式：总体置信度不得为 `高`（最多 `中`）。
- 若范围/验收/付款/关键依赖/数据合规任一无法字段化：优先输出 `暂停`，禁止“接/报价锁定/排期承诺”。
- 红线判断若关键要素无法确认：默认 `暂停`，并明确需要补充的最小信息项。

## 附录N：ROM 报价/排期（区间+假设+验证，防止过度承诺）

适用：用户明确要求“报价/排期/人天”，或你处于报价谈判阶段。

### N1 硬规则（先说不做什么）

- **禁止单点承诺**：信息充分性 < 6/8、或存在关键冲突/硬门/capping 时，不给“单点报价/锁排期承诺”，只能给 ROM 区间。
- **区间必须可审计**：ROM 区间必须显式写出“关键假设（≤6）+ 必须谈判条件（Top3）+ 验证计划（Top 3-6）”。

### N3 分数→结论的默认映射（仅作摘要，门优先）

> 只在“未命中红线/硬门，且冲突已被消解或有明确条款兜底”时可参考。任何门命中都能直接改写结论上限。

- **80-100**：默认倾向 `接`（仍需检查付款/验收/责任边界是否健康）
- **65-79**：默认倾向 `谈判后接`（必须条件 Top3 必写）
- **50-64**：默认倾向 `暂停` 或 `谈判后接`（以信息缺口与风险归属为准；通常先补齐再锁报价/排期）
- **<50**：默认倾向 `拒绝` 或 `暂停`（高概率亏损/扯皮；除非能通过条款与范围大幅重构）

### N4 与 ROM 配套的付款/验收建议（高频可复用）

- 里程碑拆分：至少 `预付款 + 中期款 + 尾款`（尾款建议 ≤30%）
- 验收窗口：建议 7-30 天；逾期未反馈视为通过
- 逾期付款：明确暂停机制（暂停=交付顺延/风险不再自担）

## 附录O：AI/LLM 需求专项扫描（叠加标签，不替代标准流程）

适用：需求命中“大模型/LLM/RAG/微调/Agent/对话/生成式”等关键词。

### O1 最小补充信息（缺失即 capping/暂停）

- **效果目标与验收**：要“提升什么指标”？验收用什么评测集/样例？成功阈值是多少？
- **数据授权与边界**：训练/检索/日志是否有授权？是否含个人信息/敏感信息？留存与删除要求？
- **部署与供应商**：使用哪家模型/是否允许调用外部 API？是否需要私有化？算力与成本谁承担？
- **成本与延迟**：目标 QPS/并发/时延，预算上限（token 成本、向量库、推理资源）？
- **安全与越权**：是否需要防 prompt injection / 数据泄露 / 工具越权？是否有权限模型与审计要求？

### O2 典型踩坑信号（命中需进风险登记表 + 条款兜底）

- “要 100% 准确/不能出错/效果必须像 ChatGPT”，但不给评测集或验收口径
- “要私有化部署/离线”，但不给算力预算与运维责任
- “直接抓取/用网上数据训练/微调”，但授权不清或明确规避版权
- “上线后再慢慢调效果”，但合同/付款与验收强绑定效果且无灰度/回滚
- “模型输出要当最终事实/法律结论”，但无人工复核链路

### O3 建议谈判条件（按需选取，必须可验证）

- [ ] **验收口径**：提供评测集/样例 + 指标（准确率/召回/幻觉率/拒答率/时延等）+ 通过阈值
- [ ] **责任边界**：模型输出为辅助，需人工复核；限制间接损失；责任上限明确
- [ ] **数据处理**：数据范围、留存删除、访问控制、日志脱敏、第三方共享边界
- [ ] **安全基线**：prompt injection 防护、权限校验、工具调用审计、红队测试/滥用检测
- [ ] **成本与可用性**：token 成本归属、限流/降级策略、供应商政策变更的处理机制

### O4 结论上限提示（防止“强承诺效果”）

- 若 **评测集/验收口径** 不存在或无法字段化：结论优先 `暂停`；即使继续推进，最多 `谈判后接`，并以“验证计划（POC）”作为前置里程碑。
- 若 **数据授权/合规边界** 不清：先按附录B（红线）处理；无法补齐时默认 `拒绝` 或 `暂停`。

## 附录P：安全与运维基线快检（公网/敏感数据场景）

适用：系统对公网、含登录鉴权/权限控制、处理个人信息/敏感信息、对外 API、支付链路、AI/LLM 工具调用/数据回流等。

> 注意：本附录用于“最小补齐与谈判抓手”，不替代安全审计/渗透测试/合规咨询。

### P1 最小必问（缺失则降置信度/触发条款；关键缺口时可 `暂停`）

- **身份与权限**：用什么身份体系（账号体系/SSO/多租户）？角色与权限边界是否明确？是否需要审计追踪“谁在何时做了什么”？
- **环境与账号**：开发/测试/生产是否隔离？云账号/集成账号/第三方 Key 由谁提供与保管？最小权限与交接流程是什么？
- **密钥与配置**：是否有 secrets 管理（禁硬编码、可轮换、权限分级）？配置变更是否可追溯？
- **日志与审计**：需要哪些关键日志（登录、权限变更、数据导出、关键操作）？日志留存多久？谁可访问？
- **数据与备份**：是否需要备份/恢复点目标（RPO/RTO）？数据留存与删除策略是什么（含日志/训练/检索/缓存）？
- **上线门禁**：是否要求安全测试/代码扫描/依赖漏洞治理/灰度发布/回滚方案？上线事故的响应窗口与责任方是谁？

### P2 高杠杆条款抓手（按需选取，必须可执行）

- [ ] **环境与账号责任**：客户按期提供生产环境/账号/白名单；逾期导致顺延与费用调整；乙方不承担客户侧账号权限错误的后果。
- [ ] **权限与审计**：明确权限模型、最小权限原则、关键操作审计项与日志留存期限；数据导出需审批与留痕。
- [ ] **安全事件与漏洞处理**：约定漏洞分级、修复 SLA、披露与通知流程；第三方组件漏洞治理与费用/排期归属。
- [ ] **数据处理边界**：数据范围、用途、留存/删除、第三方共享边界；涉及个人信息时补齐授权与告知义务。
- [ ] **备份与恢复**：备份频率、保留周期、恢复演练与验收方式；恢复失败的止损与责任边界。

## 附录Q：承诺门（Commitment Gates：ROM/锁报价/锁排期/效果承诺的上限）

适用：用户要求“现在就定死报价/排期/上线时间/效果指标”、或需要把不确定性转成“可谈判的门槛与阶段化交付”时。

核心原则：**承诺必须可审计**。门没过就降档：`暂停` 或 `ROM 区间 + 假设 + 验证计划`，禁止单点承诺。

| 承诺门 | 最小前置条件（摘要） | 允许的承诺上限 | 常见下一步 |
|--------|----------------------|----------------|------------|
| G0 暂停门 | 红线无法判断/信息充分性<=4/8/范围或验收或付款缺失 | 只能 `暂停`（或红线明确则 `拒绝`） | 补齐最小信息（附录L） |
