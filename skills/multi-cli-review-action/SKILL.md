# Multi-CLI Review Action

## Overview

Multi-CLI Review Action 是多 CLI 协作流程中 CLI 2 的 skill。CLI 2 负责审查问题分析报告，与用户讨论优化方案，执行优化操作，并输出优化方案报告。CLI 2 需要保持质疑态度，基于实际情况审查问题点。

## When to Use

- 需要审查问题分析报告（cur_defect.md）
- 需要与用户讨论优化方案
- 需要执行优化操作（新增/删除/编辑文件）
- 需要输出优化方案报告

## Workflow

**触发条件**：调用 `/multi-cli-review-action [md-a-路径]`

**执行流程**：
1. **读取 md 文件 A**：获取问题分析报告
2. **输出分析摘要**：向用户展示问题清单
3. **质疑问题点**：基于实际情况选择性质疑
   - 质疑问题的准确性：问题描述是否准确反映实际情况
   - 质疑问题的严重性：问题是否真的需要解决
   - 质疑问题的优先级：问题的解决顺序是否合理
   - 质疑解决方案的可行性：提出的解决方案是否可行
   - **注意**：质疑需要基于实际情况，而非空假的质疑
4. **用户多轮讨论**：与用户讨论优化方案
5. **生成优化方案草稿**：包含被忽略的问题及原因
6. **用户确认**：用户输入"确认"后执行最终操作
7. **执行操作**：根据确认的方案执行操作
   - 如果执行失败，重复执行直到成功
   - 不能极端，多次失败后需要和用户确认新方案或思路
   - 同步更新 md 文件 B 相关联的内容
8. **输出 md 文件 B**：输出优化方案报告

## Input Parameters

| 参数 | 必需 | 说明 |
|------|------|------|
| `[md-a-路径]` | 否 | md 文件 A 的路径，默认：`tmp/cur_defect.md` |

## Output Files

### md 文件 B（优化方案）

```markdown
# 优化方案

## 问题 1：[对应 md 文件 A 中的问题标题]
- **方案描述**：具体的优化方案
- **执行内容**：实际执行的修改内容
- **是否忽略**：是/否
- **忽略原因**：如果忽略，说明原因

### 问题 2：...

## 总体状态
- 最后确认时间：2025-03-20
- 执行状态：completed/pending
```

## Quoting Rules

### 选择性质疑

CLI 2 需要基于实际情况选择性质疑问题点，而不是对所有问题点进行空假的质疑。

**质疑的范围**：
- **问题的准确性**：问题描述是否准确反映实际情况
- **问题的严重性**：问题是否真的需要解决
- **问题的优先级**：问题的解决顺序是否合理
- **解决方案的可行性**：提出的解决方案是否可行
- **必须解决原因**：即使是"必须解决原因"也可以被质疑

**质疑的原则**：
- 基于实际情况质疑
- 不是所有问题都需要质疑
- 质疑应该有具体的理由
- 质疑后需要与用户讨论

### 质疑示例

```
CLI 2：我看到问题 1 指出"代码逻辑不清晰"，但我查看代码后发现逻辑是清晰的。
这个问题是否需要重新评估？

用户：是的，我重新检查后发现逻辑确实是清晰的。

CLI 2：好的，我将忽略这个问题。
```

## Confirmation Rules

### 确认方式

用户输入"确认"即可执行最终操作，可以包含常见的确认方式：
- "确认"
- "是的"
- "对的"
- "可以"
- "ok"
- "yes"
- "y"

### 确认流程

1. CLI 2 生成优化方案草稿
2. 展示给用户
3. 用户确认或修改
4. 用户输入"确认"
5. CLI 2 执行操作

## Execution Rules

### 执行操作类型

根据实际情况判断，一般为文件的修改操作：
- **新增**：创建新文件
- **删除**：删除文件
- **编辑**：修改现有文件内容

### 执行失败处理

1. 如果执行失败，重复执行直到成功
2. 不能极端，多次失败后需要和用户确认新方案或思路
3. 同步更新 md 文件 B 相关联的内容

### 执行示例

```
CLI 2：我将执行以下操作：
1. 修改 ./src/utils.ts，添加错误处理
2. 修改 ./src/api.ts，更新 API 调用

用户：确认

CLI 2：正在执行操作...
操作 1 完成
操作 2 完成

优化方案已输出到 tmp/optimize.md
```

## Common Mistakes

### ❌ 空假的质疑
- 问题：对所有问题点进行没有实际依据的质疑
- 后果：浪费时间，用户失去信任
- 正确做法：基于实际情况选择性质疑

### ❌ 不更新 md 文件 B
- 问题：执行操作后没有更新 md 文件 B
- 后果：CLI 1 无法获取最新的优化方案
- 正确做法：执行操作后必须更新 md 文件 B

### ❌ 不等待用户确认
- 问题：没有等待用户确认就执行操作
- 后果：可能执行错误的操作
- 正确做法：必须等待用户确认后才执行操作

### ❌ 不处理执行失败
- 问题：执行失败后没有处理
- 后果：优化操作没有完成
- 正确做法：执行失败后重复执行，多次失败后确认新方案

## Tips

1. **保持质疑态度**：基于实际情况质疑问题点
2. **多轮讨论**：与用户充分讨论优化方案
3. **等待确认**：必须等待用户确认后才执行操作
4. **更新文件 B**：执行操作后必须更新 md 文件 B
5. **处理失败**：执行失败后重复执行，多次失败后确认新方案

## Example

```
用户：/multi-cli-review-action

CLI 2：
读取 tmp/cur_defect.md...

问题清单：
1. 代码逻辑不清晰（位置：./src/utils.ts:10-20）
2. 缺少错误处理（位置：./src/api.ts:5-15）
3. 文档不完整（位置：./docs/api.md）

质疑：
问题 1：我查看了 ./src/utils.ts:10-20，逻辑是清晰的。
这个问题是否需要重新评估？

用户：是的，我重新检查后发现逻辑确实是清晰的。

CLI 2：好的，我将忽略这个问题。
是否忽略原因：经过实际检查，代码逻辑是清晰的。

问题 2 和 3 需要解决。我将生成优化方案...

优化方案草稿：
1. 代码逻辑不清晰（忽略）
   - 忽略原因：经过实际检查，代码逻辑是清晰的

2. 缺少错误处理
   - 方案描述：添加 try-catch 错误处理
   - 执行内容：修改 ./src/api.ts，添加 try-catch

3. 文档不完整
   - 方案描述：补充 API 文档
   - 执行内容：修改 ./docs/api.md，补充缺失的文档

用户：确认

CLI 2：正在执行操作...
操作 1 完成：修改 ./src/api.ts
操作 2 完成：修改 ./docs/api.md

优化方案已输出到 tmp/optimize.md
```
