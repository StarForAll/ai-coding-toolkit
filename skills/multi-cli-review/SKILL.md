# Multi-CLI Review

## Overview

Multi-CLI Review 是一个多 CLI 协作的问题分析和优化流程。CLI 1 负责分析问题并输出问题报告，CLI 2 负责审查、讨论和执行优化操作。两个 CLI 通过文件传递上下文，实现跨 CLI 的协作。

## When to Use

- 需要分析文件（代码、文档、配置文件等）中的问题
- 需要输出结构化的问题报告
- 需要在优化操作后重新分析问题
- 需要与另一个 CLI 协作进行问题审查和优化

## Workflow

### 步骤 1：首次分析（步骤 1）

**触发条件**：调用 `/multi-cli-review <问题描述> [目标路径] [--md-a <路径>] [--md-b <路径>]`

**执行流程**：
1. **质疑问题描述**：确认问题描述是否准确不模糊
   - 如果问题描述模糊，询问用户澄清
   - 如果问题描述准确，继续下一步
2. **分析问题**：分析目标文件，识别问题
3. **输出问题报告**：
   - 同步输出到对话中
   - 输出到 md 文件 A（默认：`tmp/cur_defect.md`）

### 步骤 3：重新分析（步骤 3）

**触发条件**：调用 `/multi-cli-review --md-b <路径>`

**执行流程**：
1. **读取 md 文件 B**：获取优化方案
2. **读取最新的项目文件内容**：获取最新的项目代码
3. **重新分析问题**：
   - 重点：步骤 1 用户提出的问题有没有解决
   - 其次：md 文件 A 中的问题是否还存在
   - 其次：md 文件 B 中的优化方案是否有效
   - 其次：是否有新问题出现
4. **输出更新后的问题报告**：
   - 完全覆盖 md 文件 A
   - 对于没有解决的问题，重新填写并额外补充"必须解决原因"
   - 对于新出现的问题，重新添加

## Input Parameters

| 参数 | 必需 | 说明 |
|------|------|------|
| `<问题描述>` | 是（步骤 1） | 用户提出的问题，贯穿始终 |
| `[目标路径]` | 否 | 需要分析的目标文件/目录路径 |
| `[--md-a <路径>]` | 否 | md 文件 A 的路径，默认：`tmp/cur_defect.md` |
| `[--md-b <路径>]` | 否 | md 文件 B 的路径，默认：`tmp/optimize.md` |

## Output Files

### md 文件 A（问题分析报告）

```markdown
# 问题分析报告

## 问题描述
<用户输入的问题描述，贯穿始终>

## 分析结果
### 问题 1：[问题标题]
- **位置**：文件路径 + 行号范围
- **问题描述**：具体是什么问题
- **关联部分**：
  - 关联文件/代码：[路径 + 行号]
  - 依赖关系：[哪些部分会受影响]
- **严重程度**：高/中/低
- **影响范围**：[具体影响什么]
- **必须解决原因**：（仅对未解决的问题）说明为什么 CLI 2 必须解决这个问题

### 问题 2：...

## 迭代状态
- 当前迭代次数：1
- 上次分析时间：2025-03-20
- 上次优化方案来源：tmp/optimize.md
```

### md 文件 B（优化方案）

```markdown
# 优化方案

## 问题 1：[对应 md 文件 A 中的问题标题]
- **方案描述**：具体的优化方案
- **执行内容**：实际执行的修改内容
- **是否忽略**：是/否
- **忽略原因**：如果忽略，说明原因

### 问题 2：...

## 总体状态
- 最后确认时间：2025-03-20
- 执行状态：completed/pending
```

## Iteration Rules

- **最大迭代次数**：5 次
- **达到最大迭代次数**：提醒用户，询问是否继续或调整策略
- **问题解决**：当步骤 1 用户提出的问题全部解决时，流程结束

## Common Mistakes

### ❌ 跳过问题质疑
- 问题：直接开始分析，没有确认问题描述
- 后果：分析方向错误，浪费时间
- 正确做法：先质疑问题描述，确保准确不模糊

### ❌ 不读取最新文件内容
- 问题：步骤 3 没有读取最新的项目文件内容
- 后果：分析基于旧的代码，结果不准确
- 正确做法：步骤 3 必须读取最新的项目文件内容

### ❌ 不补充"必须解决原因"
- 问题：对于没有解决的问题，没有补充"必须解决原因"
- 后果：CLI 2 可能再次忽略这个问题
- 正确做法：对于未解决的问题，必须补充"必须解决原因"

## Tips

1. **问题描述要具体**：避免模糊的描述，如"代码有问题"
2. **分析结果要详细**：包含位置、描述、关联部分、严重程度、影响范围
3. **同步输出**：分析结果需要同步输出到对话中和 md 文件 A
4. **迭代状态要更新**：每次迭代都要更新迭代状态

## Example

```
用户：/multi-cli-review 检查这个文档的逻辑完整性 ./docs/api.md

CLI 1：
1. 质疑问题描述：
   - 问题描述：检查这个文档的逻辑完整性
   - 质疑：逻辑完整性具体指什么？是指文档结构的完整性，还是内容的逻辑性？
   - 用户确认：文档结构的完整性

2. 分析问题：
   - 检查文档结构
   - 识别缺失的部分
   - 输出问题报告

3. 输出问题报告：
   - 同步输出到对话中
   - 输出到 tmp/cur_defect.md
```
