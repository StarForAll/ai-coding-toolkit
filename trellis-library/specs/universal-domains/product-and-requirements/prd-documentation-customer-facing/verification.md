# Verification

Check the following:

* the document uses plain language and does not rely on unexplained technical shorthand
* the document contains sections covering problem, users, goals, scope, solution overview, requirements, priorities, success criteria, and open questions or assumptions
* requirements are written from the customer or stakeholder point of view
* the text explains what changes and why it matters without drifting into implementation mechanics
* core commitments, later enhancements, and out-of-scope items are separated clearly
* success criteria can be checked through customer-visible outcomes or agreed business measures
* any examples, diagrams, or mockups clarify the stated requirement and do not add hidden commitments
* ambiguities, assumptions, dependencies, and pending decisions are called out explicitly
* API contracts, schema design, class structure, or deployment detail are not used as the main way to specify the requirement
