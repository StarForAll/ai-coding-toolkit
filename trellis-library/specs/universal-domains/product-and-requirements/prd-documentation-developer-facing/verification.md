# Verification

Check the following:

* the document keeps the approved business intent intact while translating it into technical requirements
* the document contains sections covering problem context, goals and non-goals, scope, functional requirements, non-functional requirements, data or state model, interfaces and integrations, error handling, dependencies, acceptance criteria, testing, risks, and open questions
* functional requirements are specific enough to implement and verify
* non-functional requirements are concrete where performance, security, reliability, scalability, maintainability, or observability matter
* data structures, state transitions, interfaces, and integrations are defined where they affect behavior
* required behavior is distinguishable from optional implementation preference
* critical error paths, failure modes, and edge cases are documented
* dependencies, assumptions, third-party services, and system constraints are identified explicitly
* acceptance criteria and testing expectations are specific enough to drive verification
* rollout, compatibility, versioning, or migration requirements are documented when external contracts or persisted data are affected
* risks, unresolved questions, and material trade-offs are visible to implementation planners
* the document does not rely on customer-friendly narrative in place of executable technical requirements
