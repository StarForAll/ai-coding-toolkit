# Verification

Check the following:

* Each technical requirement traces back to a business requirement or customer need
* Technical goals align with business objectives and success criteria
* The document includes all minimum required sections: problem context, goals and non-goals, scope definition, functional requirements, non-functional requirements, data or state model, interfaces and integrations, error handling and edge cases, dependencies and constraints, acceptance criteria, testing expectations, risks and mitigations, and open questions
* Functional requirements are specific enough to implement and verify, with inputs, outputs, processing logic, error conditions, and acceptance criteria stated
* Non-functional requirements specify concrete measurable metrics for performance, security, reliability, and scalability
* Data entities, relationships, state transitions, and validation rules are defined where they affect implementation behavior
* API contracts, integration points, event definitions, and error contracts are documented
* Required behavior is clearly separated from optional implementation suggestions
* Error handling covers all critical failure modes with error codes, trigger conditions, recovery actions, and user messages
* All technical dependencies are identified with version requirements, fallback options, and impact assessment
* Acceptance criteria and testing expectations are specific enough to support unit, integration, and end-to-end verification
* Versioning, backward compatibility, and migration requirements are stated when the change affects persisted data or external contracts
* Technical risks are documented with impact assessment, probability, mitigation strategy, and contingency plan
* The document does not contain sales narrative, marketing copy, or vague feature intent in place of executable technical requirements
