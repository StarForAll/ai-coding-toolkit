# Verification

Check the following:

* the PRD documentation set identifies whether it is customer-facing, developer-facing, or paired
* shared facts are consistent across variants for problem, users, goals, scope, and success criteria
* confirmed requirements are separated from assumptions, open questions, and pending decisions
* in-scope and out-of-scope items are both listed explicitly
* paired PRDs explain how they map back to the same requirement baseline
* later requirement changes are visible as clarifications, scope updates, or decisions
* audience-specific variants do not conflict on behavior, scope, or success measures
* problem definition and acceptance intent are stated directly instead of being implied only through implementation detail
